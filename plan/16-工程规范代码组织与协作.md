# 工程规范、代码组织与协作

## 1. 工程目标

工程规范服务于三个结果：模块边界可替换、用户资产不可因错误丢失、任何商业发布依赖都可审计。项目是绿地工程，不为短期演示破坏领域边界，也不为尚未出现的多引擎需求建立过度抽象。

## 2. 仓库与工作区

推荐单仓库结构：

```text
apps/desktop-ui/          React/TypeScript/PixiJS
apps/desktop/src-tauri/   Tauri 桌面入口与命令薄层
crates/                   Rust domain/application/adapters
workers/ai-worker/        可选 Python 本地 Worker
schemas/                  版本化 JSON Schema/协议
models/manifests/         模型清单，不默认提交权重
fixtures/                 合法合成测试资产
tools/                    许可、schema、fixture、打包工具
plan/                     设计与原子计划
docs/                     用户/维护文档
```

使用 Git；默认分支受保护。Node 使用 npm workspace 和锁文件，Rust 使用 workspace/Cargo.lock，Python 使用 uv lock。发布构建必须从干净 checkout 复现。

## 3. 依赖方向

`F2S-CON-ENG-001`：依赖只能指向内层：

```text
UI/Tauri/Adapters → Application → Domain
Python Worker → 共享协议（不依赖桌面进程）
```

Domain 禁止依赖 Tauri、React、PixiJS、Python、数据库、文件系统、HTTP 或 Spine。Application 只能依赖端口 trait。基础设施实现 Repository、Worker、PSD、Spine CLI 和网络端口。

跨边界调用必须使用显式 DTO；禁止把数据库记录、Pixi 对象或 Spine JSON 直接传入领域层。

## 4. 设计模式使用规则

| Pattern ID | 使用位置 | 采用理由 | 禁止用法 |
| --- | --- | --- | --- |
| F2S-PAT-001 Hexagonal | Domain/Application 与外部系统 | 隔离 AI、存储、Spine、UI | 为每个纯函数创建无意义端口 |
| F2S-PAT-002 Command | 编辑、审批、Undo/Redo | 单一可审计变更单位 | 在命令中执行不可控长推理 |
| F2S-PAT-003 State Machine | Job 与审批门 | 拒绝非法跳转 | 一个巨型状态机覆盖所有实体 |
| F2S-PAT-004 Saga/Process Manager | 多阶段生产任务 | 可取消、恢复、补偿 | 假装任意算子中点都能恢复 |
| F2S-PAT-005 Strategy | 分层/规划/导出策略 | 可切换本地、私有远端、手工 | 根据字符串散落 if/else |
| F2S-PAT-006 Adapter/ACL | Worker、PSD、Spine CLI | 防止外部格式污染 IR | 在 UI 直接启动进程 |
| F2S-PAT-007 Repository | 项目与 artifact | 原子持久化、可替换索引 | 把 Repository 当业务服务 |
| F2S-PAT-008 Specification | 验证和发布政策 | 规则可组合并给证据 | 用它隐藏带副作用流程 |
| F2S-PAT-009 Memento/Checkpoint | 长任务阶段边界 | 故障恢复 | 保存整个内存图作为唯一事实源 |

若新增模式不能写清“替代方案、代价、删除条件”，不得加入架构。

## 5. 语言规范

### 5.1 TypeScript

- 开启 `strict`、`noUncheckedIndexedAccess`、`exactOptionalPropertyTypes`；
- 不使用 `any` 绕过边界，外部数据先以 `unknown` 进入 schema validator；
- React 不持有权威项目状态；逐帧 Pixi 状态不触发 React 重渲染；
- hook 不直接访问文件、网络或进程；
- 公共 DTO 从生成的 schema types 引用，不复制接口；
- ESLint 与 formatter 在 CI 零警告运行。

### 5.2 Rust

- `cargo fmt --check`、`cargo clippy --all-targets --all-features -- -D warnings`；
- 库代码禁止 `unwrap/expect`，测试和不可能分支需写原因；
- 错误使用稳定 error code + source chain，不以用户语言字符串作为控制流；
- 文件路径使用 `Path/PathBuf`，进程使用参数数组，禁止 shell 字符串拼接；
- 对用户输入做上限、路径归一化和根目录约束；
- 不在 async executor 中执行无界 CPU/GPU 或阻塞文件操作。

### 5.3 Python Worker

- 独立 uv 环境与锁文件；Ruff、mypy、pytest；
- Worker 无权修改 project manifest，只能写受控 job 输出目录；
- 不捕获后忽略异常；错误映射为版本化协议错误；
- 模型加载、设备选择和降级必须显式报告；
- 禁止在 import 时联网或下载权重；
- 每个模型由 manifest 指定版本、哈希、许可和能力。

## 6. 命名、ID 与 schema

- 领域名称使用英文稳定标识，UI 展示可本地化；
- UUID/ULID 用于实例，`F2S-*` 用于需求、设计、测试和任务；全部ULID wire必须匹配`^[0-7][0-9A-HJKMNP-TV-Z]{25}$`，只接受26位大写Crockford Base32并逐字roundtrip，拒绝I/L/O/U、小写、alias归一化、错长度和首字符8–Z；
- JSON 字段采用 `camelCase`，Rust 内部可 snake_case 并显式 serde rename；
- 墙钟时间统一UTC ISO-8601；动画领域、持久化和协议时间只使用canonical i64 tick字符串与约分有理timeBase。整数frame仅为UI显示/吸附派生值，不持久化也不作为关键帧身份；
- 长度单位在 Rig IR 中明确为像素/标准化单位，坐标系必须携带 axis/origin；
- schema 删除或改变语义需要 major 版本；新增可忽略字段可 minor 版本；
- ID 永不复用，废弃项标 `deprecated` 与 `supersededBy`。

## 7. 错误模型

`F2S-ERROR-CODE-REGISTRY-001` 是错误码exact ID与语义的唯一owner。格式固定为 `F2S-<SUBSYSTEM>-<CATEGORY>-<NNN>`；`F2S-E-*`、`F2S-ERR-*`等R1格式均为deprecated，只能在迁移表出现。每个错误包含稳定code、用户可读message key、技术详情、是否可重试、建议动作、关联job/artifact。日志可以记录source chain，但UI不展示敏感绝对路径和token。

| Exact code | Canonical category / trigger owner |
| --- | --- |
| F2S-APP-RUNTIME-001 | APP / application runtime failure |
| F2S-APP-WEBVIEW2-001 | APP / required WebView2 unavailable |
| F2S-STORE-IO-001 | STORE / filesystem, permission or capacity failure |
| F2S-STORE-DATA-001 | STORE / schema, hash or reference corruption |
| F2S-WORKER-PROCESS-001 | WORKER / process, protocol or timeout failure |
| F2S-WORKER-RESOURCE-001 | WORKER / declared compute resource insufficient |
| F2S-WORKER-UNKNOWN-001 | WORKER / safely normalized unknown failure |
| F2S-GPU-RESOURCE-001 | GPU / OOM, driver or device loss |
| F2S-RGPU-NETWORK-001 | RGPU / TLS, transport or invalid response |
| F2S-SPINE-COMPAT-001 | SPINE / missing, wrong patch or CLI failure |
| F2S-MIG-SCHEMA-001 | MIG / unsupported or invalid migration |
| F2S-DOMAIN-STATE-001 | DOMAIN / lifecycle, gate input or decision precondition invalid |
| F2S-DOMAIN-CONFLICT-001 | DOMAIN / expected project revision、registry sequence/hash、target stream entry或clock recovery head CAS conflict |
| F2S-DOMAIN-POLICY-001 | DOMAIN / DecisionPolicy, waiver, expiry or trusted-clock validation blocks operation |

领域文档只登记触发器、处置和message参数，不重新拥有code。生成的`schemas/error-registry.json`记录code、messageKey、retryable、severity、owner和deprecated/supersededBy；CI拒绝未登记或重复code。

### 7.1 `F2S-DEV-M02-008`：错误 registry 原子交付

该任务是错误码可执行门禁的唯一工程落点；DEV ID 的 canonical 注册仍属于追踪矩阵/原子计划，本节只冻结该任务必须交付的工程契约，不取得 `F2S-DEV` owner：

| 项 | 契约 |
|---|---|
| 输入 | 本文错误格式、所有已登记领域错误、i18n message key schema |
| 输出 | `schemas/error-registry.schema.json`、`schemas/error-registry.json`、生成的 Rust/TypeScript/Python 常量、`tools/check-error-registry` |
| 自动测试 | 正常 registry 通过；重复 code、非法格式、未登记代码、缺 messageKey/owner、deprecated 无 supersededBy、删除已发布 code 均必须失败 |
| 证据 | `F2S-EVD-M02-008` 保存 registry/schema/generator hash、扫描文件集、命令、退出码和负测结果 |
| 回滚 | registry 与生成物回到同一 commit；不得只回滚一侧或重用已发布 code |

编译时使用生成常量，禁止业务代码手写字符串错误码。`F2S-GATE-CI-REGISTRY-001` 扫描 Rust/TypeScript/Python、schema fixture 和用户文档示例；出现 registry 外错误码即 fail closed。

禁止：静默失败、仅 `console.log`、向用户显示原始堆栈、把取消当错误、失败后自动批准降级结果。

## 8. 状态修改与并发

- 每条用户编辑提交一个 Command；拖拽过程中使用 transient state，`pointerup` 才提交；
- 原始 artifact 不可变；新结果生成 revision；
- 同一项目只允许一个 writer，其他实例只读；
- Job 使用幂等键，取消与完成由单一仲裁器决定；
- 决策append命令CAS全局project revision、三个top-level registry的expected sequence/artifact/domain三元组及命令涉及的entry/governance双heads；未变registry root必须逐字复用，只有通过合法transition的命令才构造对应next snapshot。PriorityRegistry observation只作transport完整性，持续性绑定认证slice；RequirementsChange与RuleSetActivation必须原子推进PriorityRegistry内对应governance head。Approve还CAS current Waiver/ActorRegistry、OrganizationTrustEnrollmentV1、ActorTrustBundle/ActorRevocationBundle三元组、ClockCheckpoint及目标stream，提交前自校验ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1 authoritative bytes与artifact/domain双hash，并验证同一WaiverProposalV1/proposalHash上的三个不同credential签名与current ActorRoleValidationEvidence；SignatureAlgorithmRegistryV1缺失或错配时禁用waiver，签名实现只接受ECDSA P-256/SHA-256/P1363-64/low-S或RSA-PSS SHA-256/MGF1-SHA256/salt32/trailer1/modulus≥2048，以及无padding的strict canonical base64url。ClockRecovery必须追加全局sequence+1、prev双指旧epoch head且与RecoveryRecord唯一配对的跨epochcheckpoint。每条record的prev均保存并校验artifact+domain双hash，禁止把domain hash当CAS地址。持续有效性只绑定target、DependencySnapshot/PrioritySlice、stable inputs/outcome、DecisionPolicy、精确ack及静态GovernedWaiverBinding的当前动态验证；批准自身、无关revision或无关priority/waiver/actor/bundle entry不失效，真实依赖/priority变化或相关credential/SPKI/serial/anchor失效才精确失效；
- 所有canonical command envelope必须有必填closed-enum `commandKind`，且tag进入排除expected-CAS heads后的JCS/canonicalPayloadHash。所有append先按`(actorId,idempotencyKey)`查询CAS可重建的首次`{commandKind,payloadHash,resultKind,resultRef}`：仅同kind同payload返回首次同类型结果且不推进project/registry；同key跨commandKind或同kind异payload均为different-payload conflict，零事件/零transition并禁止返回另一结果类型。ClockRecoveryRecordV1必须保存commandKind/canonicalPayloadHash/committedProjectRevision；ActorRegistrySnapshotV1 sequence=0时commandKind/actorId/commandId/idempotencyKey/canonicalPayloadHash/committedProjectRevision全null，sequence>0时全非null。空redb必须从全部append权威链重建首次结果，禁止依赖endpoint局部缓存、进程内状态或迁移器补字段；
- Gate rule crate只接受RuleCatalog成员且已签名激活的RuleSet，并只能通过`TrackedGateInputResolver`取得`DecisionFreeDomainView`中的target/object/rule/policy/priority/evidence；规则代码禁止读取文件系统、环境变量、进程、网络、线程、时钟、随机数、locale或未声明字段。架构测试禁止直连Repository/ProjectSnapshot，断言read-set规范闭包与DependencySnapshot精确相等，并用恶意fixture证明hidden evaluator state、缺边和多余边均fail closed；
- 文件保存算法、durable边界、journal状态、ApprovalRecord/AuditEvent冻结顺序和提交点只由 `F2S-DOC-STORE-001` §8 定义；本文件不得复制步骤。工程层只要求所有Repository实现通过 `F2S-TST-111` 并在证据存在前保持 `UNVERIFIED`；
- 禁止多个模块直接修改同一 manifest。

## 9. 测试与覆盖要求

覆盖率是报警指标，不是唯一门禁。Domain、serializer、migration、path security、state machine 的分支覆盖目标不低于 90%；UI 展示层以交互和 E2E 证据为主。任何 bug 修复必须先有可失败的回归测试或 fixture。

涉及存储、迁移、导出、安全和审批的变更必须包含负向/故障测试。快照更新需解释语义变化，不允许批量接受未知 golden diff。

## 10. 依赖与供应链准入

引入依赖必须填写：用途、替代方案、直接/传递许可、维护状态、是否进入发布包、二进制/联网行为、删除策略。精确allowlist和外部系统分类只引用`F2S-LIC-POLICY-001`，本文件不复制许可列表。未知、禁止或没有审计证据的项fail closed。

CI 使用 Rust/Node/Python 许可扫描、模型 manifest 校验、SBOM 和漏洞审计。未知项 fail closed。

## 11. Git 与评审

- 提交采用 Conventional Commit 风格并包含需求/任务 ID；
- 一个提交聚焦一个可回滚意图，生成物与来源同提交；
- PR 描述包含行为变化、风险、测试命令、截图/证据、迁移和回滚；
- 架构、schema、许可或安全边界变化必须增加 ADR；
- 作者不能作为唯一终审者；
- 不提交密钥、用户图片、模型大权重、构建缓存和 Spine 激活信息。

当前工作区不是 Git 仓库。`F2S-DEV-M01-001` 将初始化仓库、`.gitignore`、行尾和属性规则；执行前保留现有调研与 plan 文件。

### 12.1 单一 release pipeline

正式发布只允许从仓库根运行计划入口 `tools/release.ps1 -Profile Stable -Version <semver>`；这是一个由 hash 链状态清单约束的流水线，不是可由操作者任意跳步的命令合集。M01 创建脚本和 Schema 后，CI/CD 只调用该入口：

```text
preflight clean checkout + exact toolchain/lock hashes
  → frozen restore
  → format/lint/type/schema/error-registry negative tests
  → unit/property/contract/integration/fault tests
  → license-policy.snapshot + SBOM/NOTICE/ModelBOM + vulnerability gate
  → windows-appcontainer-v1 malicious Worker gate
  → deterministic frontend/Rust/optional Worker build
  → NSIS/portable package + clean Windows E2E + special-path launcher E2E
  → HSM/approved service signing
  → signed release-manifest/checksum/SBOM linkage verification
  → final-channel download verification
  → immutable F2S-EVD release index
```

每阶段输出 `release-stage.json`，包含输入 hash、输出 hash、runner、开始/结束时间、工具版本、状态和前一阶段 hash。后续阶段只接受前一阶段 `PASS` 且 hash 连续的清单；重跑创建新 attempt，不覆盖失败证据。签名阶段只能接收通过全部前置门的不可变 artifact hash，普通 PR runner 拿不到签名凭据。

`F2S-GATE-CI-RELEASE-001` 必须至少注入以下负测并证明流水线拒绝：dirty tree、漂移 lockfile、registry重复或governance head缺失/回滚、schema breaking diff、policy snapshot不匹配、ActorAttestationPolicy或SignatureAlgorithmRegistryV1 authoritative bytes/固定artifact/domain hash缺失或错配、ECDSA DER/high-S/错P1363长度、RSA-PSS MGF/hash/salt/trailer/modulus漂移、base64url padding/空白/宽松解码、ULID小写/I/L/O/U/alias/非26长度/首字符8+/roundtrip变化、commandKind缺失/未知/未进入hash及同key跨kind、未签名/非catalog/未激活RuleSet、hidden evaluator state、历史P1经Requirements/Rule迁移伪降P2、无/过期/错edition organization enrollment或同ProjectId替换、错WaiverProposalV1/proposalHash、同credential伪三人、Actor trust/revocation bundle回滚/过期、actor/credential伪复活、时钟回退或伪跨epoch跳转、未知许可/模型、Critical/High finding、Worker egress成功、安装/入口失败、签名或最终下载hash不符。使用governed waiver时还必须拒绝OrganizationBundleFreshnessDeclarationV1缺失、`limitationAcknowledged!=true`、错candidate/root/clock/signature、旧signingAttemptId、作业重启复用或声明至最终签名前不在`0..900s`窗口；无waiver时对应manifest双hash必须为null。禁止直接调用打包器或手工上传来绕过该入口；开发构建使用 `-Profile Dev` 且不能生成stable manifest。

### 12.2 领域与治理负测门

CI必须复用`F2S-TST-093`、`F2S-TST-094`、`F2S-TST-096`、`F2S-TST-099`、`F2S-TST-111`、`F2S-TST-112`、`F2S-TST-115`、`F2S-TST-116`、`F2S-TST-118`，并在release evidence index中链接各自不可变证据：

- RuleCatalog membership、RuleSet签名/激活与decision-free沙箱负测必须拒绝untrusted ruleset、direct repository access和hidden evaluator state；
- 幂等负测必须删除redb后从全部append权威链重建`(actorId,idempotencyKey)→commandKind+payload+resultKind`，证明同kind同payload重复100次返回首次同类型结果；同key异payload或跨commandKind均different-payload、零transition且不返回另一resultKind，并拒绝缺/未知tag、tag未入hash、Actor seq0/seq>0 command字段nullability、commandId唯一性、canonicalPayloadHash或committedProjectRevision错配；
- governance迁移必须证明PriorityRegistry两个heads可重放、历史P2→P1收紧立即使旧binding失活，且后续迁移不能再伪降P2；GC、迁移和索引重建必须从两个governance heads遍历全部records/evidence；
- waiver负测必须逐阶段篡改ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1 authoritative bytes与artifact/domain双hash，并覆盖严格ECDSA/RSA-PSS/base64url向量、无/过期/错edition OrganizationTrustEnrollmentV1、同ProjectId换enrollment、错proposal/proposalHash、同credential复用、ActorTrust/Revocation bundle回滚/分叉/过期、相关actor/credential撤权后伪复活；任一policy/algorithm registry/enrollment校验失败均禁用waiver，bundle推进必须重验并产新evidence，只有相关credential/SPKI/serial/anchor失效才inactive，无关entry推进不得误失效；
- release负测必须生成每次作业唯一signingAttemptId。存在governed waiver时，OrganizationBundleFreshnessDeclarationV1与最终manifest必须同attempt，`declaredLatestAt==ClockEvidence.effectiveNow`，最终签名前重读时间并满足`0ns <= currentEffectiveNow-declaredLatestAt <= 900s`；旧attempt、重启或超15分钟必须重新刷新bundle并签新声明。此证据仅记录release角色的人工流程声明，产品不得显示为“已证明组织全局latest”；
- clock负测必须覆盖wall/checkpoint/counter回退，以及新epoch checkpoint的prev-null、跳sequence、缺RecoveryRecord和伪跨epoch跳转。

## 13. Definition of Done

代码任务只有同时满足以下条件才可完成：

1. 可观察 outcome 与对应验收标准实现；
2. 正向、负向和适用的故障测试通过；
3. 错误、取消、恢复和资源释放路径已处理；
4. schema/迁移/用户文档按影响更新；
5. 依赖与许可证据已登记；
6. 无新增高危漏洞和未知许可证；
7. 需求—设计—测试—任务—证据矩阵已链接；
8. 有明确回滚或 feature flag；
9. 未把 UNVERIFIED 能力描述为 VERIFIED；
10. 规则、治理、Actor bundle与全epoch clock roots的重放/GC/迁移及上述负测均有PASS证据。
