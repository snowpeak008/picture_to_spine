# 需求规格与验收标准

## 1. 文档目的

本文档把 `F2S-DOC-CHARTER-001` 的产品承诺转化为可测试需求。它是功能范围、非功能属性和发布验收的权威来源。实现方案可以调整，但不得在未修改本文件并完成影响审查的情况下删除、弱化或重新解释 P0 需求。

关键字含义：

- **必须**：P0 发布门禁，不满足则首版不能完成；
- **应该**：P1 是声明范围内的商业发布要求；若延期，必须先通过需求变更删除相应功能/兼容声明，不能用 waiver 保留未兑现声明；
- **可以**：P2 增强项，不得阻塞 P0/P1；只有一个finding的全部需求/缺陷binding均为P2、severity=warning、非硬禁止类别且DecisionPolicy判为governed时，才可按`F2S-REL-WAIVER-001`用精确target/snapshot/scope、期限、owner/角色和证据延期；该可选路径还要求组织trust anchor及product/quality/release三个不同有效签名credential对同一proposalHash完成attestation，未配置时必须修复finding；
- **验证**：通过自动测试、人工操作、文件检查、网络观测或 Spine Editor 4.2.43 实测形成证据。

需求优先级不能由finding或waiver申请自报。本文件是FR/NFR及其priority的唯一active baseline来源；固定generator必须解析本文件、拒绝重复/缺项/非法priority，生成带schema/generator/source-doc hash与排序entries的`requirements-priority-baseline-v1.json`，再将artifact/domain hash签入应用manifest。21号只做反向一致性校验，不是领域artifact输入。范围变更删除声明时，09号RequirementsChangeRecord必须把旧ID保留为retired tombstone及原/更严格priority，而非降成P2或抹去历史。运行期缺陷只能由09号不可变DefectRecord链提供优先级。PriorityRegistry认证这两类来源，任何缺项、retired绑定、旧source、规则换绑、冲突或P0/P1降级都fail closed。兼容/范围声明与安全、隐私、许可、数据完整性、恢复、签名、更新根、Worker隔离、Spine声明和用户数据同属不可waive硬类别。

## 6. 核心验收场景

### 6.1 项目与母版

**`F2S-AC-PROJ-001`**

- 给定空目录和 `F2S-FIX-CHAR-001`；
- 当用户分别以 PNG、JPEG、WebP 创建项目、导入、设置侧视/地面/比例并批准母版，并尝试导入一个 PSD；
- 则三种 P0 图片格式能够进入不可变原图流程，PSD 被无副作用拒绝且给出转换说明；
- 则原图按哈希保存，工作版本与原图分离，审批审计存在，项目重开后状态一致。

**`F2S-AC-PROJ-002`**

- 给定保存中断故障注入；
- 当应用在写清单、写 blob 和切换当前指针三个阶段分别终止；
- 则重启只能恢复完整旧版本或完整新版本，并说明恢复来源。

### 6.2 规格、提示词和素材闭环

**`F2S-AC-CONTENT-001`**

- 给定动作文字和 `F2S-FIX-POSE-001`；
- 当用户完成六类 Spec 并生成内容；
- 则十个动作都有稳定 ID、阶段、表现策略、最小素材、提示词项和未决问题，且无孤儿 AssetSpec。

**`F2S-AC-CONTENT-002`**

- 给定 `attack_02` 的临时 Rig 在挥击中出现肘部透明洞和前臂强透视；
- 当诊断完成；
- 则系统分别创建隐藏区域修正和透视附件任务，关联证据帧，不通过简单加大 mesh 拉伸静默处理。

**`F2S-AC-PROMPT-001`**

- 给定已批准 StyleSpec 和母版；
- 当生成并导出 PromptPack；
- 则包内含母版引用、不可改变项、每个关键姿势正/负向提示、输出规格和 AssetSpec 映射，且未产生任何外部网络请求；外部回导无法提供生成元数据时保存 `unknown_external` 和用户声明。

### 6.3 分层与 Rig

**`F2S-AC-LAYER-001`**

- 给定自动分层服务不可用；
- 当用户选择手工路径；
- 则用户仍可完成蒙版、层级、隐藏区域外部导入、重组差异和图层批准。

**`F2S-AC-LAYER-002`**

- 给定母版与批准分层；
- 当执行重组检查；
- 则母版可见区域的未授权像素变化、透明洞和重叠不足均可定位到层和像素区域。

**`F2S-AC-RIG-001`**

- 给定批准分层；
- 当用户建立骨架、槽位、网格、权重、握持点和约束并批准；
- 则所有附件有稳定父级与 pivot，诊断无阻塞错误，保存重开后结构和数值一致。

### 6.4 动画与玩法审核

**`F2S-AC-ANIM-001`**

- 给定批准 Rig；
- 当用户完成十个动作；
- 则每个动作可播放、逐帧、循环或结束，诊断结果可追踪；`attack_01`、`attack_02`、`attack_03` 分别具有 anticipation/active/recovery 和武器接触。

**`F2S-AC-GAME-001`**

- 给定 AI 或模板建议的命中帧；
- 当用户未点击批准；
- 则动作和项目不得进入 `gameplay_approved` 或 `export_ready`。

**`F2S-AC-GAME-002`**

- 给定已经批准的攻击命中帧；
- 当用户移动相关关键帧或改变 FPS；
- 则审批变为 `stale`，旧值保留供比较，导出门提示重新审核。

### 6.5 导出与 Spine

**`F2S-AC-EXP-001`**

- 给定 `F2S-FIX-PROJ-001`；
- 当连续两次以相同配置导出；
- 则Rig IR、透明PNG、满足P0最小可重建契约的PSD、Spine JSON、`atlas-input-manifest.json`、PromptPack和报告先作为完整canonical snapshot原子提交到项目内`exports/<export-id>`，确定性字段一致；关闭CLI的运行必须断言snapshot内不存在`.atlas/.spine/.skel`。高级PSD降级逐项记录且不会掩盖最小契约失败。若用户选择项目外目录，系统随后以该外部根内同卷staging→校验→rename发布可重建副本；publish失败保留内部成功snapshot并显示独立失败，不出现半成品或把外部目录当真值。

**`F2S-AC-EXP-002`**

- 给定合法 Spine Editor 4.2.43；
- 当运行可选 CLI 验证；
- 则目标版本、命令摘要、退出码、日志和产物哈希被记录，Golden项目通过约定结构与动作检查；若请求`.atlas/.spine/.skel`，每个文件必须位于CLI-owned输出区并由本轮4.2.43 operation evidence证明，内置阶段在CLI运行前后均不写这些bytes。

**`F2S-AC-EXP-003`**

## 7. 需求失效与变更规则

| 上游变化 | 必须失效的下游 |
| --- | --- |
| 母版像素、朝向、比例或地面变化 | 分层比较、Rig 映射、全部动作预览、导出 |
| 图层边界、尺寸、pivot 或语义变化 | 关联网格、权重、附件轨道、动作诊断、导出 |
| 骨骼层级或 rest pose 变化 | 关联约束、权重、全部动画和游戏性审批 |
| MotionSpec 时长/FPS/阶段变化 | 动画诊断、关键姿势、事件与命中帧审批 |
| WeaponSpec 握持点或尺寸变化 | 手型、武器附件、attack 动作接触和命中审批 |
| 导出目标或 Spine patch 变化 | 全部兼容验证；新 patch 视为新的适配任务 |

实现必须通过依赖图计算失效范围，禁止用单个布尔 `dirty` 掩盖影响原因。

## 9. 发布阻断条件

以下任意一项存在时，发布判定失败：

- P0 需求没有测试或证据；
- 母版、分层、Rig、关键姿势、命中帧任一门禁可被后台任务静默越过；
- 禁止联网模式仍存在未说明的外发；
- 项目源资产仅存在于数据库或缓存；
- AI 派生像素无法追踪到源和遮罩；
- 提示词功能调用了第三方生图 API；
- Spine JSON 未在 4.2.43 实测却显示已验证；
- 内置writer生成`.atlas/.spine/.skel`，或这些文件缺少本轮合法4.2.43 Professional CLI provenance；
- 发布包包含白名单外且未获批准的代码、权重、字体、图标或样例资产；
- 双击入口无法启动且没有可读诊断；
- 保存或迁移故障可能破坏最后一个确认版本；
- 十个动作中任一缺少规格、预览或必要审批；
- `ReleaseGateCoverageReportV1`未从current配置枚举全部required Master/Specs/AssetPlan/Layers/Rig、十动作Pose+Gameplay和交付ExportProfile target，报告集合不全等，或任一target不是current active approval、使用旧head、stale/inactive、缺current ack/waiver validation；
- 任何P0/P1 requirement/defect或compatibility、scopeClaim、security、privacy、license、dataIntegrity、recovery、signature、updateRoot、workerIsolation、spineClaim、userData硬类别被waive、binding或以人工备注放行；
- `ReleaseWaiverPolicyV1`、`ActorAttestationPolicyV1`或`SignatureAlgorithmRegistryV1`的authoritative bytes、artifact hash、domain hash与签名application manifest固定值不一致；
- 任一governed waiver缺少当前有效且product/edition匹配的`OrganizationTrustEnrollmentV1`、product/quality/release三个不同current-valid credential对同一精确proposal的签名，或缺current Actor/Trust/Revocation/Clock validation；
- 含governed waiver的Stable候选没有由current release credential签署`OrganizationBundleFreshnessDeclarationV1`，声明与最终release manifest的`signingAttemptId`不同，或最终签名前`currentEffectiveNow-declaredLatestAt`为负/大于900秒；旧attempt、换attempt、作业重启和900s+1ns声明一律拒签。无waiver候选不得伪造声明，其manifest对应双hash必须为null；
- UI、报告或发布声明把离线bundle链/freshness declaration描述为能发现未导入的组织更新、证明global latest，或能抵抗管理员/磁盘快照的全状态回滚。
