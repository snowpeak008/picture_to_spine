# 数据模型、项目存储与迁移

## 1. 目标和权威边界

本文档定义 `.f2sproj` 项目目录、版本化 schema、内容寻址 Artifact、审计、原子保存、索引和 schema 迁移。故障注入、启动恢复和兼容性操作细节以 `F2S-DOC-RECOVERY-001` 为准；本文档是存储格式和存储接口的 canonical owner。

基本承诺：

1. 用户选择的 `.f2sproj` 目录是项目真值；
2. 清空全局缓存或删除 `index.redb` 不会删除源图、已批准产物或审计证据；
3. 原始输入、版本、批准和导出不原位覆盖；
4. 只有 Rust Project Store 可写项目；React、PixiJS、Python Worker 和私有远程无项目写权；
5. 不支持的未来 schema 只读打开，不猜测并覆盖。

依 `F2S-ADR-REGISTRY-001`，`F2S-ADR-DATA-001`至`003`的implementation-local canonical body由本文件拥有；24号只登记其exact owner和状态。

## 2. 存储决策

| 决策 ID | 决策 | 取舍 |
| --- | --- | --- |
| `F2S-ADR-DATA-001` | `.f2sproj` 是目录而不是单 ZIP/数据库文件 | 可搬移、可人工取证、可增量备份；代价是文件数较多 |
| `F2S-ADR-DATA-002` | 版本化 JSON manifest + 不可变 CAS 是真值，`redb` 只作可重建索引/调度投影 | 避免索引损坏导致项目丢失，但所有查询字段必须可从 manifest 重建 |
| `F2S-ADR-DATA-003` | 多文件保存使用 staging + journal + 不可变 revision + 原子切换 `project.json` 指针 | Windows 不提供跨多文件事务，因此以可验证指针提交实现事务语义 |

SQLite 不是 V1 选择。`redb` 是纯 Rust、ACID、MIT/Apache-2.0 候选；SQLite 虽然为 Public Domain 且商业可用，但会引入不同的许可分类和 C 供应链，对本项目的可重建索引用例没有必要。

## 3. 项目目录规范

```text
character.f2sproj/
├── project.json
├── clock-state.json              # 单调安全时间pointer；不承载艺术revision
├── revisions/
│   ├── 000000000041.json
│   └── 000000000042.json
├── artifacts/
│   └── sha256/ab/cd/<full-64-char-hash>
├── approvals/                   # 由 CAS 事件块生成的可审计投影，非唯一真值
│   └── <approval-segment>.jsonl
├── exports/
│   └── <export-id>/
├── work/
│   ├── journal/
│   ├── staging/
│   ├── quarantine/
│   ├── autosave/
│   └── project.lock
├── index.redb
└── recovery-report.json
```

### 3.1 命名和可移植性

- 项目目录的显示名可为中文，项目身份使用 UUID，不使用目录名作为主键。
- 项目内部引用只使用相对逻辑路径或 ContentHash，不存用户机器绝对路径作为产物引用。
- 导入文件必须复制到 CAS，原外部路径只可作为本地脱敏来源提示，不是打开项目的必要前置。
- Windows 文件名不能使用的字符只影响显示导出名映射，不改变内部 ID。
- 项目目录可整体搬移、复制或进入用户自有备份系统；打开时以内容而非最近项目索引识别。

### 3.2 轻量双击项目入口

Windows 对目录后缀关联支持有限。P0 是应用内“打开 `.f2sproj` 目录”；P1 可定义 `.f2slink` JSON 入口，只包含 schemaVersion、projectId、用户确认的相对/绝对定位提示，不包含图像、Token 或项目真值。项目搬移后定位失败时必须让用户重新选择，不扫描整块磁盘。

全部项目、领域、发布与证据schema的`Ulid` wire共用pattern `^[0-7][0-9A-HJKMNP-TV-Z]{25}$`：恰26字符、大写Crockford Base32、首字符0–7。schema/parser必须拒绝`I/L/O/U`、小写、连字符、空白、非26长度、首字符8–Z和`O→0`/`I/L→1`别名；decode→encode必须逐字相等，禁止迁移器或reader先归一化再接受。ULID原始wire参与JCS与所有artifact/domain/canonical payload hash。

## 6. `project.json` 与 RevisionManifest

### 6.1 ProjectPointer 示意

```json
{
  "schemaVersion": "2.0.0",
  "projectId": "01900000-0000-7000-8000-000000000001",
  "displayName": "character",
  "currentRevision": 42,
  "autosaveRevision": 43,
  "createdAt": "2026-07-11T00:00:00.000000000Z",
  "updatedAt": "2026-07-11T00:15:00.000000000Z"
}
```

`project.json` 尽可能小，不放入图像、Rig IR、完整审批日志或最近 Job 详情。`currentRevision` 是最后一个已提交项目版本；`autosaveRevision` 可指向更新的候选版本，不使人工审批自动恢复为有效。

以上是合法wire形状示意而非golden内容，重复零hash只为可读。`*Hash`均为裸64 lowercase hex。Manifest直接保存Priority/Approval/Waiver三个top-level registry三元组；PriorityRegistry snapshot再保存requirements-change与rule-set-activation的sequence/artifact/domain双heads，WaiverRegistry snapshot保存current ActorRegistry sequence/artifact/domain。四条registry snapshot链均有previous artifact+domain，governance/defect/approval/waiver records各自有双heads/prev。ActorRegistry唯一empty genesis（sequence=0）必须使enrollment、bundle和change-authorization双hash、commandKind、actorId、commandId、idempotencyKey、canonicalPayloadHash、committedProjectRevision全null，bundle sequence=0且actors为空；首次启用只能以单一sequence=1 `enrollOrganization` transition原子固定OrganizationTrustEnrollmentV1双hash、两条bundle sequence=1 roots、bootstrap authorization与initial actors，部分成功不得提交。sequence>0的六个command字段必须全部非null，commandKind必须是09号closed enum且进入canonical payload JCS/hash，commandId全项目唯一，committedProjectRevision固定首次使snapshot可达的revision。同一ProjectId的enrollment双hash此后永不替换；过期、错edition/签名/root时禁用governed waiver，只能克隆内容到新ProjectId后重新enroll/reapprove，旧grant不得迁移。sequence>0的每个ActorRegistry snapshot必须以sequence/artifact/domain三元组锚定current ActorTrustBundle与ActorRevocationBundle，两条bundle链的离线root签名、expiry、previous双hash与downgrade规则逐条验证；bundle head推进必须重验并产新evidence，只有相关credential/SPKI/serial/anchor撤销、过期或失效才使binding inactive，无关entry推进不得误失效。ActorRegistry引用的credential/certificate chain、WaiverProposalV1、三份不同credential签名attestation及ActorRoleValidationEvidence均必须以artifact+domain可取并绑定同一proposalHash；不得从actorId或UI角色推测签名事实。打开时按artifact寻址并重算domain，验证所有间接root及不可复活transition。ClockCheckpoint/Recovery亦双hash；clock-state可单调领先。sequence=0缺enrollment是合法禁用态；sequence>0缺任一root或enrollment过期时艺术项目仍可只读/修复，但governed waiver与签名发布fail closed。

### 6.3 `clock-state.json` 单调安全指针

该文件严格使用09号`ClockCheckpointPointerV1` schema；它不是艺术项目revision、不能指向领域对象，也不允许redb替代。普通checkpoint提交顺序固定为：持有project write lock并验证current pointer/manifest anchor→冻结09号ClockCheckpointRecord与双hash→同盘staging写/flush/schema+hash验证→promote record CAS→写/flush临时pointer→原子替换`clock-state.json`并flush父目录。pointer提交后即使ProjectPointer不变，时间只可前进。任何active waiver存在时最长5分钟执行一次，并在首次验证、approve/export/sign、resume、expiry和干净关闭前同步执行。

项目创建的首条checkpoint是唯一允许prev artifact/domain双null的记录。ClockRecovery创建新epoch的首条checkpoint仍必须按全项目sequence+1追加并双指旧epoch head，同时由唯一RecoveryRecord证明`oldEpoch→oldEpoch+1`；用prev=null、跳sequence或不经recovery伪造epoch的record即使hash合法也不得成为head。

RevisionManifest中的checkpoint是该revision观察到的最低祖先；当前`clock-state.json`允许领先。current manifest anchor必须与pointer同epoch并可沿prev artifact+domain双hash回到；保留的old manifest anchor允许跨越每一个由ClockRecoveryRecord唯一配对的合法epoch boundary追溯。pointer缺失/落后/分叉、current manifest与pointer epoch不等、old ancestry缺边界证明或任一hash错误时，历史可读而waiver验证fail closed。普通project decision先耐久推进checkpoint，再把观察head写入ApprovalRecord/next manifest；若项目提交失败，领先pointer无害。ClockRecovery使用OperationJournal：先把新epoch recovery+checkpoint artifacts和引用它们的新manifest一起提交ProjectPointer，再替换sidecar；崩溃在两者之间时，以已提交manifest物化精确新pointer，绝不把旧epochpointer继续用作active。

当前manifest anchor必须与pointer同epoch；回溯保留的旧epoch anchor时，每个epoch边界必须恰有一条连续ClockRecoveryRecord与紧随其后的checkpoint双head交叉验证。同epoch的checkpointUtc/lastSeenUtc/interrupt counter回退、跨epoch断链或伪跳均fail closed，不允许等下一条记录“自愈”。

项目复制/备份必须包含`clock-state.json`及其可达checkpoint CAS；若复制发生在应用运行中或新机器boot identity不同，09号规则使既有waiver fail closed并要求ClockRecovery，新epoch只使waiver-dependent approvals失活。删除应用缓存不影响该项目内指针；手工删除/回滚该文件不会丢艺术数据，但绝不能保留授权状态。

## 7. CAS Artifact Store

### 7.1 写入协议

1. 将候选字节写入同盘 `work/staging/<operation-id>/objects/<temp-id>`；
2. flush 文件，计算完整 SHA-256 和尺寸；
3. 按 media type 执行解码/schema/安全验证；
4. 根据哈希构造 `artifacts/sha256/ab/cd/<hash>`；
5. 如目标已存在，校验尺寸/哈希后复用，不覆盖；
6. 如不存在，同盘原子 rename 到 CAS；
7. ArtifactDescriptor 随下一个 RevisionManifest 提交。

完整 256-bit SHA-256（规范化为 64 个小写十六进制字符）是唯一内容身份。禁止截断为 64 bit、短哈希或以文件名替代；前缀目录只用于减少单目录文件数，不作为安全边界或唯一键。

### 7.2 图像和大对象

- 持久化图像与代理/缩略图使用不同 ArtifactDescriptor；清缩略图不会删源图。
- Mask 和 PixelProvenanceMap 是一等 Artifact，必须包含尺寸、编码和父哈希。
- 大 JSON 领域快照也可进 CAS，RevisionManifest 只引用 ContentHash。
- 预览 render target 可为可重建 cache；如它是 ApprovalRecord 的证据快照，则升格为不可变 Artifact，不随缓存删除。
- 每次决定所见的GateReport、RuleCatalog/RuleFamilyMigration/RuleSet/RuleSetActivation、Dependency/Priority observation+slice/RequirementsPriorityBaseline/RequirementsChange/GovernanceApprovalEvidence/Defect/FindingDraft/ResolutionEvidence/DecisionPolicy、ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1、3个top-level+waiver-owned ActorRegistry snapshots、OrganizationTrustEnrollmentV1/ActorRegistryChangeAuthorizationV1/ActorCredential/ActorTrustBundle/ActorRevocationBundle/WaiverProposalV1/Attestation/ActorRoleValidationEvidence、waiver records、binding/validation/clock evidence、OrganizationBundleFreshnessDeclarationV1/ReleaseGateCoverageReportV1、Approval/ClockCheckpoint/Recovery均为CAS对象；引用保存双hash或明确内嵌，只存domain不可提交。ActorAttestationPolicy authoritative bytes的artifact hash固定`561d25223c507fe89f680ac5a1327c2c9500a85524e6929c90ae8d79eeef9cad`、domain hash固定`b370ab9a28a68c010297e706e71d62c063bc97167494a051439fc1b3328c0d90`；SignatureAlgorithmRegistryV1 artifact/domain固定为`43185e9a1a5bdf5a5ee4b6e3ce13a7c27852d1c8487ea9ec4125094cd54b5ac8`/`1419106463bf0193c2885c90ef68520914ea426d3c3ffe1848fce6b7acd77c40`。启动、proposal、grant、binding、active重放和signer均须对这些authoritative bytes与双hash自校验，缺失或错配即禁用governed waiver。

SignatureAlgorithmRegistryV1只允许两种精确验证profile：ECDSA为P-256/SHA-256、64-byte IEEE-P1363 `r||s`且low-S；RSA-PSS为SHA-256、MGF1-SHA256、saltLen=32、trailer=1且modulus≥2048 bits。签名字段必须为RFC 4648 URL-safe、无padding的strict canonical base64url；DER ECDSA、high-S、错长度/越界、RSA参数漂移、空白/`=`/非最短编码均拒绝，producer与validator不得宽松解码后再规范化。

### 7.3 垃圾回收

垃圾回收不在正常保存路径顺带执行。它是显式、可取消的 mark-and-sweep 维护任务：

- root 包含current/autosave/history、clock-state+全部epoch anchors/recovery、Gate/RuleCatalog/RuleSet/slice+observations/baseline/GovernanceApprovalEvidence/DecisionPolicy/ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1、三top-level registries、PriorityRegistry内requirements-change/rule-set-activation双heads及其record链、每个WaiverRegistry引用的ActorRegistry链，以及ActorRegistry锚定的OrganizationTrustEnrollmentV1、bootstrap/后继change authorization、current trust/revocation bundle双链、credential/certificate chain、defect/approval/waiver streams、WaiverProposalV1/attestation/binding/validation/ActorRole/clock evidence、OrganizationBundleFreshnessDeclarationV1/ReleaseGateCoverageReportV1、签名application/release manifests、导出/journal/pin；无governed waiver的release manifest freshness双hash为null，有waiver时声明及其current roots/ClockEvidence/签名链必须全部标记；
- 候选删除对象先进 `work/quarantine/gc-<operation-id>` 或产生可审核列表；
- 默认不删除任何已审批、导出证据或近期恢复可能需要的 Artifact；
- 运行前显示文件数、总尺寸和恢复窗口；
- 索引的引用计数只是优化，最终存活集必须从权威 manifests 计算。

## 8. 原子保存与日志

### 8.1 提交算法

```text
validate expected project revision, project lock, registry/stream/governance/clock heads
  → journal(state=preparing)
  → write/flush/verify staging artifacts and domain event blocks
  → freeze Gate/RuleCatalog/RuleSet/Dependency/Priority slice+observation/policy, governance/defect/decision/waiver/clock records and evidence
  → validate clock-state; reuse every unchanged registry root
  → only for the command-owned legal transition, build the corresponding next Priority/Approval/Waiver snapshot（ActorRegistry empty-genesis→enrollOrganization必须原子扎根enrollment+两bundle+authorization+actors）and ClockState anchor
  → promote every immutable artifact to CAS
  → write/flush/verify next revision manifest temp
  → atomic rename immutable revision manifest
  → atomic replace project.json pointer
  → journal(state=committed)
  → update rebuildable redb index
  → asynchronous safe staging cleanup
```

`project.json` 切换是提交点。它之前失败时 current revision 不变；指针切换后必须先把 journal 记为 `committed`，再更新可重建索引。此后 `redb` 失败时新 revision 仍有效，索引标为需重建。Windows 上原子替换实现必须在目标目录和防病毒占用场景进行故障注入，不将 POSIX rename 假设直接照搬。

该 Windows/NTFS 原子能力在 `F2S-TST-111` 以目标 API、flush/close 顺序、同卷约束、杀进程、磁盘将满和防病毒占用产生真实证据前，状态必须是 `UNVERIFIED`。文档中的“原子”是设计目标，不是已有实测结论；失败时只允许保持完整旧 current、完整新 current 或进入只读恢复，不能继续发布。

ApprovalRecord/AuditEvent/RequirementsChange/RuleSetActivation/Defect/ClockRecovery不得在指针提交后“补写”。本次命令的全部artifact/record先在staging验证schema/JCS、普通artifact hash、domain hash、prev artifact+domain双hash、registry/governance transition、ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1 authoritative bytes与双hash、严格签名profile/base64url、enrollment publisher签名/edition/expiry与同ProjectId固定性、proposalHash/签名credential/当前role evidence、clock pointer ancestry及引用可达性，再提升CAS；未变registry三元组必须逐字复用，只有拥有合法transition的命令才切换对应next root。PriorityRegistry transition还必须原子推进其requirements-change或rule-set-activation双head（不适用者必须不变）；ActorRegistry首次enroll若enrollment、两bundle、bootstrap authorization或initial actors任一失败则保持empty genesis，既有ProjectId不得用后继transition替换enrollment。RevisionManifest同步扎根ClockState checkpoint/recovery双heads及auditHeadArtifactHash。project revision、全局registry三元组、governance/target stream双heads与waiver/checkpoint/recovery expected heads任一不符则指针不切换、零事件。ClockRecovery遵循§6.3特殊顺序并产生连续跨epoch checkpoint；普通decision先推进sidecar再提交manifest anchor。JSONL/redb active均非真值。OrganizationBundleFreshnessDeclarationV1只在release candidate使用governed waiver时生成：每次签名作业创建唯一signingAttemptId，声明由current release-role credential签署并绑定同attempt的final manifest、candidate、enrollment、ActorRegistry、trust/revocation heads及同次ClockEvidence；`declaredLatestAt == ClockEvidence.effectiveNow`，最终签名前重读current effectiveNow并要求差值在`0..900s`。作业重启、attempt变化或超15分钟必须重新刷新bundle并签新声明。它记录操作者已按组织流程刷新和承认离线限制，不证明全局latest，也不得被旧candidate/旧attempt复用。

ClockRecoveryRecordV1必须持久化`commandKind=clockRecovery`、canonicalPayloadHash与committedProjectRevision；ActorRegistrySnapshotV1按上文保存六个command字段。所有canonical command payload都是含必填closed-enum `commandKind`的tagged JCS对象，tag明确属于hash域；redb幂等索引只能从全部append权威链重建`(actorId,idempotencyKey)→{commandKind,canonicalPayloadHash,resultKind,resultRef}`。仅kind与payload都相同才返回首次同类型record/snapshot并且registry/project不再推进；同key跨commandKind或同kind异payload均为different-payload conflict，零record、零snapshot、零registry/项目transition，禁止返回不同resultKind。ActorRegistry的canonical payload按18号固定投影计算，ClockRecovery及其他领域append按09号投影计算：command envelope中的裸expected-CAS heads不进入payload，`commandKind`绝不排除；Actor是明确例外地保留签名`authorizationArtifactHash/authorizationHash`作为业务语义引用，而authorization正文按18号exact schema有意绑定expected ActorRegistry heads，恢复器必须复算该传递绑定，不能以内层含head为由删掉authorization引用。不得用进程内缓存替代，也不得在迁移时重写首次committedProjectRevision。

### 8.2 OperationJournal

Journal 包含 operationId、kind、state、base/intended revision、idempotency key、必要的相对 staging refs、外部步骤状态和 heartbeat。不包含图像字节、PromptPack 正文、令牌、Spine 激活信息或非必要绝对路径。

应用不能只根据“临时文件存在”猜测提交状态。启动恢复按 journal 状态、manifest 完整性、CAS 哈希和 ProjectPointer 共同判定，详见 `F2S-DOC-RECOVERY-001`。

### 8.3 JobExecutionRecord 权威日志

每个 Job 的可恢复事实存于项目 `work/journal/jobs/<job-id>/`，而不是 `redb` 或 Worker 内存：

```text
jobs/<job-id>/
├── request.json                 # immutable request + idempotency + base revision
├── transitions/
│   ├── 000000000001.json       # immutable, previousHash chain
│   └── 000000000002.json
└── current.json                 # atomically replaced pointer to latest verified sequence/hash
```

每条 `JobExecutionRecord` 至少包含：

- schemaVersion、JobId、严格递增 sequence、previous record hash；
- requested method、normalized params hash、idempotency key、base Project revision；
- Provider/Worker/model/manifest/protocol provenance 与执行位置；
- 状态、时间、取消/中断原因、retry/resume parent JobId；
- checkpoint descriptor hash；
- `succeeded` 时必有已验证且已提升到 CAS 的 `JobOutputArtifact` hash；不得包含 candidate ID；
- `Register*Candidate` 成功后可由新的 Project revision 反向引用 JobOutputArtifact，但不改写 Job 终态记录。

写 transition 的顺序为：写/flush/验证不可变 record → 原子提升 → 原子替换 `current.json` → 更新 redb 投影。崩溃后依据连续 hash 链和 current pointer 恢复；无法证明顺序时由恢复器追加 `interrupted`，不能猜测 `cancelled/succeeded`。删除或损坏 `redb` 后，调度器从 request/transitions、OperationJournal 和 CAS 输出重建 queued/running/interrupted/terminal 投影；运行中进程不存在时不能恢复成 running。

未绑定 `JobOutputArtifact` 在用户显式丢弃或保留期到期前是 GC root。Job 记录不保存图片字节、完整 prompt、令牌或非必要绝对路径。

### 8.4 Autosave

- Autosave 使用正常 RevisionManifest 格式，`kind=autosave`，不另造一套不可迁移格式。
- Autosave 可保存候选编辑和未完命令边界，但不自动新建审批。
- 恢复 Autosave 时生成新 revision，不将原 Autosave 原位提升；下游审批按依赖重算。
- Autosave 保留数量/时间是用户可见配置，清理不能删除 current revision 引用。

## 9. 单写者锁

`work/project.lock` 包含 projectId、sessionId、PID、host fingerprint 的非敏感摘要、app version、openedAt 和 heartbeat。锁规则：

1. 正常情况下只有一个写会话；其他实例只读打开。
2. PID 不存在不能单独证明锁可直接删除，还需 heartbeat、session 和用户确认。
3. 网络盘/同步目录无法保证锁语义时显示持续警告，建议复制到本地。
4. 只读打开不创建持久写锁，不启动会修改项目的 Worker/GC/迁移。
5. 强制接管必须先生成恢复扫描和用户确认，并记录 AuditEvent。

## 10. `redb` 索引

### 10.1 允许存储

- project/revision 的查询投影；
- Artifact 元数据、引用计数缓存和缩略图映射；
- 资源树/全文搜索索引；
- Job 排队与 journal 的调度投影；
- 诊断和审批查询投影；
- schema 迁移和索引重建状态。

### 10.2 禁止成为唯一副本

- 源图/分层/mask/Rig IR/AnimationClip；
- GateReport/RuleCatalog/RuleSet/RuleSetActivation/RequirementsPriorityBaseline/RequirementsChange/GovernanceApprovalEvidence/ResolutionEvidence/DependencySnapshot/Priority slice+observation/DecisionPolicy/ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1/registry/defect/waiver/OrganizationTrustEnrollmentV1/Actor credential+trust/revocation bundle+WaiverProposalV1+attestation+role evidence/binding/validation/OrganizationBundleFreshnessDeclarationV1/ReleaseGateCoverageReportV1/ClockCheckpoint/ApprovalRecord/AuditEvent 的唯一副本或active真值；
- Worker 未完任务恢复所需的唯一信息；
- 导出清单和官方验证证据；
- 任何密钥、远程 Token 或 Spine 凭据。

IndexStore 打开失败时，项目仍能从 ProjectPointer、RevisionManifest、CAS descriptor 和 journals 以只读或降速模式打开，然后重建新索引。

`%LOCALAPPDATA%/<Publisher>/FlashToSpine/` 只保存：

logs/

### 11.1 Job sandbox 的 canonical 位置

本地 Worker 的任务目录固定在项目外：

```text
%LOCALAPPDATA%/<Publisher>/FlashToSpine/job-sandboxes/<job-id>/
├── in/          # Rust 材料化的只读输入副本
├── out/         # Worker 唯一可写输出区
├── tmp/         # 可丢弃临时数据
└── sandbox.json # 限额、状态和相对文件清单，不含凭据
```

这是 Job sandbox 的唯一 canonical 路径，禁止在 `.f2sproj` 中创建 `.jobs/` 或把项目根作为 Worker 工作目录。Project 内的 OperationJournal 只记录 job ID、输入 Artifact hash、协议状态和脱敏 sandbox reference。Worker 完成后，Rust 从 `out/` 重新校验媒体签名/schema/hash，通过正常 ArtifactStore 提交协议导入 CAS；不允许 Worker 直接提升。

## 12. 存储接口

### 12.1 ProjectRepository `F2S-IFC-401`

`F2S-IFC-401` 是工程中唯一名为 `ProjectRepository` 的基础设施 trait。`F2S-DOC-ARCH-001` 的 `F2S-IFC-002 ProjectPersistencePort` 是 Application 用例端口，由 `ProjectRepositoryAdapter` 映射到本文接口；不得再实现第二个不同签名的 Repository。恢复和索引重建分别调用 Recovery 用例与 `F2S-IFC-403`，不作为 Repository 的隐藏副作用。

```rust
trait ProjectRepository {
    fn inspect(&self, root: &ProjectRoot) -> Result<ProjectInspection, StoreError>;
    fn open(&self, root: &ProjectRoot, mode: OpenMode) -> Result<ProjectSession, StoreError>;
    fn commit(&self, session: &ProjectSession, tx: ProjectTransaction)
        -> Result<CommittedRevision, StoreError>;
    fn autosave(&self, session: &ProjectSession, tx: ProjectTransaction)
        -> Result<AutosaveRevision, StoreError>;
    fn close(&self, session: ProjectSession) -> Result<(), StoreError>;
}
```

`inspect` 是只读安全扫描，不自动迁移/恢复。`open(ReadWrite)` 必须成功获取单写者锁；失败时可明确选择 `ReadOnly`，不偷偷共享写入。

### 12.2 ArtifactStore `F2S-IFC-402`

```rust
trait ArtifactStore {
    fn stage(&self, op: OperationId, source: ArtifactSource)
        -> Result<StagedArtifact, StoreError>;
    fn verify(&self, staged: &StagedArtifact, policy: VerificationPolicy)
        -> Result<VerifiedArtifact, StoreError>;
    fn promote(&self, verified: VerifiedArtifact)
        -> Result<ArtifactDescriptor, StoreError>;
    fn open_read(&self, hash: &ContentHash) -> Result<ArtifactReader, StoreError>;
    fn mark_for_gc(&self, roots: &[RevisionRef]) -> Result<GcPlan, StoreError>;
}
```

UI/Worker 不获得 `promote`。Worker 输出先作为受控 staging source 进入 `verify`，验证后才能提升。

### 12.3 IndexStore `F2S-IFC-403`

```rust
trait IndexStore {
    fn status(&self) -> IndexStatus;
    fn apply_revision(&self, revision: &RevisionManifest) -> Result<(), IndexError>;
    fn rebuild(&self, source: &dyn ProjectTruthReader) -> Result<IndexReport, IndexError>;
    fn query(&self, query: ProjectQuery) -> Result<QueryResult, IndexError>;
}
```

`apply_revision` 失败不回滚已原子提交的 ProjectPointer，而是将 IndexStatus 标为 stale/corrupt 并安排重建。

### 12.4 MigrationRegistry `F2S-IFC-404`

```rust
trait MigrationRegistry {
    fn plan(&self, inventory: &SchemaInventory, target: &SchemaTarget)
        -> Result<MigrationPlan, MigrationError>;
    fn execute(&self, source: &ReadOnlyProject, plan: &MigrationPlan,
               staging: &MigrationStaging)
        -> Result<MigratedCandidate, MigrationError>;
    fn verify(&self, candidate: &MigratedCandidate)
        -> Result<MigrationVerification, MigrationError>;
}
```

迁移不原位修改 source Project，不同时打开 Worker 或执行导出。它先产生候选和报告，通过验证后以新 revision 提交。

## 13. Schema 版本和迁移

### 13.1 版本语义

- patch：格式与语义兼容的澄清/验证修复；
- minor：有明确默认值的新可选字段，或老 reader 可安全忽略的增量；
- major：删除/改义、单位/坐标改变、身份/引用规则改变，必须显式迁移。

新字段只有在“老应用忽略后仍不会破坏数据”时才能定义为 minor。审批、像素 provenance、坐标单位和导出语义的改动默认是 major。

### 13.2 链式迁移

```text
1.0 → 1.1 → 1.2 → 2.0
```

不为每个历史版本实现一条直跳最新的散乱路径。每一步迁移必须：

- 只读旧输入，新建候选 Artifact/manifest；
- 在类型层显式处理每个字段，不以无类型 JSON patch 取代语义迁移；
- 产生源/目标哈希、警告、默认值和审批失效报告；
- 通过 schema、领域不变量、Artifact 引用、重要预览和导出 preflight；
- 反复执行幂等，中途失败不改变旧项目；
- 只在用户确认后提交新 migration revision。

### 13.3 审批在迁移中的处理

- 字节变化但语义经证明等价时，迁移报告记录旧/新哈希映射，审批可保留但必须有专用迁移规则证据；
- 坐标、像素、Rig 结构、动画时间或游戏 marker 语义发生改变时，相关审批必须 stale；
- 无法证明等价时默认 stale，不以“尽量保留”为由假设有效；
- 旧审批若缺DependencySnapshot、RuleCatalog/RuleSet、认证priority slice+observation、stable inputs/outcome、DecisionPolicy正文/hash、精确ack或static waiver binding，只能原样保留为历史并标stale/inactive。旧schema若缺PriorityRegistry内requirements-change/rule-set-activation heads、RequirementsPriorityBaseline/RuleFamilyMigration/GovernanceApprovalEvidence/ResolutionEvidence、ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1正文及双hash、OrganizationTrustEnrollmentV1、Actor credential/current trust+revocation bundle/WaiverProposalV1/三签名role evidence、三registry roots、clock checkpoint/recovery双hash或跨epoch链证明，显式迁移只能建立新roots/epoch，旧waiver bindings及依赖它们的approvals inactive；不得从当前应用配置补写policy/algorithm registry proof，也不得为旧ProjectId合成/替换enrollment、active角色、bundle head、proposal、priority floor、迁移head或当时不存在的证明。ActorRegistry sequence>0或ClockRecovery/其他append record缺commandKind/canonicalPayloadHash/committedProjectRevision等幂等字段时不得合成active transition或首次结果，只能保留历史并要求新命令；非法ULID不得大小写/别名归一化或补位，引用该ID的对象保持历史不可写。无enrollment或enrollment过期的旧项目只能禁用governed waiver；若需要该能力，显式克隆到新ProjectId、重新enroll并重批。依赖governed waiver的旧release candidate若缺OrganizationBundleFreshnessDeclarationV1、signingAttemptId或`0..900s`时间窗证明，只能保留历史且不得promotion/re-sign；迁移不得合成声明、延长窗口或复用旧attempt，无waiver candidate则声明双hash保持null；
- 迁移本身不创建新艺术/玩法审批。

### 13.4 未来版本和降级

当项目或任一关键子 schema 主版本高于当前应用：

- 禁止写入、迁移和导出；
- 尽可能显示安全可读的项目 ID/名称/版本，未知字段不映射为默认值；
- 允许原样复制项目和生成不含私密内容的诊断；
- 提示安装兼容的新应用版本。

不支持就地“降级保存”。如未来实现向旧格式导出，它必须是新 ExportAdapter，输出到新目录，有明确丢失报告。

## 14. 导出事务与项目关系

- 每次导出绑定精确 Project revision、ExportProfile revision、adapterVersion 和 preflight report hash。
- PSD/PNG/Spine JSON/提示词包/报告先写 `work/staging/<operation-id>`；必选产物全部验证后提升到 `exports/<export-id>`。
- 上一个成功 export 目录不覆盖；相同输入可生成语义稳定的内容 ID，运行记录仍保留自己的 operation ID。
- canonical产物字节、清单和哈希始终完整保存在项目内`exports/<export-id>`。用户选择项目外目录时，项目在内部提交成功后创建独立`PublishAttempt`：在外部root内部的`.f2s-publish-<operation-id>`逐文件复制、flush、hash校验，再同卷rename为版本目录；禁止从项目staging跨卷rename。外部目录不是项目真值，失败/取消只把PublishAttempt记为failed/cancelled并可从内部snapshot重试，不能把已成功Export改为失败或留下无报告半成品。
- Spine.com 临时生成的 `.spine` 只能在 staging 用于验证，本工具不自行写入其内容。

### F2S-SCH-010：PublishAttempt持久记录与恢复

每次项目外发布使用append-only transition链，不存在可原位覆盖的`publish-attempts/<attempt-id>.json`。首记录冻结`schemaVersion`、符合`^[0-7][0-9A-HJKMNP-TV-Z]{25}$`的ULID `attemptId`、`idempotencyKey`、`sourceExportId`、`sourceManifestHash`、本机设置中的`externalRootRef`（项目中不保存可分享绝对路径）、目标版本相对名、外部volume identity、`tempRelativePath`、payload inventory/size/hash、createdAt和可空`retryOf`。每条transition包含`attemptId/sequence/prevTransitionHash/state/occurredAt/errorCode/externalReceiptHash`及必要的root重定位事实，序列化为不可变CAS对象；RevisionManifest的`publishAttemptHeads[attemptId]`只通过11号正常提交算法切到新head。命令必须同时携带`expectedProjectRevision/expectedHeadHash/expectedSequence`，任一不匹配则不追加、不覆盖；非法或经alias/case规范化才合法的attemptId在任何阶段都拒绝。

状态只允许`requested→copying→verifying→committed`；`requested/copying/verifying`均可进入非终态`recovery_pending`，重新定位并校验后可回到`copying/verifying`或在已有有效receipt时追加`committed`。活动态可到终态`failed/cancelled`；`committed/failed/cancelled`不可原位重开，重试创建新attempt并引用`retryOf`。root缺失、换盘、进程终止或提交暂时冲突只能进入`recovery_pending`，不能先写terminal再复活。`idempotencyKey`绑定source manifest、externalRootRef和目标版本名；同key已committed时返回既有结果，禁止覆盖不同内容。

外部临时目录在rename前写入`publish-receipt.json`。`payloadInventoryHash`仅覆盖待发布payload文件，明确排除receipt和临时元数据；receipt再包含attemptId、sourceExportId、sourceManifestHash、payloadInventoryHash和目标名，并单独计算`externalReceiptHash`。审计总摘要按域分隔的`SHA256("f2s-publish-v1" || payloadInventoryHash || externalReceiptHash)`计算，不存在自引用。payload与receipt全部flush后才同卷rename。rename成功时项目head仍可处于`verifying`；Rust随后以expected revision/head追加`committed + externalReceiptHash`。若在rename后、项目提交前崩溃，启动恢复通过已登记root和receipt重新校验，从`verifying/recovery_pending`追加committed，或显示人工处理；不得猜测成功、删除完整目录或回滚内部Export。绝对root只保存在本机配置，不可达时保持`recovery_pending`并提示重新定位。

## 15. 备份、打包和交换

项目交换包是独立 Export/Import 用例，不把工作目录改成长期 ZIP 数据库。

- 打包前冻结所需 revision root，通过 manifests 收集 CAS 存活集；
- 默认排除缩略图、可重建 index、过期 staging、未选择日志和凭据；
- 包内带 inventory、每文件哈希、schema 和许可/来源摘要；
- 导入时解压到新 staging，防止 path traversal、超限尺寸、重复路径和哈希不符；
- 验证完整后由用户选择新 `.f2sproj` 目录，不覆盖现有项目。

如所选压缩库/格式许可不在批准的宽松白名单中，打包功能不进入发布。V1 不依赖打包交换来满足项目可搬移性；目录本身已可搬移。

## 16. 失败与恢复边界

| 失败 | 存储层结果 | 不得执行 |
| --- | --- | --- |
| 预期 revision 冲突 | 提交前失败，无新指针 | 覆盖 current |
| 磁盘不足/权限 | 保留旧 current，安全清理本操作 temp | 删除不相干 Artifact 腾空间 |
| CAS 已有同哈希但字节不符 | 严重完整性错误，隔离并停止写 | 以长度或文件名猜测复用 |
| revision manifest 已写但指针未切换 | 作为孤立恢复候选 | 启动时无用户确认自动提升 |
| 指针已切换但 index 失败 | 新 revision 有效，索引待重建 | 回滚已提交事实去迁就缓存 |
| 未知 schema major | 只读检查 | 强制打开且保存 |
| 迁移中断 | 删/隔离 migration staging，旧项目不变 | 原位局部修复旧 manifests |
| `redb` 损坏 | 保留损坏副本到 quarantine，从真值重建 | 删除 CAS 或认为项目丢失 |
| 工程多实例 | 一个写者，其他只读 | 在应用层“最后写入胜出” |

## 18. 许可、隐私与数据保留

- `redb` 为 MIT/Apache-2.0 候选，实际锁版后需运行 Cargo 许可和传递依赖门禁。
- 项目数据默认不加密，与普通用户文件具有相同的 Windows 权限边界；UI 必须如实说明，不将 CAS 哈希宣传为加密。
- 凭据不进项目。与私有远程的任务记录只保存端点 profile ID、发送 Artifact 哈希/类型、批准记录、结果哈希和删除回执，不存 Token。
- 系统不以“节省空间”为理由自动删除用户源图、旧批准版、导出证据或未过保留期的恢复对象。
- 项目打包所用的压缩算法/库必须通过宽松许可审计；不因格式普及性引入未批准的 LGPL/GPL/CPL 组件。

## 19. 完成定义

1. `.f2sproj` 目录可搬移，不依赖全局数据库和旧绝对路径。
2. 导入、编辑、Autosave、审批、导出和迁移都通过统一 revision/CAS/Journal 提交模型。
3. 任意保存步骤崩溃后不会得到半写 current；该结论在 `F2S-TST-111` 真实 Windows/NTFS 证据存在前保持 `UNVERIFIED`。
4. 删除或损坏 `redb` 可重建，不丢真值。
5. 所有关键 schema 有精确版本、fixture、迁移链和未来版本只读行为。
6. 审批、像素 provenance、Worker 结果和 Spine 验证证据都可追溯到精确输入哈希。
7. 应用卸载、清缓存、Worker 损坏和远程失联不删除用户项目。
