# 领域逻辑与工作流编排

## 1. 目的和领域边界

本文档定义 Production Assist 的业务对象、不变量、人工门禁、审批失效、工作流编排和幂等命令。它回答“什么时候一个结果有资格成为下一阶段的输入”，而不回答文件如何落盘或 Worker 如何启动。

领域不包含：

- 第三方生图 API 调用；
- AI 自主批准母版、Rig、关键姿势或命中帧；
- Unity、Godot、Cocos 或自研引擎运行时逻辑；
- Spine Runtime 语义、`.spine` 私有格式或 Spine 激活管理；
- 对用户未提供的动作视频、BVH 或 3D 资产的假设。

ADR owner按 `F2S-ADR-REGISTRY-001` 逐项解析：本文只读引用跨领域ADR，并唯一拥有frontmatter枚举的三个DOMAIN implementation-local ADR canonical body。

## 3. 领域决策

以下三项是领域编排子系统的 implementation-local ADR；canonical body 与语义由本文维护。`F2S-DOC-ADR-001` 只在全局 ADR owner registry 登记归属并提供跨文档索引，不复制或改写这些决定。

| 决策 ID | 决策 | 原因 |
| --- | --- | --- |
| `F2S-ADR-DOMAIN-001` | 不存储一个线性“当前阶段”作为项目真值；每类产物独立拥有状态，工作流可用性由 GateEvaluator 派生 | Production Assist 允许回退、并行修复和部分失效，单一阶段枚举会丢失信息 |
| `F2S-ADR-DOMAIN-002` | 审批绑定产物内容哈希、revision、规则版本和下游依赖快照 | 名称或当前指针不足以证明用户批准了哪个结果 |
| `F2S-ADR-DOMAIN-003` | 后台 Job 成功只产生“已验证并持久到 CAS、但尚未绑定领域对象”的 `JobOutputArtifact`；另一个带 expected revision 的 `Register*Candidate` 事务才能产生 candidate | 技术执行成功不等于领域注册成功，更不等于艺术、动画或玩法批准 |

## 4. 与 04 完全一致的五维状态模型

`F2S-DOC-UX-001` 是工作流词汇 owner。领域只保存必要事实或可重建投影，禁止把五个维度压成一个枚举。

### 4.1 工作流阶段

项目阶段严格派生为：

`draft → master_approved → content_approved → layers_approved → rig_approved → motion_review → gameplay_approved → export_ready → exported`

`spine_verified` 不属于阶段。阶段表达已到达的生产里程碑；上游变化通过新鲜度和阻塞维度表达，不伪造历史倒退，也不持久第二套项目阶段真值。

### 4.2 对象生命周期

```text
missing
  └─ draft
       └─ candidate
            ├─ rejected
            ├─ approved
            └─ superseded

approved
  └─ superseded
rejected
  └─ draft
```

1. `candidate` 可预览、修改、比较，不能进入 release export。
2. `approved`必须存在匹配target、DependencySnapshot、stable inputs/outcome、DecisionPolicy及active ack/static waiver bindings，且尚未被revoke的active approve；transport GateReport与全局revision只用于当次审计/CAS；reject/acknowledgeWarning/revoke绝不单独满足批准条件。
3. `rejected` 保留原因和证据，可派生新 `draft`；`superseded` 只读。
4. 生命周期中不存在 `stale`、`blocked`、`failed` 或 `unverified`。

### 4.3 新鲜度、阻塞与导出验证

| 维度 | 允许值 | 领域来源 |
| --- | --- | --- |
| Freshness | `current / stale` | InvalidationEngine 比较精确依赖 revision/hash；`stale` 不改生命周期 |
| Blockage | `ready / blocked` | GateEvaluator 根据 blocker 推导；必须携带 issue ID 与解除条件 |
| ExportVerification | `not_exported / exported_unverified / verification_running / spine_verified / incompatible` | ExportRun 与真实 4.2.43 evidence；Job 失败不等于 verification 值 |

Job 的 `requested/queued/running/cancel_requested/cancelled/failed/succeeded/interrupted` 是执行维度，不覆盖对象生命周期。UI 组合投影必须分别给出 stage/lifecycle/freshness/blockage/export verification。

### 4.4 唯一映射

| 领域事实 | 04 投影 | 规则 |
| --- | --- | --- |
| 必需对象不存在 | Lifecycle=`missing` | GateEvaluator 推导 |
| 未提交工作版 | Lifecycle=`draft` | 可持久 autosave，但不是 candidate |
| `Register*Candidate` 事务成功 | Lifecycle=`candidate` | Job `succeeded` 本身不产生此状态 |
| candidate 满足审查前置 | Lifecycle=`candidate` + ReviewReadiness=`needs_review` | `needs_review` 只由 GateReport 推导为 UI 视图，不是 Lifecycle 或额外存储终态 |
| 当前target有active `approve` record | Lifecycle=`approved` | record必须由ApprovalStreamRegistry的target head重放为最新有效批准：stable inputs/outcome、DependencySnapshot、PriorityRegistry、policy、ack及每个静态waiver binding的当前动态验证仍匹配，且无后继revoke；不比较全局revision |
| 最新决定为`reject` | Lifecycle=`rejected` | 保留candidate与审批流历史，不伪造approved |
| 当前批准被`revoke` | Lifecycle=`candidate` | 若对象已superseded则保持superseded；精确下游Freshness=`stale`且release gate重算 |
| approved 依赖改变 | Lifecycle=`approved` + Freshness=`stale` | 保留审批历史 |
| GateReport 有 blocker | Blockage=`blocked` | 不改生命周期 |
| Job 失败 | Job=`failed` | 不改对象或导出验证 |
| 已输出但无外部证据 | Stage=`exported` + ExportVerification=`exported_unverified` | 只有真实证据可转 `spine_verified` |

## 6. 依赖图和失效传播

```text
SourceImage
   └─ MasterRevision (approved)
       ├─ CharacterSpec / StyleSpec / WeaponSpec / RuntimeSpec
       │   └─ MotionSpec[10]
       │       └─ AssetSpec[]
       │           └─ PromptPack
       └─ LayerSet
           └─ RigProject
               └─ AnimationClip[10]
                   ├─ GameplayMarker / approvals
                   └─ Diagnostics
                       └─ ExportRun
```

实际图保存精确边，上图只是主干。`InvalidationEngine` 使用产物的 `derivedFrom` 和变更类型计算影响：

| 变更 | 必须失效 | 可保持有效 |
| --- | --- | --- |
| 替换/修改批准母版 | LayerSet、RigProject、全部 Clip、marker、诊断、导出 | 项目身份、用户设置；StyleSpec 需用户重新确认映射 |
| StyleSpec 线条/色彩规则变更 | PromptPack、相关新图候选、LayerSet QA、导出报告 | Rig 几何可保留但标记需重渲染检查 |
| WeaponSpec 具体化 | attack AssetSpec、PromptPack、grip/contact、attack Clip marker | idle/run 等无武器依赖内容，由实际边决定 |
| AssetSpec 变更 pivot/尺寸/父骨 | 相关 Layer attachment、Rig 映射、使用该素材的 Clip | 无使用关系的动作 |
| Rig 骨骼结构变更 | 受影响轨道、marker、诊断、导出 | 能精确重映射的不相干 Clip，必须有映射报告 |
| Clip 时长/关键帧变更 | 该 Clip 的关键姿势、命中帧、接触、QA 和导出 | 其他 Clip |
| 诊断规则版本变更 | 相关 DiagnosticRecord 和 QA 门禁 | 艺术产物本身 |

失效计算必须在命令提交前向用户展示摘要，提交时与主修改位于同一 ProjectTransaction。

## 7. 人工门禁

| 门禁 | 输入 | 必显示的检查 | 通过后产物 |
| --- | --- | --- | --- |
| Master Gate | 导入原图、规范化候选 | 尺寸、alpha、色空间、方向、地面、差异 | approved MasterRevision |
| Spec Gate | Character/Style/Weapon/Runtime/Motion | schema、未决字段、十动作覆盖 | approved spec snapshots |
| Asset Plan Gate | AssetSpec 集合 | 动作覆盖、重复、无使用、表现策略、placeholder | approved AssetPlan |
| Layer Gate | LayerSet | 重组差异、洞、边缘污染、重叠、AI 像素 | approved LayerSet |
| Rig Gate | RigProject | 成环、孤儿、mesh、weight、draw order、临时动作试跑 | approved RigProject |
| Pose Gate | AnimationClip | 动作阶段、轮廓、脚/武器接触、循环或终止 | approved clip revision |
| Gameplay Gate | GameplayMarker | 命中帧、anticipation/active/recovery、weapon_contact | approved markers |
| Export Gate | 全部产物 | 缺失/过期审批、诊断、兼容、许可、目标版本 | release export 资格 |

退回门禁必须包含原因、对象链接和建议修复。“我知道了”不是审批；接受艺术warning必须产生kind=`acknowledgeWarning`的单独ApprovalRecord，它永不构成active approve；error不可越过。

## 8. GateEvaluator 接口 `F2S-IFC-201`

### 8.1 诊断严重度、需求优先级与分区

`severity=error|warning|info`是04号文档的诊断词汇；每个关联需求或缺陷分别用`priorityBindings[{subjectKind,subjectId,priority,sourceArtifactHash}]`保存，禁止把多subject压成一个“最高优先级”字段。binding按`subjectKind/subjectId`全序且该二元键唯一；同一subject重复、同时出现p0/p2、source hash不同或与当前PriorityRegistry entry不逐字段相等，整份报告拒绝。

`PriorityRegistrySnapshotV1.registryHash = SHA256("f2s-priority-registry-v1\0" || JCS(snapshot without registryHash))`，entries按`subjectKind/subjectId`全序唯一，普通CAS artifact hash另存。genesis固定registrySequence=0且previous artifact/domain均null并含当前签入baseline；requirements-change与rule-set-activation的sequence均为0且双head均null。后续必须sequence+1，previous artifact/domain分别等于当前snapshot的CAS/domain hashes；两个治理head在未发生相应迁移时逐字复用，发生迁移时只允许对应sequence+1并以artifact/domain双hash指向合法next record。普通变更一次只可新增一个sequence=1 open defect，或用同defectId的下一条DefectRecord更新一个entry；新record的previous artifact/domain必须等于旧source record，`p2→p1/p0`与`p1→p0`可收紧，`p0→p1/p2`或`p1→p2`禁止。resolved必须保持原/更严格priority并含resolution evidence；resolved entry永久保留且不能供finding binding，同origin复发只能在原stream reopen并继承历史floor，只有真正不同origin key才新建defectId。requirements baseline整体变化只能在应用/项目显式迁移事务中进行并保存`RequirementsChangeRecordV1`、产品+质量批准、旧/新baseline双hash与逐项diff，不能与defect变化混合；移出范围的ID不删除而转为`retired` tombstone并保留原/更严格priority，retired不能供finding/waiver binding，重新启用也不得降低历史priority。普通entry删除、active P0/P1降级、sequence倒退、任一previous不符、无source变化却改priority均拒绝。active requirement entry只能来自02号canonical FR/NFR表，由固定generator生成、冲突为零、签入应用manifest并以schema/generator/source-doc artifact/domain hashes证明；21号只校验镜像一致性。defect entry只能引用项目CAS内上述不可变`DefectRecordV1`。GateReport transport保存完整registry observation；stable DependencySnapshot只保存实际rule用到的active认证slice。缺subject、retired/旧/不可达registry、source不可达、resolved source或priority不等均fail closed。

`RequirementsPriorityBaselineV1.baselineHash=SHA256("f2s-requirements-priority-baseline-v1\0" || JCS(baseline without baselineHash))`，完整bytes普通SHA-256为baseline artifact hash；entries按subjectId排序唯一且必须与02号active FR/NFR ID及priority恰好全等，不能含retired（retired只存在项目PriorityRegistry历史）。DocumentRevision固定grammar为`(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)`并逐字等于02号frontmatter，禁止静默补SemVer第三段。generator和source document均保存artifact/domain双hash；最终应用manifest签入baseline双hash，启动时重跑或用golden验证generator输出，不接受21号镜像作为输入。

`RequirementsChangeRecordV1.recordHash=SHA256("f2s-requirements-change-record-v1\0" || JCS(record without recordHash))`，完整bytes普通SHA-256为artifact hash；genesis/后继的sequence与previous双hash规则同DefectRecord。`diffs`按subjectId排序唯一，每项完整保存old/new status、priority和source双hash；新增的old字段为null，retire的new status只能retired且priority不得变弱，重新active继承历史floor。record以`proposalCreatedAt`映射proposal的`proposedAt`，其余字段逐项唯一重建`RequirementsChangeProposalV1`，且`proposalHash=SHA256("f2s-requirements-change-proposal-v1\0" || JCS(proposal without proposalHash))`。product与quality两份`GovernanceApprovalEvidenceV1`必须是不同actor且不同credential、同proposalKind/hash，并满足`proposalCreatedAt <= each signedAt <= record.occurredAt`；签名消息固定为`UTF8("f2s-governance-approval-v1\0") || UTF8(proposalKind) || 0x00 || hexToBytes(proposalHash) || 0x00 || UTF8(actorId) || 0x00 || UTF8(role) || 0x00 || UTF8(signedAt)`。证据以artifact hash取回并重算`evidenceHash=SHA256("f2s-governance-approval-evidence-v1\0" || JCS(evidence without evidenceHash))`。certificate chain artifact必须可取且解析SPKI/anchor/algorithm逐字段匹配，governance trust bundle双hash必须等于签名application manifest允许的current bundle，再验证签名；只给credentialId/hash而无公钥材料拒绝。旧/新baseline也必须双hash可取。PriorityRegistry内current governance head使该链间接进入RevisionManifest/GC；迁移snapshot一次只能应用该record声明的精确diff。

RequirementsChangeRecordV1固定`commandKind=requirementsChange`，RuleSetActivationRecordV1固定`commandKind=ruleSetActivation`；各tag必须进入对应canonical command payload JCS/hash并与record schema逐字段一致。tag缺失、未知或错variant时，即使其余字段与hash链合法也拒绝。

`PrioritySliceV1.sliceHash=SHA256("f2s-priority-slice-v1\0" || JCS(slice without sliceHash))`；entries按subjectKind/subjectId排序唯一且必须恰等于所有finding经RuleDescriptor解析的并集，不得缺、多或包含未使用entry。Assembler先用observation.artifactHash取回完整snapshot，重算artifact/domain hash与sequence，再逐项证明slice成员相等；Approval active replay则从**当前**manifest registry加载相同subject并比较priority/source artifact。全局registry sequence/root只属于transport观察，不进入DependencySnapshot/EvaluationInputs/Outcome或binding持续有效性键；因此无关defect新增/收紧不会使本gate失效，关联entry变化一定改变slice或使重放失败。

`DefectRecordV1.recordHash=SHA256("f2s-defect-record-v1\0" || JCS(record without recordHash))`，完整record bytes普通SHA-256为recordArtifactHash且不写入自身。DefectStatus只允许`open,resolved`，未知值拒绝；genesis固定sequence=1/previous artifact+domain均null，后继严格n+1并双指向前一record。每条append治理record都保存actorId/commandKind/commandId/idempotencyKey/canonicalPayloadHash/committedProjectRevision；`canonicalPayloadHash=SHA256("f2s-command-payload-v1\0" || JCS({commandKind,...variantSemanticFields} excluding expected-CAS heads))`，重建索引按`(actorId,idempotencyKey)`找到首次record，kind与payload均相同才返回它；同key跨kind或同kind异payload均零事件拒绝，commandId全项目唯一。`originKeyHash=SHA256("f2s-defect-origin-v1\0" || JCS({ruleFamilyId,targetStableIdentity,locationIdentity}))`，defectId由完整origin key确定且PriorityRegistry对originKey唯一；同一origin不能换ID重置。ruleSetHash/ruleId/version只作当次审计，普通规则升级必须保留ruleFamilyId。每条后继priority必须不弱于该stream历史最严格值。resolved后同origin再次命中只能在原stream追加open record，priority取历史最严格值与descriptor default中更严格者。数组/引用按schema canonical key排序，所有record进入`defect-record-v1.schema.json`与Rust/TS golden。

open DefectRecord的resolution evidence artifact/domain字段必须同时为null；resolved时两者必须同时非null，用artifact hash取回正文后重算两hash。`ResolutionEvidenceV1.evidenceHash=SHA256("f2s-resolution-evidence-v1\0" || JCS(evidence without evidenceHash))`，完整bytes普通SHA-256为artifact hash；它双绑定current defect head、RuleSet、target和origin。`negativeMatch` variant必须恰有negative-match report双hash且scope-removal双hash为null；`approvedScopeRemoval`反之并引用current RequirementsChangeRecord；两个成员缺一、同时存在、自由文本或旧head均拒绝。DefectRecord/FindingDraft都双引用origin RuleSet artifact/domain；只存domain hash拒绝。

`RuleCatalogV1.catalogHash=SHA256("f2s-rule-catalog-v1\0" || JCS(catalog without catalogHash))`，完整bytes普通SHA-256为catalog artifact hash。allowedRuleSets按version/artifact/domain排序唯一，各entry descriptors按ruleFamilyId/ruleId/version排序唯一，familyMigrations按migrationId排序唯一，from/to family数组分别排序唯一；每个entry的descriptorManifestHash重算自其descriptors并逐字段（含supersedesRuleFamilyIds）等于RuleSet正文。每个非空supersedes family必须在catalog中恰有一条方向一致的migration覆盖，migration不得成环、冲突或遗失旧family。catalog固定记录generator和**构建输入**sourceRuleManifest双hash，该输入不是最终应用manifest，避免自引用；catalog artifact/domain双hash再签入已签名应用manifest并在启动自校验。项目把实际RuleSet artifact复制进CAS。Register/Resolve/Activate只接受catalog成员，新增family、修改default或迁移必须随签名应用版本及产品+质量批准证据发布，项目/UI不能注入自定义RuleSet。

RuleSet不是只有版本字符串。`RuleSetSnapshotV1.ruleSetHash=SHA256("f2s-rule-set-v1\0" || JCS(snapshot without ruleSetHash))`，普通artifact hash另存；descriptor按ruleFamilyId/ruleId/version排序唯一。`fixedRequirements`列出必须精确绑定的requirement IDs，assembler要求finding bindings与这些ID及当前baseline逐项全等；`noneArtAdvisory`只允许category=artAdvisory且bindings为空；`governedDefect`要求恰一个current open DefectRecord，其ruleFamily/target identity/originKey及latest finding fingerprint与当前draft逐项相等且priority不得弱于descriptor default或历史floor。缺绑、多绑、换绑任意真实P2 subject、使用resolved/旧defect source或descriptor/category不符均拒绝。`findingFingerprintHash=SHA256("f2s-finding-fingerprint-v1\0" || JCS({ruleId,ruleVersion,target,location,expected,actual,evidenceHashes}))`，明确排除priority binding、findingId和本地化文本，避免循环；origin key另按上一段排除易变actual/evidence以防轻微变化重置priority。rule family拆分/合并必须有`RuleFamilyMigrationV1`，并通过`RuleSetActivationRecordV1`把每个项目内旧origin映射到新origin、引用原defect head并继承祖先最严格floor；存在未resolved旧defect却缺映射时新RuleSet不可激活，禁止靠换rule/version/family回到default P2。

`RuleSetActivationRecordV1.recordHash=SHA256("f2s-rule-set-activation-record-v1\0" || JCS(record without recordHash))`，完整bytes普通SHA-256为artifact hash；record链previous双hash连续。originMigrations按oldOriginKeyHash/newOriginKeyHash排序唯一，每项含old/new origin完整组成、defectId、old defect head双hash、new appended defect head双hash和inheritedPriorityFloor；未变化family也必须检查defaultPriority。record必须能按字段逐项唯一重建`RuleSetActivationProposalV1`，且`proposalHash=SHA256("f2s-rule-set-activation-proposal-v1\0" || JCS(proposal without proposalHash))`；两份GovernanceApprovalEvidence按上一段同消息算法签该proposal，错kind/hash、同credential或跨record复用拒绝。激活事务先为所有current open defect生成必要后继：若新default更严格，即使fingerprint未变也追加同stream、同open status且收紧priority；default放宽仍保留历史floor。随后以一个专用`ruleSetActivation` PriorityRegistry transition精确批量替换这些entry，并与新RuleSet/current activation head在同一RevisionManifest原子切换；不得混入普通defect/baseline变化。任一漏项、额外项、弱化、未签名catalog或批准证据失败则保持旧RuleSet/registry全不变，因此不会形成“新规则已生效但旧priority无法登记”的死锁。

一次activation可能生成多个DefectRecord，但用户`(actorId,idempotencyKey)`只归属唯一RuleSetActivationRecord。每个派生DefectRecord使用`commandKind=derivedRuleSetActivationDefect`、`commandId=UUIDv5(parentActivationCommandId, defectId)`与`idempotencyKey="derived:" + lowerhex(SHA256("f2s-derived-idempotency-v1\0" || parentActivationCommandId || 0x00 || defectId))`，其canonicalPayloadHash显式包含commandKind/parent commandId/proposalHash/defectId/new head；这些内部key不能由外部请求提交。由此空redb后既能从父record返回用户首次结果，又能验证每个child唯一，不会因批量迁移共享一个key而冲突。

`governedDefect`命中但没有匹配current open DefectRecord时不伪造GateFinding：evaluator产生decision-free `FindingDraftV1`（rule/target/location/expected/actual/evidence/fingerprint），用`draftHash=SHA256("f2s-finding-draft-v1\0" || JCS(draft without draftHash))`保存CAS，返回`DefectRegistrationRequired`而不生成可审批GateReport。用户在问题中心执行`RegisterDefectFromFindingDraft`：首次origin以descriptor.defaultPriority创建sequence=1 open record；已有open同origin+同fingerprint仅在current.priority不弱于当前catalog descriptor default时幂等返回current；若default已收紧则必须在原stream追加同fingerprint、收紧priority的record并以合法registry transition提交；已有open但fingerprint变化或resolved复发同样追加并保持历史priority floor。脚本、AI或UI payload不能指定priority。`ResolveDefect`只接受当前重评中该origin不再命中的`ResolutionEvidenceV1`或已批准范围移除record，priority不变/收紧；任意自由文本evidence不能关闭。成功后必须重新评估。错rule/target/origin/fingerprint、旧draft、跳过登记直接申请waiver、同origin换ID或resolve→recur降级均零事件拒绝。

三组finding ID必须分别按FindingId字典序，且是`findings`的完整无交叉分区：error只进入`blockerFindingIds`，warning只进入`warningFindingIds`，info只进入`infoFindingIds`；无悬空、重复、跨组或乱序ID。`blockage=blocked iff blockerFindingIds`非空。

V1决策矩阵取最严格结果：error恒为`forbidden`；任一P0/P1或类别为`compatibility/scopeClaim/security/privacy/license/dataIntegrity/recovery/signature/updateRoot/workerIsolation/spineClaim/userData`时，规则必须产生error+forbidden；warning+非空bindings且全部P2+非硬类别才可`governed`；warning+bindings为空+`artAdvisory`才可`userAcknowledgement`；info不产生通过义务且只能`forbidden`。不满足矩阵的报告无效，不能靠UI降级。`status=reviewable`只表示target/current schema具备人工审查前置，不代表已批准；`notReviewable`不可决定。

### 8.2 Wire registry 与规范 JSON

V2 hash DTO不是Rust内存结构的默认serde。所有实现使用独立schema、UTF-8/NFC、lowerCamelCase、`denyUnknownFields=true`；某variant/schema内声明的可空字段显式JSON `null`，禁止省略；tagged union其他variant专属字段不属于当前schema，必须缺席且出现null也拒绝。整数grammar为`0|-?[1-9][0-9]*`，无符号整数为`0|[1-9][0-9]*`，十进制为`0|-?(?:[1-9][0-9]*(?:\.[0-9]*[1-9])?|0\.[0-9]*[1-9])`；允许`0.5/-0.5`，禁止整体`-0`、前导/尾随零、指数、NaN/Infinity和JSON number。ID经NFC后匹配`[A-Za-z0-9][A-Za-z0-9._:-]{0,127}`；ContentHash/FindingId固定64个小写十六进制字符。

规范UTC instant wire固定为`YYYY-MM-DDTHH:mm:ss.fffffffffZ`（恰好9位小数、UTC Z、合法Gregorian日期、秒00–59、不接受闰秒/offset/截断），比较前转换为checked i128 Unix纳秒；一个time tick恰为1ns。所有跨语言边界测试使用该格式，非法/overflow拒绝而非取整。

`f2s-gate-wire-registry-v2` 的 canonical owner 是本文，集合成员和 wire 大小写固定为：

- severity：`error,warning,info`；priority：`p0,p1,p2`；subjectKind：`requirement,defect`；prioritySourceKind：`requirementsBaseline,defectRecord`；waiver：`forbidden,userAcknowledgement,governed`；
- category：`artAdvisory,compatibility,content,dataIntegrity,license,performance,privacy,recovery,scopeClaim,security,signature,spineClaim,updateRoot,userData,workerIsolation`；
- entityType：`animationClip,artifact,assetPlan,assetSpec,characterSpec,defectRecord,diagnosticRecord,exportProfile,exportRun,gameplayMarker,layerSet,masterRevision,motionSpec,priorityRegistry,project,promptPack,rigProject,ruleSet,runtimeSpec,styleSpec,waiver,weaponSpec`；
- location kind：`object,actionTick,attachment,imageRegion`；coordinateSpace：`canvasPixels,attachmentLocal,boneLocal,projectWorld`；
- dependency kind：`direct,transitive,rule,evidence,policy`；role：`content,geometry,timing,diagnostic,policy,evidence`。
- gate status：`reviewable,notReviewable`；freshness：`current,stale`；blockage：`ready,blocked`；ApprovalRecord kind：`approve,reject,acknowledgeWarning,revoke`；GateKind与scalar type以紧随其后的authoritative registry bytes为准，`ack_warning`不是alias。

### 8.3 四类互不混用的哈希

1. `DependencySnapshotHashV1 = SHA256("f2s-dependency-snapshot-v1\0" || JCS(snapshot))`。dependencies按`kind/role/entityType/entityId/revision/contentHash`全序唯一，并与Tracked resolver的规范read closure精确相等；相同identity冲突revision/hash、隐藏读取、缺边、多余边、未知kind/role均拒绝。snapshot唯一拥有target、完整依赖、RuleSet artifact/domain hash、DecisionPolicy artifact/domain hash、认证priority slice（见下段）和evidence hashes，不包含ack/waiver/approve或下游输出。
2. `EvaluationInputsHashV1 = SHA256("f2s-gate-evaluation-inputs-v1\0" || JCS({gate,wireRegistryHash,dependencySnapshotHash}))`。所有stable target/rule/policy/priority slice/evidence只由snapshot提供；report外层PriorityRegistryObservation仅证明评估时完整current registry并被GateReportHash覆盖，不进入本hash。它不含全局project revision、全局registry root/sequence、时钟或决策记录。
3. `GateOutcomeHashV1 = SHA256("f2s-gate-outcome-v1\0" || JCS({evaluationInputsHash,status,freshness,blockage,findings,blockerFindingIds,warningFindingIds,infoFindingIds}))`。它是未应用 ack/waiver 的基础结果；决策覆盖变化不得改变它。
4. `GateReportHashV2 = SHA256("f2s-gate-report-v2\0" || JCS(full GateReportV2))`，包含 `evaluatedProjectRevision`和PriorityRegistryObservation，仅用于证明某次评估正文、registry观察和审批命令的乐观并发。CAS artifact hash 是对同一 JCS bytes 的普通 SHA-256，二者都持久化，不得互相替代。

`GateFindingV2.findingId = SHA256("f2s-gate-finding-v2\0" || JCS(finding without findingId))`的64 lowercase hex。priorityBindings按subjectKind/subjectId全序且identity唯一，evidence按hash排序唯一；findings及三组分区ID各自按findingId全序。messageKey只能是非本地化规则key，messageParams参与hash；翻译文本不进入DTO。

Schema验证后的MUST-equal顺序固定为：①`report.dependencySnapshotHash`重算等于内嵌snapshot；②snapshot target等于resolver绑定target；③snapshot中RuleSet、policy与priority证明各自domain hash/artifact bytes一致；④每个finding的dependencySnapshotHash等于report值、rule descriptor/category与priority binding policy精确匹配、每个binding逐字段命中认证entry/source；⑤三组IDs严格等于findings按severity分区；⑥evaluationInputs/outcome/report hash依次重算；⑦GateDecisionContext的target、report schema/algorithm/hash/artifact/evaluated revision及stable hashes逐字段等于加载的report/snapshot，ApproveConcurrency的registry/clock值逐字段等于提交时current heads，request重复值永不作为真值。任一split-brain组合拒绝且零事件。

全局 `evaluatedProjectRevision` 只用于提交前 CAS：ack 提交把 Project 从 R 推到 R+1 后，只要 target、依赖、规则、policy、evidence 和基础 findings 未变，inputs/outcome hash 必须仍相同，ack 可供下一次 approve 使用；无关项目编辑同样不得使它失效。相关输入变化则 inputs/outcome hash 改变并精确失效。

### 8.4 DecisionPolicyV1

DecisionPolicy是本文拥有的不可变规范artifact。`matrixAlgorithmId=f2s-decision-matrix-v1`按顺序求值：①severity=error、任一binding为p0/p1或category在hardCategories时，结果为error/forbidden/blocker；②否则severity=warning、bindings非空且全部p2时为warning/governed；③否则severity=warning、bindings为空且category=artAdvisory时为warning/userAcknowledgement；④否则severity=info且bindings为空时为info/forbidden/info；⑤其余组合拒绝。binding先经PriorityRegistry认证，算法不读取决策记录。

所有versioned command envelope必须有必填、lowerCamelCase、closed-enum `commandKind` tag；自由字符串、缺tag、未知tag或依据endpoint/结果类型猜测tag均拒绝。领域append子集`AppendCommandKindV1`固定为`requirementsChange|ruleSetActivation|registerDefect|resolveDefect|derivedRuleSetActivationDefect|approve|reject|acknowledgeWarning|revokeApproval|clockRecovery|enrollOrganization|addActor|removeActorRoles|deactivateActor|revokeActorCredential|advanceActorTrustBundle|advanceActorRevocationBundle|grantWaiver|revokeWaiver`；其他§13.1命令同样以其versioned command schema中的closed enum成员作tag，不得省略。

任一`canonicalPayloadHash`都对排除命令envelope中裸expected-CAS heads后的**完整tagged JCS对象**计算，即投影固定为`{commandKind, ...variantSemanticFields}`，`commandKind`明确属于哈希域。排除裸CAS前置条件不允许删除作为业务授权本身的签名证据引用：18号ActorRegistry command payload必须保留`authorizationArtifactHash/authorizationHash`；该不可变authorization的签名正文按18号exact schema有意绑定expected registry heads，属于防跨head复用的语义能力，不能被替换为当前head或在重试时重签。全项目幂等索引先按`(actorId,idempotencyKey)`查找首次命令，再同时比较首次`commandKind`与canonical payload hash：两者都相同才返回首次同类型结果；同key跨`commandKind`即使其余字段逐字相同也必须作为different-payload conflict零record/零registry/零project transition，禁止分派到另一handler后返回不同结果类型。redb只缓存该索引，空索引必须从各append权威链的tag、payload hash和result ref重建。

首版不依赖云账号，但governed waiver是可选的本地多签能力，不等于“单用户点三次”。普通编辑可用稳定本地actor审计；产品/质量/发布attestation必须由18号ActorRegistry中三个不同、当前有效且具有不同credentialId的签名身份完成。未配置组织trust anchor、少于三份合法凭据或仅单人模式时，governed waiver能力明确不可用，只能修复finding；产品不声称离线技术手段能证明三个自然人，人员分离责任由组织证书签发和流程控制承担。

`GovernedWaiverBindingV1.bindingHash=SHA256("f2s-governed-waiver-binding-v1\0" || JCS(binding without bindingHash))`，只含grant/proposal/三attestation与stable绑定，不含checkedAt/动态Clock/Actor evidence；attestationRefs按role/actorId排序唯一且必须恰为三项。`WaiverValidationEvidenceV1.evidenceHash=SHA256("f2s-waiver-validation-evidence-v1\0" || JCS(evidence without evidenceHash))`，每次用**当前**priority/waiver/Actor registry、trust/revocation bundle和clock重新产生；actor-role evidence直接双引用且必须在policyValidationEvidenceRefs中恰出现一次相同双hash，refs按artifactHash/domainHash排序唯一。reasonCodes排序唯一且schema enum固定为`waiverHeadChanged,proposalMismatch,attestationInvalid,actorInactive,credentialInvalid,trustBundleInvalid,revocationBundleInvalid,priorityChanged,ruleChanged,targetChanged,scopeChanged,clockInvalid,expired`；decision=active时必须为空，inactive时必须非空。Approve只能提交active evidence；inactive evidence只供UI/审计并阻断。历史ApprovalRecord保存binding及提交当时的evidence作审计，active重放永不信任历史evaluatedAt，也不要求当前evidenceHash等于历史值。

所有target decision append（approve/reject/acknowledgeWarning/revokeApproval）先按全项目`(actorId,idempotencyKey)`查询首次commandKind+canonicalPayloadHash：kind与payload都相同才返回首次同类型结果；跨commandKind或同kind异payload立即以different-payload conflict拒绝，零事件且不得返回另一variant结果。随后才在一个ProjectTransaction中CAS expected project revision、ApprovalStreamRegistry sequence/artifact/domain hashes、目标entry sequence及artifact+domain双head。Approve另CAS当前WaiverRegistry sequence/artifact/domain、PriorityRegistry observation与clockEpoch/checkpoint双head，并在commit前重读effectiveNow、重评base report、按当前registry逐slice subject重验每个binding。ClockRecovery没有target且不触碰ApprovalStreamRegistry；其独立事务见下文。任何并发变化都明确返回`F2S-DOMAIN-CONFLICT-001`，刷新后人工重放。

`RegisterDefectFromFindingDraft`与`ResolveDefect`分别固定`commandKind=registerDefect|resolveDefect`并执行全项目`(actorId,idempotencyKey)`去重；同key跨这两个kind或跨任意其他command variant是different-payload conflict。originKey唯一索引另防不同key重复，但命令只CAS project revision及PriorityRegistry sequence/artifact/domain双head。服务从draft/resolution artifact取回正文并重算domain/fingerprint/origin key，重读RuleSet/target/evidence，生成符合default/history floor的下一DefectRecord、下一registry snapshot与新project revision；record、registry previous artifact/domain、origin唯一索引和project pointer在11号事务中共同提交。任一旧draft、冲突或校验失败均零record/零registry transition，成功后UI只能触发重新评估，不能直接申请waiver/approve。

`revoke`可指向active approve或active warning acknowledgement，reason非空。撤回未被任何active approve引用的ack只关闭该ack，不传播stale；只有因此使active approve失活时，对象才回candidate（superseded除外）并使精确下游stale。ack不能覆盖/删除。

`ApprovalRecordV2`是 append-only tagged union并拥有完整重放字段，不能从 UI 或当前对象反查补齐：

每条record以lowerCamelCase/NFC/显式null DTO做JCS，`recordHash=SHA256("f2s-approval-record-v2\0" || JCS(record without recordHash))`，完整record bytes的普通SHA-256是recordArtifactHash且不写入自身。每stream genesis固定sequence=1/prev artifact+record均null，之后严格n+1且两个prev分别等于前一record的CAS artifact hash/domain recordHash；recordId、commandId全项目唯一，全项目`(actorId,idempotencyKey)`索引比较commandKind+canonicalPayloadHash，跨kind或异payload均拒绝且不得返回不同record variant；committedProjectRevision必须等于首次使该record从RevisionManifest可达的revision且沿stream严格增加。任一双hash不匹配、缺一、缺失/重复/截断、跨stream revoke、未知kind/schema/algorithm全部fail closed。

`ApprovalStreamRegistrySnapshotV1.registryHash=SHA256("f2s-approval-stream-registry-v1\0" || JCS(snapshot without registryHash))`。genesis为registrySequence=0、previousRegistryArtifactHash/previousRegistryHash均null、entries=[]；每次只准sequence+1，两个previous分别等于旧snapshot的CAS/domain hashes，并新增一个target entry或把一个entry从n/双head推进到n+1/新record artifact+domain hashes；新record的prev双hash必须等于旧entry。禁止删除、倒退、改streamId、缺任一hash或无record跳转。entries按targetEntityType/targetEntityId排序唯一。RevisionManifest保存该snapshot的CAS artifact/domain hash而非单一approval head，重启/GC先按registry previous artifact/domain验证snapshot链，再用每个entry.headArtifactHash取当前record、沿record prev双hash重放；不得假设domain hash可直接寻址CAS。

ClockRecoveryRecord同样从sequence=1/prev artifact+record均null追加，之后n+1且prev双hash指向前一record；`recordHash=SHA256("f2s-clock-recovery-record-v1\0" || JCS(record without recordHash))`，完整bytes普通SHA-256为CAS artifact hash；newClockEpoch必须=previous+1。record保存`commandKind=clockRecovery`及actor/command/idempotency/canonicalPayloadHash/committedProjectRevision，payload hash显式覆盖该tag，并按通用append幂等规则从CAS重建，不能依赖redb记忆首次结果；同key若已被任一其他commandKind占用则different-payload conflict且零ClockRecovery transition。每条record必须把old checkpoint current双head与new checkpoint next双head逐项固定，且`newCheckpointSequence=oldCheckpointSequence+1`。sequence=0时recovery artifact/record heads均为null，非零时两者必须同时存在、分别重算并证明同一record。事务更新manifest中的clockEpoch/lastSeenUtc/recovery双head，并使所有较早epoch的waiver bindings永久inactive；旧grant不得因重启或时间修正复活。

每个稳定target identity拥有独立stream。ActiveWarningAcknowledgement要求current target、inputs/outcome/policy/finding匹配且无revoke，不比较evaluatedProjectRevision/reportHash。ActiveApprovalHead还要求ack仍active、每个static waiver binding由18号verifier以当前waiver entry/effectiveNow/clockEpoch验证、current PriorityRegistry中slice涉及的每个subject仍逐字段等于绑定entry，且current snapshot仍产生同一stable inputs/outcome。observed全局priority/waiver registry roots只证明批准当时读到什么；任一无关entry推进不失效，关联priority source变化或当前特定waiver entry撤销/替换才失效。

`ClockCheckpointRecordV1.recordHash=SHA256("f2s-clock-checkpoint-record-v1\0" || JCS(record without recordHash))`，完整bytes普通SHA-256为recordArtifactHash。全项目checkpoint sequence连续，**仅项目最初一条**是双prev=null的genesis；所有后继（包括ClockRecovery产生的新epoch首条）均为n+1并双指向前一record。项目根`clock-state.json`只保存`ClockCheckpointPointerV1`；其projectId/epoch/sequence/bootIdHash/interruptTime100ns/cleanShutdown必须逐项等于head record，`lastSeenUtc == head.checkpointUtc`，两个head hash分别重算同一record。record先flush/promote到CAS，再同卷原子替换pointer。RevisionManifest的ClockState保存提交时最低锚点，pointer可以在不改变艺术project revision时单调超前；active验证要求current pointer能回到**current manifest同epoch**anchor。保留的旧epoch manifest checkpoint只是GC/audit root，可沿合法epoch boundary从current链追溯，不要求与current pointer同epoch。pointer缺失、落后、分叉或双hash不符均fail closed。GC把current pointer及所有保留manifest checkpoint heads作为root。

合法同epoch后继必须满足`checkpointUtc == effectiveNow >= previous.checkpointUtc`；同boot还必须`interruptTime100ns >= previous.interruptTime100ns`。不同boot不能在旧epoch追加checkpoint，只能经ClockRecovery原子创建`oldEpoch+1`的**epoch-boundary successor**：它仍双指旧epoch checkpoint，且old/new双head必须被同一次ClockRecoveryRecord固定。链验证只允许在匹配recovery record时epoch恰好+1；其他后继epoch必须不变，禁止重置sequence、伪epoch跳跃或无recovery跨epoch。任何签名和双hash都合法但时间、counter、pointer lastSeen字段回退的record仍非法且不得成为head；不存在可提交的“时间倒退但以后再修正”状态。

Windows唯一时基实现为`QueryInterruptTimePrecise`的100ns biased interrupt time（包含sleep/hibernate），checked乘100转换纳秒；`QueryUnbiasedInterruptTime*`明确禁止。BootIdentityProvider必须给出经M00 Spike验证的本次Windows boot稳定ID；无法证明boot identity或API语义即`clockState=rollbackBlocked`。同一boot的`effectiveNow=max(wallUtc, checkpointUtc + checked((currentInterrupt100ns-checkpointInterrupt100ns)*100ns))`；wallUtc<lastSeenUtc、counter倒退/overflow均阻断。由于V1没有可信外部时钟，**任何boot identity变化或不确定都不能继续旧epoch waiver**，即使上次cleanShutdown且wallUtc看似前进也必须fail closed并执行ClockRecovery；cleanShutdown只用于恢复诊断，不作为离线时间证明。由此不声称防篡改RTC，却通过使旧epoch binding永久inactive避免离线回拨延长授权。

应用取得项目写锁后、首次active验证前先耐久追加`cleanShutdown=false` checkpoint；之后在任一active waiver存在时最长每5分钟追加一次，并在approve/export/sign、resume、最早expiry、project close前同步追加。干净关闭最后追加`cleanShutdown=true`；同boot进程崩溃重启可用持续interrupt time精确补足checkpoint后流逝，任何系统重启、无法确认boot或复制中间态均按上一段fail closed。因此5分钟是恢复进度间隔，不是可放宽的时间窗口。Gate stable hashes不含时钟；每次validation生成含pointer双head/boot/interrupt/effectiveNow的ClockEvidence。唯一expiry边界仍为`grantedAt <= effectiveNow < expiresAt`，恰好expiresAt失效；18号强制grant.occurredAt=grantedAt且所有attestation.approvedAt<=grantedAt。

ClockRecovery只在用户修正时钟后显式执行：它CAS expected project revision、WaiverRegistry sequence/artifact/domain、clock checkpoint双head、clockEpoch和recovery双head，追加ClockRecoveryRecord与旧head相连的新epoch checkpoint successor并提交新RevisionManifest；不读取/推进ApprovalStreamRegistry。11号OperationJournal先提交含新epoch/checkpoint的project revision，再原子替换pointer；中途崩溃时启动器只可从已提交manifest物化同一新pointer，旧epoch已永久inactive。之后必须新grant/重审。它不删除历史，也不是“忽略时间继续”。

`reject`把candidate投影为rejected；ack不改Lifecycle；只有active approve因revoke、被引用ack撤回、policy/priority/依赖变化或waiver失效而inactive时，对象回candidate（superseded除外）、精确下游stale并阻断release。失败命令不追加事件。

```text
  → Run QA and resolve/accept warnings

  → Optional Spine 4.2.43 validation
```

主干是默认导航，不是禁止回退的向导。用户可在任意时候修改上游，但在提交前必须看到 InvalidationPlan。

### 12.2 后台 Job 与领域迁移

```text
Domain command: RequestLayerSuggestion
  → create persisted Job(requested)
  → scheduler runs provider
  → provider returns artifacts
  → verify schema/hash/generation provenance
  → persist immutable JobOutputArtifact in CAS
  → persist Job(succeeded)
  → separate domain command: RegisterLayerCandidate(expected project revision, output hash)
  → human reviews
  → domain command: ApproveLayerSet or RejectLayerSet
```

Worker 不能直接触发产物状态迁移。Job `succeeded` 与 `Register*Candidate` 是两个独立、可审计的事务：

- `succeeded` 只表示输出通过 schema/hash/安全/provenance 校验，已经作为不可变 `JobOutputArtifact` 持久到 CAS；它没有绑定 AssetSpec/LayerSet，也没有创建 project revision candidate。
- `Register*Candidate` 必须携带 output hash、target ID、`expected_revision` 和 idempotency key；成功后才创建 candidate revision。
- 注册遇到 revision 冲突时 Job 仍保持 `succeeded`，输出处于 `unbound`，用户可刷新后重新注册或显式丢弃；不得重跑昂贵推理来掩盖事务冲突。
- 注册前崩溃可从 `JobExecutionRecord` 和 CAS 重建；保留期到期前不能把未绑定输出 GC。
- 如输出校验失败，Job 进入 `failed` 或 `interrupted`，领域中不会出现 candidate。

### 12.3 补素材闭环

```text
Temporary Rig/Animation Preview
  → DiagnosticRecord(transparent hole/stretch/silhouette/contact/...)
  → Create SupplementAssetRequest
  → merge/deduplicate into AssetSpec candidate
  → user approves
  → PromptItem or local repair task
  → import/register new candidate
  → map into LayerSet/Rig
  → rerun affected diagnostics only
```

补素材不默认重做整套图；其 AssetSpec 必须引用证据诊断、所需动作和表现策略。

### 12.4 持久化 Job 状态与终态仲裁

Job 只允许以下持久状态：

```text
requested
queued
running
cancel_requested
cancelled
failed
succeeded
interrupted
```

| 当前状态 | 合法后继 | 说明 |
| --- | --- | --- |
| `requested` | `queued`, `cancel_requested`, `failed` | 请求和资源/协议校验已持久；校验失败直接 failed |
| `queued` | `running`, `cancel_requested`, `failed`, `interrupted` | 调度器尚未将任务交给 Provider |
| `running` | `cancel_requested`, `failed`, `succeeded`, `interrupted` | 进度/阶段是投影，不是额外持久状态 |
| `cancel_requested` | `cancelled`, `failed`, `interrupted` | 不允许直接返回 `running`；后到 success 由下述仲裁 |
| `cancelled` | 无 | 终态 |
| `failed` | 无 | 终态；重试创建新 JobId |
| `succeeded` | 无 | 终态；只表示结果已验证并作为未绑定 `JobOutputArtifact` 持久到 CAS，不表示 candidate 或 approved |
| `interrupted` | 无 | 本次执行终态；恢复/重试使用新 JobId 或明确 resume operation |

`cancel_requested` 和 success 竞态由 Rust 持久化调度器按顺序号仲裁：

1. 如已通过 schema/hash/安全/provenance 验证且 CAS 持久完成的 success 终态先于 cancel command 持久，`succeeded` 胜出，后到取消返回“已终态”。
2. 如 `cancel_requested` 先持久，后到 success 不能转 `succeeded`；输出进 quarantine，Provider 确认停止后转 `cancelled`。
3. 如断连/崩溃使持久顺序无法证明，终态必须是 `interrupted`，不猜测 cancelled 或 succeeded。
4. Worker/Remote 只提供终态候选；它们无权直接写 Job 持久状态。

### 12.5 Job 权威存储

Job 状态的唯一可恢复事实使用 `F2S-DOC-STORE-001` 的 `F2S-SCH-009 JobExecutionRecord` 与按序不可变 transition 记录；`redb` 只保存队列/查询投影。记录至少绑定 JobId、sequence、previous record hash、请求与幂等键、base project revision、Provider/model/protocol provenance、重试/恢复父链、状态和 JobOutputArtifact hash。删除/损坏 `redb` 后必须从这些权威记录重建 queued/running/interrupted/terminal 投影，不能根据 Worker 进程或 sandbox 文件猜测状态。

## 13. 命令、事件和幂等

### 13.1 必要命令族

- Project：`CreateProject`、`OpenProject`、`SaveAs`、`RecoverProject`；
- Import/Master：`ImportSourceImage`、`SetGroundAndScale`、`ApproveMaster`、`RejectMaster`；
- Specs：`UpdateCharacterSpec`、`UpdateStyleSpec`、`UpdateWeaponSpec`、`UpdateRuntimeSpec`、`UpdateMotionSpec`；
- Planning：`DraftAssetPlan`、`OverrideRepresentationStrategy`、`ApproveAssetPlan`、`ComposePromptPack`；
- Layers：`RegisterLayerCandidate`、`ApplyMaskStroke`、`RelinkExternalLayer`、`ApproveLayerSet`；

- QA/Export：`RunDiagnostics`、`AcceptWarningRisk`、`RequestExport`、`RegisterSpineVerification`。

每个命令包含closed-enum `commandKind`、command ID、project ID、expected revision、idempotency key、actor 和语义 payload；commandKind必须进入canonical payload JCS/hash。只有同一actor/key、同一kind且同一payload才返回首次同类型结果；跨kind复用同key一律different-payload conflict并零状态迁移。

### 13.2 领域事件

必须有稳定 event type 的事件至少包含：

- `MasterApproved`、`MasterSuperseded`；
- `SpecChanged`、`AssetPlanApproved`、`PromptPackComposed`；
- `LayerCandidateRegistered`、`LayerSetApproved`；
- `RigChanged`、`RigApproved`；
- `ClipChanged`、`ClipApproved`、`GameplayMarkersApproved`；
- `ApprovalBecameStale`、`WarningRiskAccepted`；
- `ExportRequested`、`ExportCompleted`、`SpineVerificationRecorded`。

事件用于审计、索引更新和 UI 刷新，不承诺首版使用完全事件溯源重建 Project。项目快照仍是快速打开的权威形式，事件日志是不可变审计和恢复辅助。

## 14. 领域不变量

1. 每个 Project 必须且只有一个当前母版指针，指向不可变 MasterRevision。
2. 十个基线 action ID 必须存在，可标记尚未制作，不可被删除。
3. 只有 Lifecycle=`approved`、Freshness=`current`、Blockage=`ready` 的产物可进 release export。
4. 每个approved产物都必须有匹配target、DependencySnapshot、PriorityRegistry、stable inputs/outcome、DecisionPolicy、active acknowledgement及经当前registry/clock动态重验的静态GovernedWaiverBinding，且由ApprovalStreamRegistry目标head证明未被撤回的active `approve` ApprovalRecord；全局project revision或transport report hash不作为持续有效性键。
5. 内部 AI/Worker 输出只有通过完整 GenerationProvenance 硬门才能注册 candidate；外部缺失信息必须是 `unknown_external` 加用户声明；手工派生版本引用父产物且不可擦除来源。
6. 母版可见像素的 AI 改写必须为显式候选并有差异证据。
7. GameplayMarker 不能在修改 clip 时长或相关关键帧后保持 approved。
8. AssetSpec 的 pivot、父骨建议、重叠和 requiredBy 不可全部缺失后通过 Asset Plan Gate。
9. `primary_weapon` 未具体化时，不允许将武器接触/命中数据标为 release approved。
10. 官方 Spine 验证状态只能由实际 4.2.43 CLI 运行证据产生。

## 15. 失败语义

| 失败类型 | 是否可重试 | 领域结果 |
| --- | --- | --- |
| 输入/decision前置验证失败（`F2S-DOMAIN-STATE-001`） | 修改输入后 | 命令无事件、revision 不变 |
| project/stream双head冲突（`F2S-DOMAIN-CONFLICT-001`） | 刷新两者后人工重放 | 不覆盖、不自动合并 |
| GateReport/policy/waiver/clock无效（`F2S-DOMAIN-POLICY-001`） | 重新评估、修正时钟或新grant | 审批不产生；旧记录不改 |
| Worker 失败/取消 | 视技术错误 | 不注册 candidate，不修改已批准产物 |
| Worker 结果 schema/hash 错误 | 不盲目自动重试 | 结果隔离，记录安全/完整性错误 |
| 导出 preflight 错误 | 修复项目后 | 禁止 release export，可保留不对外的调试 staging |
| Spine CLI 缺失/版本错误 | 配置合法 4.2.43 后 | 可有 `exported_unverified`，不得有 `verified` |
| 用户撤回审批/确认 | 不适用 | 追加引用active approve/ack的revoke；被依赖的active approval失效，当前对象Lifecycle回到`candidate`（superseded除外），精确下游Freshness=`stale`，历史不删除 |
| waiver到期/撤销或时钟回拨 | 修正时钟或新建合规grant后重新评估 | active approval立即派生为inactive；禁止批准/导出，不改旧记录 |

## 17. 完成定义

- 上述 15 类领域对象均有 Rust 类型、JSON Schema、不变量和迁移策略。
- 每个人工门禁都有不可变GateReport正文、DependencySnapshot、PriorityRegistry、DecisionPolicy、证据和ApprovalRecord；stable base outcome与ack/static-waiver/approve覆盖分离，当前waiver状态只由动态验证产生，不存在UI-only批准、自引用哈希或历史checkedAt复用。
- 失效引擎可以精确解释每个 Freshness=`stale` 对象的原因，而不改写生命周期。
- Worker、本地规则、手工编辑都通过同一 candidate 注册和审批路径。
- 十个基线动作在无 AI/GPU/网络下也能被建立、编辑、审核和导出。
- 导出门禁能区分 release-ready、exported-unverified 和 incompatible，不伪造 Spine 官方验证。
