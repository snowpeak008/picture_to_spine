# 内容、动作、素材与 AI 提示词设计

## 1. 文档目的

本文档定义首版角色内容模型、十个动作的语义、从动作反推素材的规则，以及作为正式交付物的 AI 动作提示词包。它不调用任何图片生成服务；提示词由用户复制到其自行选择且符合其隐私要求的外部工具，生成后的图片再由用户导回本产品。

内容层的核心关系为：

`StyleSpec + CharacterSpec + WeaponSpec + RuntimeSpec + MotionSpec → RepresentationPlan → AssetSpec → PromptPack → 导入素材 → 临时 Rig → 诊断 → 补充 AssetSpec`

这是一条有反馈的生产闭环。任何动作都不能仅因已有骨骼和图片而被强行变形完成；当轮廓、透视、拓扑或遮挡超过阈值时，系统应要求新增素材。

## 2. 内容基线与禁止假设

### 2.1 已确认基线

- 二次元类人角色；
- 横版侧视，默认角色面向画面右侧；
- 单一武器语义 ID：`primary_weapon`；
- 输入为角色图、动作关键帧图片和动作描述文字；
- 动作集固定为 `idle/run/jump/fall/dash/attack_01/attack_02/attack_03/hit/death`；
- Spine 目标为 4.2.43；
- 母版、分层、Rig、关键姿势、命中帧由用户批准；
- AI 提示词包属于正式输出。

### 2.2 禁止假设

- 不得把 `primary_weapon` 默认解释为剑、刀、枪、法杖或其他具体类型；
- 未获得 WeaponSpec 前，不得固定挥击轨迹、双手/单手、武器长度、重量感或攻击接触部位；
- 不得从单张侧视图宣称准确恢复背面、内层服装或不可见纹样；
- 不得把动作关键帧图片自动解释为精确碰撞、命中或 root motion 数据；
- 不得在角色主体图内合并 `weapon_trail`、火焰、残影或尘土，除非 AssetSpec 明确要求动作专用合成附件；只有 WeaponSpec 明确为 blade 类时，UI 才可把 `weapon_trail` 本地化显示为“刀光”；
- 不得用“保持一致”一句话代替可执行的身份、画布、姿势和输出约束。

## 3. 六类结构化规格

### 3.1 StyleSpec

`F2S-SPEC-STYLE-001` 是所有提示词和视觉 QA 的风格真值。

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `styleId/schemaVersion` | 是 | 稳定 ID 与 schema 版本 |
| `masterAssetId/masterHash` | 是 | 已批准母版及内容哈希 |
| `view` | 是 | `side_right`；镜像只作为显示/导出策略，不重写母版 |
| `projection` | 是 | 首版固定 `orthographic_like`，禁止鱼眼和动态镜头 |
| `proportion` | 是 | 头身比、肩胯比例、手脚夸张程度 |
| `lineArt` | 是 | 线宽范围、颜色、闭合程度、抗锯齿特征 |
| `palette` | 是 | 主色、辅色、肤色、武器色，使用采样值和容差 |
| `shading` | 是 | 平涂/赛璐璐层数、高光类型、阴影边界 |
| `lighting` | 是 | 光向、强度和是否烘焙；首版建议固定弱方向光 |
| `textureDensity` | 是 | 纹理和小装饰密度，标明不可由 AI 改写的标志 |
| `canvas` | 是 | 宽高、透明背景、角色占比、地面线、pivot 基准 |
| `edgePolicy` | 是 | alpha 边缘、预乘策略、颜色扩边和安全留白 |
| `animationFriendlyRules` | 是 | 肢体分离、关节重叠、长发/裙摆拆分、禁止动态光影 |
| `lockedFeatures` | 是 | 脸型、发型、瞳色、服装纹样、徽记、武器外观等身份锚点 |
| `authoringConfirmation` | 否 | `{confirmedByActorId,confirmedAt,note}`只表示编辑者确认规格字段已准备审阅；不得包含或派生`approved`布尔值，不参与Gate批准判定 |

母版批准是StyleSpec进入审查的前置，但StyleSpec只有在09号ApprovalStreamRegistry目标head重放出active `approve`时才投影为`approved`；`authoringConfirmation`不能替代该记录。母版、StyleSpec内容或lockedFeatures变化会使相关提示词、派生图和旧批准进入`stale`，历史记录不改写。

### 3.2 CharacterSpec

`F2S-SPEC-CHAR-001` 描述身体语义，不直接等同于骨架。

必填内容：

- 角色 ID、显示名、母版引用、默认朝向；
- 身体区域：头、颈、胸、腹/腰、骨盆、近/远侧上臂、前臂、手、大腿、小腿、脚；
- 可选区域：前后发组、前后衣摆、披风、尾巴、饰物；
- 近侧与远侧定义，避免使用含糊的 left/right；
- 各关节中心建议、活动幅度、重叠安全区；
- 地面点、角色原点、视觉中心；
- 永远不可被补图模型改变的身份锚点；
- 不确定区域及用户确认状态。

### 3.3 WeaponSpec

`F2S-SPEC-WEAPON-001` 以 `primary_weapon` 为固定 ID，但具体内容在用户提供前保持未决。

| 字段 | 状态 | 说明 |
| --- | --- | --- |
| `weaponId` | 固定 | `primary_weapon` |
| `motionFamily` | 待用户填写 | slash/thrust/blunt/ranged/cast/custom；不能由系统猜测 |
| `gripMode` | 待用户填写 | one_hand/two_hand/other |
| `gripSocketPrimary` | 必填后才可批准攻击 | 主手握持点 |
| `gripSocketSecondary` | 条件必填 | 双手武器或辅助握持 |
| `contactRegion` | 必填后才可批准攻击 | 可能接触区域，不等同于碰撞盒 |
| `visualLength/massClass` | 必填后才可批准攻击 | 影响轮廓与惯性提示 |
| `restAttachment` | 必填 | 待机/跑跳时挂载方式 |
| `attackAttachmentVariants` | 按需 | 强透视或 motion blur 变体 |
| `lockedAppearance` | 必填 | 不允许外部生成改变的外形和纹样 |

在 WeaponSpec 未批准时，系统可以生成带占位符的“动作开发提示词草案”，但 `attack_01`、`attack_02`、`attack_03` 的生产提示词状态必须是 `blocked_by_weapon_spec`。

### 3.4 RuntimeSpec

`F2S-SPEC-RUNTIME-001` 依据 canonical `F2S-ADR-TIME-001` 约束内容规模与导出；本节只投影内容字段，不得另定时间算法：

- 目标：`spine_editor_4_2_43`；
- 持久化时间：有符号整数 tick 与已约分的正有理 `timeBase={numerator:u32,denominator:u32}`；`seconds = tick × numerator / denominator`，i64 tick 在 JSON/IPC 中使用 canonical 十进制字符串而非 JSON number；
- 默认 `timeBase=1/30000 s`；编辑帧率为有理 `30/1`，仅用于吸附和帧号换算，不改变 tick，也不降低 Spine 连续时间精度；
- 世界/像素比例、角色地面原点、画布裁剪；
- 最大纹理尺寸、图集 padding、premultiplied alpha 选择；
- 默认混合、颜色空间和导出缩放；
- 骨骼、影响骨、网格顶点、附件、序列帧和动作时长软/硬预算；V1 单动作硬上限为精确有理60秒；
- 引擎碰撞不在首版，但事件和命中帧以语义标记输出。

### 3.5 MotionSpec

`F2S-SPEC-MOTION-001` 每个动作一份，字段如下：

```yaml
motion_id: F2S-ACT-IDLE-001
clip_name: idle
source:
  description_text: ...
  key_pose_assets: []
timing:
  loop: true
  time_base: { numerator: 1, denominator: 30000 }
  suggested_duration_ticks: { min: "36000", max: "60000" }
  edit_frame_rate: { numerator: 30, denominator: 1 }
phases:
  - id: phase_name
    suggested_tick_range: { start: "0", end: "6000" }
    author_confirmed: false
contacts:
  feet: []
  weapon: []
key_poses: []
representation:
  preferred: [bone, mesh]
  allowed: [attachment_swap]
  forbidden_shortcuts: []
asset_requirements: []
diagnostics: []
authoring_confirmation:
  key_pose_refs_complete: false
  gameplay_intent_complete: false
```

`author_confirmed`与`authoring_confirmation`只是内容编写完整性，不是Lifecycle、ReviewReadiness、GateOutcome或审批真值，也不得被导入文件用来解锁下游。tick 范围是启动建议，不是自动确定的玩法数值。秒、帧号和归一化时间都只由上述有理时间基确定性派生；导入十进制秒时必须按 half-even 规则量化到 tick 并记录原值与舍入差，不能把 `f64` 秒写回规格。用户的动作描述和关键帧图优先，但真正批准只来自09号ApprovalStreamRegistry中的active `approve`。

### 3.6 AssetSpec

`F2S-SPEC-ASSET-001` 是最小素材集合的权威清单：

| 字段 | 说明 |
| --- | --- |
| `assetSpecId/semanticId` | 稳定 ID 与身体/武器/效果语义 |
| `assetKind` | base_part/corrective/pose_swap/sequence_frame/weapon/vfx/reference |
| `view` | side_right 等；首版新视角会触发范围警告 |
| `requiredBy` | 至少一个 MotionSpec 或诊断 ID |
| `sourceMaster/sourceMask` | 母版和派生遮罩 |
| `visiblePixelPolicy` | preserve/approved_redraw/generated_new_view |
| `parentBoneHint/slotHint` | Rig 建议，不自动批准 |
| `pivot/groundOffset` | 与角色原点一致的坐标 |
| `overlapMargins` | 四边留量和对应关节中心 |
| `drawOrderBand` | far/body/near/weapon/fx 等逻辑层 |
| `promptItemId` | 需要外部生成时关联提示词 |
| `generationProvenance` | 内部 AI 任务强制完整记录；外部回导无法取得生成信息时固定为 `unknown_external` 并附用户来源声明 |
| `lifecycle` | 只使用 `F2S-WFS-LIFECYCLE-001`：missing/draft/candidate/rejected/approved/superseded；`missing` 是必需 AssetSpec 不存在时的派生值，不序列化到不存在的对象 |
| `freshness` | 只使用 `F2S-WFS-FRESHNESS-001`：current/stale；由依赖 revision/hash 推导，不改写 lifecycle |
| `approvalRecordRef` | 可空的不可变ApprovalRecordV2引用；必须按09号从ApprovalStreamRegistry目标head重放为未被后继revoke撤销的active `approve`，且target、DependencySnapshot、认证PrioritySlice、stable inputs/outcome、DecisionPolicy、精确active acknowledgement及每个静态GovernedWaiverBinding的当前动态验证全部仍有效时lifecycle才可为approved；全局PriorityRegistry observation或project revision变化不单独失效，关联slice entry变化才失效；reject/acknowledgeWarning/revoke均不合格 |
| `verification` | `not_run/failed/passed` 的素材**基础规则验证**投影，并绑定精确GateReportV2 artifact/outcome/evidence；它不是lifecycle、active approval或 `F2S-WFS-EXPORT-VERIFY-001`，passed不能显示成approved |
| `availability` | `unavailable/partial/ready` 的派生投影，由 source/mask/materialized artifact 引用共同计算，不表示批准或验证通过 |

无 `requiredBy` 的素材是孤儿，不能进入默认导出；同一视觉素材服务多个动作时必须去重，而不是复制多个不一致版本。AssetSpec 不持久化任何混合 `status` 枚举：生成进度只形成 availability/verification 事实，审查结论只进入 lifecycle 与 ApprovalRecord，依赖失效只进入 freshness。任何 UI 汇总标签都必须可从这些正交字段重建，不能反向写回领域对象。

#### 3.6.1 Provenance 强制规则

本地 Worker 和私有远程 GPU 都属于产品编排的内部 AI 任务，其 `generationProvenance` 必须包含：

- `providerKind`、Provider/Worker ID 与协议版本；
- 模型/算法 ID、精确版本和可验证包哈希；
- 参数、seed（算法适用时）、设备/执行位置和任务 ID；
- 输入资产、遮罩、StyleSpec/MotionSpec/AssetSpec revision 与内容哈希；
- 输出哈希、开始/结束时间、取消/重试链和质量降级记录。

任一必填 provenance 缺失时，内部结果只能进入隔离区，不能提升为 `candidate`。用户从产品外部生成或编辑后回导的图片不伪造不可得信息：若用户无法提供工具、模型、版本或 seed，系统写入 `sourceKind=unknown_external`、文件哈希、导入时间和必填的用户来源声明。`unknown_external` 可以进入人工审查，但必须持续显示来源未知警告，不能被报告成可复现的内部 AI 结果。

## 4. 基础分层语义

### 4.1 最小身体集合

实际拆分由角色设计决定，系统模板至少提供下列稳定语义槽：

| Asset ID | 语义 | 常见父骨 | 注意事项 |
| --- | --- | --- | --- |
| `F2S-AST-BODY-001` | torso | chest | 保留肩、腰隐藏延展 |
| `F2S-AST-BODY-002` | pelvis | pelvis | 与 torso、大腿有双向重叠 |
| `F2S-AST-BODY-003` | head | head | 脸、后发是否拆分由诊断决定 |
| `F2S-AST-BODY-004` | upper_arm_near | upper_arm_near | 超过肩中心并隐藏接缝 |
| `F2S-AST-BODY-005` | lower_arm_near | lower_arm_near | 肘部至少覆盖允许弯曲范围 |
| `F2S-AST-BODY-006` | hand_near | hand_near | 默认手与 grip 变体分离 |
| `F2S-AST-BODY-007` | upper_arm_far | upper_arm_far | 远侧命名不可用 left/right |
| `F2S-AST-BODY-008` | lower_arm_far | lower_arm_far | 与 torso draw order 可变化 |
| `F2S-AST-BODY-009` | hand_far | hand_far | 需要 grip/pose 变体时单列 |
| `F2S-AST-BODY-010` | upper_leg_near | upper_leg_near | 胯部隐藏延展 |
| `F2S-AST-BODY-011` | lower_leg_near | lower_leg_near | 膝部轮廓修正可另建附件 |
| `F2S-AST-BODY-012` | foot_near | foot_near | pivot 与地面接触点分开 |
| `F2S-AST-BODY-013` | upper_leg_far | upper_leg_far | 远侧层级 |
| `F2S-AST-BODY-014` | lower_leg_far | lower_leg_far | 远侧层级 |
| `F2S-AST-BODY-015` | foot_far | foot_far | 远侧接触点 |
| `F2S-AST-WEAPON-001` | primary_weapon | weapon socket | 外观锁定，攻击透视变体按需 |

头发、裙摆、披风、饰物不得用固定层数强套。系统根据母版分区与动作轮廓，为它们创建 `F2S-AST-SECONDARY-<ULID>` 实例；`F2S-AST-SECONDARY` 是本文拥有的动态命名空间，`<ULID>` 在创建时固化，实体、审批和追踪中不得保留通配符。

### 4.2 关节重叠规则

- 部件必须跨过关节中心，不能在可见接缝处刚好切断；
- 重叠目标以关节直径和最大旋转范围表达，不使用适用于所有分辨率的固定像素；
- 默认建议每一相邻部件越过关节中心至少 0.5 个关节直径；大幅屈曲时由试跑验证提高；
- 隐藏延展不得覆盖母版可见像素；只在原本不可见区域补全；
- 对无法通过重叠解决的肘、膝、肩、胯轮廓，创建 corrective AssetSpec；
- 透明发丝、半透明布料和发光边缘必须保存独立 alpha 策略，不能用普通实色扩边。

## 5. 表现策略决策规则

| 变化检测 | 首选策略 | 触发新增素材 |
| --- | --- | --- |
| 平面内小幅旋转/位移，轮廓稳定 | bone | 否 |
| 局部连续弯曲，纹理允许形变 | weighted mesh | 通常否 |
| 眨眼、嘴型、手型、武器状态 | attachment swap | 是，局部变体 |
| 关节暴露隐藏像素或局部轮廓坍塌 | corrective attachment | 是 |
| 前臂、武器等明显朝向镜头的强缩短 | pose/foreshortening attachment | 是 |
| 角色转面或整体视角离开侧视 | new-view rig | 是，且首版提示超范围 |
| 拓扑改变、smear、极端压缩/拉伸 | pose swap 或短 sequence | 是 |
| `weapon_trail`、烟尘、残影、命中特效 | 独立 vfx/sequence | 是，不污染基础角色层；具体显示名由已批准 WeaponSpec 决定 |

### 5.1 禁止的自动降级

- 不得仅通过增加 mesh 顶点掩盖缺少的视角；
- 不得把透明洞自动涂成相邻肤色并标为完成；
- 不得因生成失败而删除动作需求；
- 不得将新附件静默绑定到猜测的骨骼；
- 不得用镜像生产带非对称服装、脸部或武器的另一方向素材；
- 不得用帧插值自动批准关键姿势或命中帧。

## 6. 十个动作规格

动作时长均为建议区间，最终时序由用户审批。`near/far` 以当前侧视绘制层级定义；角色面向右。

### 6.0 Canonical Action Registry

下表是 V1 动作 ID 与 `clip_name` 的唯一 canonical 映射。下游 MotionSpec、PromptPack、时间轴、测试、追踪矩阵和导出只能引用这些 ID 与名称，不得省略 `-001`、改用 `attack_1` 或另计第十一个 clip。

| Action ID | `clip_name` | 数量语义 |
| --- | --- | --- |
| `F2S-ACT-IDLE-001` | `idle` | 1 |
| `F2S-ACT-RUN-001` | `run` | 1 |
| `F2S-ACT-JUMP-001` | `jump` | 1 |
| `F2S-ACT-FALL-001` | `fall` | 1 |
| `F2S-ACT-DASH-001` | `dash` | 1 |
| `F2S-ACT-ATTACK-001` | `attack_01` | 连击第 1 段 |
| `F2S-ACT-ATTACK-002` | `attack_02` | 连击第 2 段 |
| `F2S-ACT-ATTACK-003` | `attack_03` | 连击第 3 段 |
| `F2S-ACT-HIT-001` | `hit` | 1 |
| `F2S-ACT-DEATH-001` | `death` | 1 |

总数严格为十个 clip。

### 6.1 Idle — `F2S-ACT-IDLE-001`

- 语义：战斗准备或自然待机，保持角色识别、脚部稳定和武器控制；
- 循环：是；
- 建议时长：1.2–2.0 秒；
- 阶段：neutral → inhale/rise → exhale/settle → neutral；
- 接触：至少一只脚全程保持地面接触，除非描述明确悬浮；
- 首选：bone + mesh，头发/衣摆可有小幅次级运动；
- 常见素材：基础身体、武器休息附件；通常不新增动作专用身体图；
- 诊断：循环接缝、脚滑、呼吸导致徽记拉伸、武器握持漂移。

### 6.2 Run — `F2S-ACT-RUN-001`

- 语义：横版战斗跑动；
- 循环：是；
- 建议时长：0.55–0.85 秒/完整左右步周期；
- 阶段：near_contact → down → far_contact → up → near_contact；
- 接触：脚接触与离地分段明确；root 位移可作为参考，不自动决定游戏速度；
- 首选：bone + mesh + 次级摆动；
- 按需素材：极端膝肘修正、握持手、披风/长发动作附件；
- 诊断：脚滑、步幅不对称、地面穿透、武器与身体碰撞、循环跳变。

### 6.3 Jump — `F2S-ACT-JUMP-001`

- 语义：起跳及向上阶段，不包含无限空中循环；
- 循环：否；
- 建议时长：0.30–0.50 秒；
- 阶段：anticipation/crouch → takeoff_contact → extension → airborne_handoff；
- 接触：takeoff_contact 由用户确认；
- 首选：bone + mesh；
- 按需素材：深蹲膝胯修正、衣摆压缩/展开、武器惯性附件；
- 诊断：脚与地面断开时机、躯干体积、起跳轮廓和与 fall 的衔接。

### 6.4 Fall — `F2S-ACT-FALL-001`

- 语义：空中下降保持姿势；
- 循环：可选短循环或持续姿势；
- 建议时长：0.45–0.80 秒；
- 阶段：airborne_entry → falling_hold → exit_ready；
- 接触：无地面接触；
- 首选：bone + mesh，衣摆/头发向上响应；
- 按需素材：强风下发型/衣摆替换；
- 诊断：与 jump 衔接、肢体轮廓、武器脱手和循环漂移。

### 6.5 Dash — `F2S-ACT-DASH-001`

- 语义：短促横向突进的身体动作，不包含由游戏引擎决定的位移距离；
- 循环：否；
- 建议时长：0.25–0.45 秒；
- 阶段：compress → launch → travel → settle/handoff；
- 接触：起步脚和恢复接触候选由用户确认；
- 首选：bone + mesh；极端前倾可使用 torso/head corrective；
- 按需素材：前倾躯干、强风头发/衣摆、独立残影/VFX；
- 诊断：轮廓可读性、地面穿透、武器穿身、过度网格拉伸。

### 6.6 Attack 01 — `F2S-ACT-ATTACK-001`

- 语义：三段连击的快速起手，位移和攻击轨迹服从 WeaponSpec；
- 循环：否；
- 建议时长：0.40–0.65 秒；
- 阶段：neutral → anticipation → acceleration → contact_candidate → follow_through → recovery/cancel；
- 接触：武器接触和 active/hit 帧必须人工批准；
- 首选：bone + mesh + grip hand；按动作需要使用前臂/武器透视附件；
- 目标：启动清晰、轮廓不与 idle 混淆、可自然衔接 attack_02；
- 禁止：在 WeaponSpec 未批准时固定为“横斩”或任何具体武器技法。

### 6.7 Attack 02 — `F2S-ACT-ATTACK-002`

- 语义：延续第一击动量的中段攻击，姿势和轮廓必须与第一击可区分；
- 循环：否；
- 建议时长：0.50–0.80 秒；
- 阶段：chain_entry → counter_anticipation → acceleration → contact_candidate → follow_through → chain_exit；
- 接触：人工批准；
- 首选：hybrid；允许动作专用 torso/arm corrective；
- 目标：清楚展示重心转移并能衔接 attack_03；
- 诊断：动作方向重复、武器轨迹跳变、手部滑动、draw order 环。

### 6.8 Attack 03 — `F2S-ACT-ATTACK-003`

- 语义：三段连击的终结动作，最大力度和最长恢复，但仍服从 WeaponSpec；
- 循环：否；
- 建议时长：0.70–1.10 秒；
- 阶段：commitment → large_anticipation → acceleration → primary_contact_candidate → follow_through → recovery_to_idle；
- 接触：人工批准，可有一个或多个候选但不能由 AI 自定；
- 首选：hybrid；极端姿势允许 pose swap 或短 sequence；
- 按需素材：躯干扭转、强透视手臂/武器、终结关键姿势、独立 VFX；
- 诊断：轮廓、体积、序列与骨骼过渡、恢复时间、武器接触连续性。

### 6.9 Hit — `F2S-ACT-HIT-001`

- 语义：角色受击反应，不定义伤害系统；
- 循环：否；
- 建议时长：0.30–0.55 秒；
- 阶段：impact → recoil → settle/handoff；
- 接触：impact 帧由用户批准；受击方向来自动作描述；
- 首选：bone + mesh，脸部/头发/衣物修正按需；
- 诊断：地面脚滑、武器脱手、面部身份漂移和与 death 的混淆。

### 6.10 Death — `F2S-ACT-DEATH-001`

- 语义：从可战斗状态进入不可逆倒地/结束姿势，不定义游戏状态机；
- 循环：否；末帧可保持；
- 建议时长：1.0–1.8 秒；
- 阶段：loss_of_control → descent → ground_contact → settle → final_hold；
- 接触：身体首次地面接触和最终静止由用户批准；
- 首选：hybrid；倒地引发大视角变化时优先动作专用附件/短 sequence；
- 按需素材：侧躺躯干/头部、受压衣发、武器落地状态；
- 诊断：地面穿透、强拉伸、层级翻转、武器状态、最终轮廓和画布裁切。

## 8. AI 动作提示词包

### 8.1 正式输出定义

`F2S-PROMPT-PACK-001` 是目录或 ZIP 交付物：

```text
prompt-pack/
  manifest.json
  specs/
    style-spec.json
    character-spec.json
    weapon-spec.json
    runtime-spec.json
    motions/*.json
    assets.json
  prompts/
    00-global-identity.md
    01-negative.md
    actions/<action>/<pose-id>.md
    assets/<asset-spec-id>.md
  references/
    reference-manifest.json
    thumbnails/              # 用户选择是否包含
  import-map.json
  README.md
```

`manifest.json` 必须记录 schema、项目 ID、母版哈希、生成时间、语言、提示词模板版本、未决占位符和每项 PromptItem 状态。提示词包不得包含 API token、外部服务凭据或未经用户选择的原始大图。

### 8.2 全局身份提示词模板

下面模板由系统填入 StyleSpec 值；方括号是结构字段，不允许静默猜测：

> 以已批准角色母版 [master-reference] 为唯一身份与服装真值。生成同一个二次元类人角色，严格保持 [头身比]、[脸型]、[发型与发色]、[瞳色]、[服装剪裁与纹样]、[色板]、[线宽与线色]、[赛璐璐阴影层数]、[固定光向] 和 [primary_weapon 外观]。固定横版游戏侧视、角色面向右、近似正交投影、完整全身、固定画布 [W×H]、固定角色像素高度 [Hpx]、双脚地面线 y=[value]、角色原点 [x,y]、透明背景。只改变当前提示项明确指定的姿势或隐藏区域；未指定的可见设计不得重画。

### 8.3 全局负向提示词模板

> 不要改变角色身份、脸型、年龄、体型、发型、发色、瞳色、服装、徽记、配色或武器设计；不要正面、背面、三分之四视角、俯视、仰视、广角、鱼眼或动态镜头；不要裁切头、手、脚或武器；不要第二个角色、重复肢体、多余手指、缺失关节、断裂武器、左右侧混乱；不要写实光照、渐变背景、投影、地面纹理、文字、水印、UI、边框；不要把 weapon trail、烟雾、火焰、残影与基础角色主体合层；不要改变画布、人物尺度、地面线、透明背景和 pivot；不要用非计划的模糊、强景深、过曝或运动模糊掩盖结构。

全局负向约束允许一个窄化的局部覆盖：只有当 AssetSpec 明确为 `pose_swap`、`sequence_frame` 或独立 `vfx`，MotionSpec 指定 smear 的目的/部位/帧范围，且 PromptItem 标记 `localNegativeOverride.allowSmear=true` 时，动作专用的定向 smear 才可出现。该覆盖只作用于对应局部资产，不解除身份、画布、透明背景、单角色、武器连接和“不得掩盖结构错误”等约束；覆盖原因和审批必须进入 PromptPack。

系统还应根据 StyleSpec 加入角色专属负向项，例如“不得删除非对称发饰”。负向提示不保证模型服从，因此导入后仍需视觉与几何检查。

### 8.4 单关键姿势模板

> [全局身份提示词]。当前输出仅生成动作 [action-id] 的关键姿势 [pose-id]。动作语义：[用户动作描述]。阶段：[phase]，建议归一化时间 [t]。角色重心：[center-of-mass description]；近侧脚：[contact description]；远侧脚：[contact description]；躯干：[torso intent]；近/远侧手臂：[arm intent]；头部与视线：[head intent]；primary_weapon：[来自已批准 WeaponSpec 的握持、轨迹和透视约束]；头发和衣物：[secondary motion intent]。保持完整清晰轮廓，关节结构可读，武器与手不分离。只输出一个角色、一个姿势、透明背景 PNG 候选，不制作接触表或多格 sprite sheet。

生产资产优先每张图一个姿势。多格接触表只能用于动作讨论，因为格间分辨率、角色尺度和身份一致性难以作为正式图层素材。

### 8.5 隐藏区域补全模板

> 以 [source-layer] 的可见像素和 [master-reference] 为不可修改真值。仅在遮罩 [mask-reference] 指定的透明隐藏区域内补全 [body semantic]，延续相同线宽、颜色、赛璐璐阴影和局部纹理；补全部分必须跨过关节中心并提供 [overlap requirement]。遮罩外像素逐像素保持不变。输出与源层完全相同画布、位置、alpha 语义和分辨率的透明 PNG，不增加其他身体部位、背景、阴影或特效。

### 8.6 动作专用修正附件模板

> 生成角色 [action-id]/[pose-id] 所需的局部修正附件 [semantic]。参考批准母版和当前重组预览，只绘制 [region bounds] 内的 [部位]，目标是解决 [diagnostic-id: 透明洞/强透视/轮廓坍塌/拓扑变化]。与相邻部件保持 [overlap]，pivot 固定为 [x,y]，画布和母版一致，其他区域透明。不得重画无关部位，不得附带特效。

## 9. 各动作提示词要点

系统为每个关键姿势生成独立 PromptItem；下面是动作语义片段，与全局模板合并。

| 动作 | PromptItem 建议 | 动作语义片段 |
| --- | --- | --- |
| idle | neutral/rise/settle | 稳定战斗待机，呼吸只造成轻微胸肩和次级物变化，至少一脚锁地，primary_weapon 保持批准的休息握持 |
| run | near_contact/down/far_contact/up | 横版战斗跑步完整步态，重心前移，脚接触清晰，身体和 primary_weapon 有受控惯性，循环首尾一致 |
| jump | crouch/takeoff/extension | 起跳压缩、脚离地接触和向上伸展，保持侧视与完整轮廓，不自行定义跳跃高度 |
| fall | entry/hold | 下降姿态，肢体保持可读，头发和衣摆向上响应，没有地面接触 |
| dash | compress/travel/settle | 短促突进姿态，清晰前倾和压缩/展开，位移距离留给游戏逻辑，残影单独生成 |
| attack_01 | anticipation/contact/follow/recover | 三连击快速起手，武器运动完全使用 WeaponSpec，占位未解决时禁止生产生成 |
| attack_02 | chain_entry/contact/follow/exit | 延续第一击动量的第二击，重心转移和轮廓不同于第一击，保持可衔接第三击 |
| attack_03 | anticipation/contact/follow/final | 终结攻击，最大力度和轮廓变化，允许经计划的姿势替换，命中和恢复由用户审核 |
| hit | impact/recoil/settle | 来自 [direction] 的受击反应，先冲击再回弹，不添加伤口、血液或伤害数值，除非用户规格明确 |
| death | loss/descent/contact/final | 失去控制、落地、安定和最终保持，避免穿地和裁切，武器状态依 WeaponSpec |

### 9.1 攻击提示词阻塞规则

如果以下任一项为空，attack PromptItem 只能导出为带明显 `UNRESOLVED` 标签的开发草案：

- `motionFamily`；
- 主/副握持模式；
- 武器视觉长度和重量级；
- rest 与攻击握持附件；
- 接触区域语义；
- 用户提供的 attack 动作描述或关键姿势图。

这避免系统把未确认的单武器强行设计成剑术。

## 10. 外部图片回导与一致性检查

### 10.1 回导关联

用户回导图片时必须：

1. 选择 PromptItem 或 AssetSpec；
2. 填写可取得的外部工具、模型、版本、seed 和采样参数；若无法取得，必须选择 `unknown_external` 并填写用户来源声明，系统不得猜测或补造；
3. 系统计算文件哈希和画布信息；
4. 检查角色尺度、地面线、朝向、alpha 和裁切；
5. 与母版比较 lockedFeatures、色板、轮廓和局部纹样；
6. 由用户接受为候选、退回或进入修正；
7. 接受候选仍不等于资产批准。

### 10.2 一致性诊断

系统至少产生：

- 画布、角色包围盒和地面偏移；
- 颜色差、线宽差和透明边缘异常；
- 脸、发饰、徽记、服装纹样等身份锚点差异；
- 手指、关节和武器连接的结构风险；
- 近/远侧肢体语义冲突；
- 与参考关键姿势的轮廓相似度，仅作为提示；
- 可能混入背景、文字、VFX 或第二角色的区域。

诊断是辅助证据，不可自动通过身份和关键姿势审批。

## 11. 临时 Rig 反馈闭环

### 11.1 闭环步骤

1. 用已批准基础层和占位附件构建 starter Rig；
2. 根据 MotionSpec 的关键姿势与阶段建立粗动画；
3. 对十个动作批量渲染证据帧；
4. 运行透明洞、拉伸、轮廓、透视、draw order、脚滑和武器连接诊断；
5. 将每个问题分类为 Rig 修复、权重修复、动作修复或新增素材；
6. 只有新增素材类问题才创建 AssetSpec 与 PromptItem；
7. 用户外部生成并回导，完成分层/附件和 Rig 更新；
8. 重新渲染受影响动作，而不是无条件重跑全项目；
9. 全部阻塞问题关闭后，进入关键姿势和命中帧审批。

### 11.2 问题到动作映射

| 问题 | 首选处理 | 升级条件 |
| --- | --- | --- |
| 小范围透明洞 | 调重叠/pivot/权重 | 原图没有可用隐藏像素时补全 |
| 肘膝轮廓变尖 | 网格/权重调整 | 目标姿势仍失真时 corrective |
| 强透视前臂像纸片 | 不继续拉伸 | 直接生成 foreshortening 附件 |
| 武器从手中滑出 | socket/约束/握持姿势 | 透视或握持形态变化时附件 |
| 发/衣摆穿身 | draw order/次级骨 | 拓扑改变或极端压缩时附件/sequence |
| 角色转面 | 不镜像、不扭曲基础图 | 新方向 Rig；首版标记超范围 |
| smear/终结冲击 | pose swap/sequence | 由用户批准其艺术风格和帧数 |

| `F2S-GATE-CONTENT-005` | PromptPack | 无未解决占位符，引用的是当前批准规格，包可追踪且未调用云服务 |
| `F2S-GATE-CONTENT-006` | 关键姿势 | 动作轮廓、接触、武器握持和身份由用户逐动作批准 |
| `F2S-GATE-CONTENT-007` | 命中帧 | 三个攻击的候选帧由用户确认，修改后会失效 |

审批不是“看过”状态，StyleSpec/MotionSpec/AssetSpec等内容对象不得持久化可自报的approved真值。每次批准绑定具体target、完整DependencySnapshot、认证PrioritySlice、stable GateOutcome、DecisionPolicy、所依赖的ack记录和静态GovernedWaiverBinding；full PriorityRegistry只作为当次transport observation。active投影必须用current registry逐slice subject、WaiverRegistry、ActorRegistry中相关credential角色、clockEpoch/checkpoint与effectiveNow动态重验，不能信任历史checkedAt。批准动作不反向改变基础GateReport。实际依赖、关联slice、相关credential角色或被引用waiver改变时下游`stale`/批准inactive，仅全局revision或无关entry改变不失效。

## 13. 内容质量检查表

### 13.1 母版

- 是否只有一个角色、透明背景、完整全身且无裁切；
- 是否严格侧视、近远侧肢体可辨；
- 是否存在动态光、强透视、遮挡过多或无法拆分的复杂结构；
- 身份锚点和非对称元素是否登记；
- `primary_weapon` 是否可独立拆分、握持点是否可见。

### 13.2 素材

- 可见像素是否保持；
- 隐藏补全是否明确标记；
- 关节是否跨中心重叠；
- pivot、画布和地面是否一致；
- 图层命名是否使用稳定英文语义；
- 动作专用附件是否只服务有证据的需求；
- VFX 是否与身体分离；
- 是否存在未引用或重复的资产。

### 13.3 动作

- 十个动作是否全部覆盖；
- 关键姿势与动作描述是否一致；
- 跑步和 idle 循环是否无缝；
- jump/fall 和连击衔接是否可读；
- 脚、武器和身体接触是否稳定；
- attack 的 anticipation/active/recovery 是否由用户确认；
- death 最终姿势是否保持且不裁切；
- 极端姿势是否选择了合适的混合表现方式。

## 14. 版本与追踪

- 修改 StyleSpec 会使全部 PromptItem 和视觉派生资产待复核；
- 修改 CharacterSpec 只使涉及区域及依赖 Rig/动作失效；
- 修改 WeaponSpec 会使所有武器附件和 `attack_01`、`attack_02`、`attack_03`，以及实际依赖武器的 idle/run/jump/fall/dash 失效；
- 修改某一 MotionSpec 只重算关联 AssetSpec、PromptItem、动画和游戏性审批；
- 每个 PromptItem 保留 templateVersion、resolvedTextHash 和规格 revision；
- 每个回导候选保留 promptItemId 和外部生成来源；缺少外部生成元数据时保留 `unknown_external` 与用户声明，内部 AI 任务则必须保留完整 provenance；
- 每次诊断修复保留 before/after 证据帧；
- 导出提示词包必须包含当前未决项，不得为了“完整”而删除警告。
