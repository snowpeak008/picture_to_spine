# 领域逻辑与工作流编排

## 1. 目的和领域边界

本文档定义 Production Assist 的业务对象、不变量、人工门禁、审批失效、工作流编排和幂等命令。它回答“什么时候一个结果有资格成为下一阶段的输入”，而不回答文件如何落盘或 Worker 如何启动。

领域不包含：

- 第三方生图 API 调用；
- AI 自主批准母版、Rig、关键姿势或命中帧；
- Unity、Godot、Cocos 或自研引擎运行时逻辑；
- Spine Runtime 语义、`.spine` 私有格式或 Spine 激活管理；
- 对用户未提供的动作视频、BVH 或 3D 资产的假设。

ADR owner按 `F2S-ADR-REGISTRY-001` 逐项解析：本文只读引用跨领域ADR，并唯一拥有frontmatter枚举的三个DOMAIN implementation-local ADR canonical body。

## 3. 领域决策

以下三项是领域编排子系统的 implementation-local ADR；canonical body 与语义由本文维护。`F2S-DOC-ADR-001` 只在全局 ADR owner registry 登记归属并提供跨文档索引，不复制或改写这些决定。

| 决策 ID | 决策 | 原因 |
| --- | --- | --- |
| `F2S-ADR-DOMAIN-001` | 不存储一个线性“当前阶段”作为项目真值；每类产物独立拥有状态，工作流可用性由 GateEvaluator 派生 | Production Assist 允许回退、并行修复和部分失效，单一阶段枚举会丢失信息 |
| `F2S-ADR-DOMAIN-002` | 审批绑定产物内容哈希、revision、规则版本和下游依赖快照 | 名称或当前指针不足以证明用户批准了哪个结果 |
| `F2S-ADR-DOMAIN-003` | 后台 Job 成功只产生“已验证并持久到 CAS、但尚未绑定领域对象”的 `JobOutputArtifact`；另一个带 expected revision 的 `Register*Candidate` 事务才能产生 candidate | 技术执行成功不等于领域注册成功，更不等于艺术、动画或玩法批准 |

## 4. 与 04 完全一致的五维状态模型

`F2S-DOC-UX-001` 是工作流词汇 owner。领域只保存必要事实或可重建投影，禁止把五个维度压成一个枚举。

### 4.1 工作流阶段

项目阶段严格派生为：

`draft → master_approved → content_approved → layers_approved → rig_approved → motion_review → gameplay_approved → export_ready → exported`

`spine_verified` 不属于阶段。阶段表达已到达的生产里程碑；上游变化通过新鲜度和阻塞维度表达，不伪造历史倒退，也不持久第二套项目阶段真值。

### 4.2 对象生命周期

```text
missing
  └─ draft
       └─ candidate
            ├─ rejected
            ├─ approved
            └─ superseded

approved
  └─ superseded
rejected
  └─ draft
```

1. `candidate` 可预览、修改、比较，不能进入 release export。
2. `approved` 必须存在匹配内容哈希、GateReport、输入revision且尚未被后继`revoke`撤销的active `approve` ApprovalRecord；`reject/ack_warning/revoke`绝不满足批准条件。
3. `rejected` 保留原因和证据，可派生新 `draft`；`superseded` 只读。
4. 生命周期中不存在 `stale`、`blocked`、`failed` 或 `unverified`。

### 4.3 新鲜度、阻塞与导出验证

| 维度 | 允许值 | 领域来源 |
| --- | --- | --- |
| Freshness | `current / stale` | InvalidationEngine 比较精确依赖 revision/hash；`stale` 不改生命周期 |
| Blockage | `ready / blocked` | GateEvaluator 根据 blocker 推导；必须携带 issue ID 与解除条件 |
| ExportVerification | `not_exported / exported_unverified / verification_running / spine_verified / incompatible` | ExportRun 与真实 4.2.43 evidence；Job 失败不等于 verification 值 |

Job 的 `requested/queued/running/cancel_requested/cancelled/failed/succeeded/interrupted` 是执行维度，不覆盖对象生命周期。UI 组合投影必须分别给出 stage/lifecycle/freshness/blockage/export verification。

### 4.4 唯一映射

| 领域事实 | 04 投影 | 规则 |
| --- | --- | --- |
| 必需对象不存在 | Lifecycle=`missing` | GateEvaluator 推导 |
| 未提交工作版 | Lifecycle=`draft` | 可持久 autosave，但不是 candidate |
| `Register*Candidate` 事务成功 | Lifecycle=`candidate` | Job `succeeded` 本身不产生此状态 |
| candidate 满足审查前置 | Lifecycle=`candidate` + ReviewReadiness=`needs_review` | `needs_review` 只由 GateReport 推导为 UI 视图，不是 Lifecycle 或额外存储终态 |
| 当前hash有active `approve` record | Lifecycle=`approved` | record必须是该target决策流最新有效批准：kind=approve、hash/revision/GateReport匹配，且无后继revoke |
| 最新决定为`reject` | Lifecycle=`rejected` | 保留candidate与审批流历史，不伪造approved |
| 当前批准被`revoke` | Lifecycle=`candidate` | 若对象已superseded则保持superseded；精确下游Freshness=`stale`且release gate重算 |
| approved 依赖改变 | Lifecycle=`approved` + Freshness=`stale` | 保留审批历史 |
| GateReport 有 blocker | Blockage=`blocked` | 不改生命周期 |
| Job 失败 | Job=`failed` | 不改对象或导出验证 |
| 已输出但无外部证据 | Stage=`exported` + ExportVerification=`exported_unverified` | 只有真实证据可转 `spine_verified` |

## 6. 依赖图和失效传播

```text
SourceImage
   └─ MasterRevision (approved)
       ├─ CharacterSpec / StyleSpec / WeaponSpec / RuntimeSpec
       │   └─ MotionSpec[10]
       │       └─ AssetSpec[]
       │           └─ PromptPack
       └─ LayerSet
           └─ RigProject
               └─ AnimationClip[10]
                   ├─ GameplayMarker / approvals
                   └─ Diagnostics
                       └─ ExportRun
```

实际图保存精确边，上图只是主干。`InvalidationEngine` 使用产物的 `derivedFrom` 和变更类型计算影响：

| 变更 | 必须失效 | 可保持有效 |
| --- | --- | --- |
| 替换/修改批准母版 | LayerSet、RigProject、全部 Clip、marker、诊断、导出 | 项目身份、用户设置；StyleSpec 需用户重新确认映射 |
| StyleSpec 线条/色彩规则变更 | PromptPack、相关新图候选、LayerSet QA、导出报告 | Rig 几何可保留但标记需重渲染检查 |
| WeaponSpec 具体化 | attack AssetSpec、PromptPack、grip/contact、attack Clip marker | idle/run 等无武器依赖内容，由实际边决定 |
| AssetSpec 变更 pivot/尺寸/父骨 | 相关 Layer attachment、Rig 映射、使用该素材的 Clip | 无使用关系的动作 |
| Rig 骨骼结构变更 | 受影响轨道、marker、诊断、导出 | 能精确重映射的不相干 Clip，必须有映射报告 |
| Clip 时长/关键帧变更 | 该 Clip 的关键姿势、命中帧、接触、QA 和导出 | 其他 Clip |
| 诊断规则版本变更 | 相关 DiagnosticRecord 和 QA 门禁 | 艺术产物本身 |

失效计算必须在命令提交前向用户展示摘要，提交时与主修改位于同一 ProjectTransaction。

## 7. 人工门禁

| 门禁 | 输入 | 必显示的检查 | 通过后产物 |
| --- | --- | --- | --- |
| Master Gate | 导入原图、规范化候选 | 尺寸、alpha、色空间、方向、地面、差异 | approved MasterRevision |
| Spec Gate | Character/Style/Weapon/Runtime/Motion | schema、未决字段、十动作覆盖 | approved spec snapshots |
| Asset Plan Gate | AssetSpec 集合 | 动作覆盖、重复、无使用、表现策略、placeholder | approved AssetPlan |
| Layer Gate | LayerSet | 重组差异、洞、边缘污染、重叠、AI 像素 | approved LayerSet |
| Rig Gate | RigProject | 成环、孤儿、mesh、weight、draw order、临时动作试跑 | approved RigProject |
| Pose Gate | AnimationClip | 动作阶段、轮廓、脚/武器接触、循环或终止 | approved clip revision |
| Gameplay Gate | GameplayMarker | 命中帧、anticipation/active/recovery、weapon_contact | approved markers |
| Export Gate | 全部产物 | 缺失/过期审批、诊断、兼容、许可、目标版本 | release export 资格 |

退回门禁必须包含原因、对象链接和建议修复。“我知道了”不是审批；接受warning风险必须产生kind=`ack_warning`的单独ApprovalRecord，它永不构成active approve；error不可用普通风险接受越过。

## 8. GateEvaluator 接口 `F2S-IFC-201`

```rust
enum GateKind {
    Master, Specs, AssetPlan, Layers, Rig, Pose(ActionId),
    Gameplay(ActionId), Export(ExportProfileId),
}

struct GateReport {
    gate: GateKind,
    evaluated_revision: ProjectRevision,
    status: GateStatus, // reviewable | passed；仅为审查结果投影
    freshness: Freshness, // current | stale
    blockage: Blockage, // ready | blocked
    blockers: Vec<GateFinding>,
    warnings: Vec<GateFinding>,
    evidence_hashes: Vec<ContentHash>,
    rule_set_version: RuleSetVersion,
}

trait GateEvaluator {
    fn evaluate(&self, project: &ProjectSnapshot, gate: GateKind) -> GateReport;
}
```

GateReport 是派生结果，可缓存但不是不可重建的事实。`GateStatus` 不包含 `stale`、`blocked` 或导出验证；这些分别位于 `freshness`、`blockage` 和 ExportVerification。

`GateReportHashV1`固定为SHA-256的域分隔输入：`"f2s-gate-report-v1\0" || RFC8785/JCS canonical JSON`。JSON必须包含schemaVersion、gate、evaluatedRevision十进制字符串、status、freshness、blockage、ruleSetVersion、blockers、warnings和evidenceHashes；finding按`severity/ruleId/objectRef/evidenceHash`稳定排序，evidence hash去重后字典序排序，禁止本地化message或map迭代顺序影响hash。任何字段增删或语义变化必须新建hash版本和迁移，不能沿用V1。

批准命令同时携带并绑定当前GateReport的hash、evaluatedRevision、ruleSetVersion和完整evidence hash集合。Rust在同一项目revision上重新评估、规范化并计算hash；任一显式字段或重算hash不一致，请求必须失败并重新检查。

## 9. ApprovalService 接口 `F2S-IFC-202`

```rust
enum ApprovalDecision {
    Approve,
    Reject,
    AcknowledgeWarning,
}

struct ApprovalRequest {
    target: ApprovalTarget,
    target_revision: u64,
    target_hash: ContentHash,
    gate_report_hash: ContentHash,
    gate_report_revision: ProjectRevision,
    gate_report_rule_set_version: RuleSetVersion,
    gate_report_evidence_hashes: Vec<ContentHash>,
    decision: ApprovalDecision,
    note: Option<String>,
}

trait ApprovalService {
    fn decide(&self, request: ApprovalRequest, ctx: CommandContext)
        -> Result<ApprovalRecord, DomainError>;
    fn revoke(&self, approval_id: ApprovalId, reason: String, ctx: CommandContext)
        -> Result<ApprovalRecord, DomainError>;
}
```

首版是本地单用户，`actor` 仍使用稳定本地身份，方便审计和未来迁移。该身份不是云账号，不需要登录。

每个target revision拥有单调sequence的不可变决策流。`ActiveApprovalHead`不是另存的可变真值，而是按sequence重放得到：最近的匹配`approve`仅在没有后继指向它的`revoke`、目标hash/revision和GateReport仍匹配时有效。`reject`把当前candidate投影为`rejected`；`ack_warning`只满足可确认warning，不改变Lifecycle；`revoke`必须引用active approval ID，撤回后当前对象回到`candidate`（已superseded者除外），所有依赖该批准的下游Freshness变`stale`并使release export重新阻断。重复/越序/撤回非active记录均拒绝且不追加事件。

## 10. InvalidationEngine 接口 `F2S-IFC-203`

```rust
struct ChangeSet {
    changed: Vec<ArtifactRevisionRef>,
    semantic_changes: Vec<SemanticChange>,
}

struct InvalidationPlan {
    stale_targets: Vec<InvalidatedTarget>,
    preserved_targets: Vec<ArtifactRevisionRef>,
    requires_confirmation: bool,
    explanation: Vec<InvalidationReason>,
}

trait InvalidationEngine {
    fn plan(&self, project: &ProjectSnapshot, changes: &ChangeSet)
        -> Result<InvalidationPlan, DomainError>;
    fn apply(&self, project: &mut Project, plan: InvalidationPlan)
        -> Result<Vec<DomainEvent>, DomainError>;
}
```

影响计算必须是确定性规则，不交给 LLM 决定。LLM/VLM 可以提供变更摘要候选，但无权决定哪些审批仍有效。

## 11. AssetPlanner 与 PromptComposer `F2S-IFC-204`

```rust
trait AssetPlanner {
    fn draft(&self, input: PlanningInput) -> Result<AssetPlanCandidate, DomainError>;
    fn validate_coverage(&self, plan: &AssetPlanCandidate)
        -> AssetCoverageReport;
}

trait PromptComposer {
    fn compose(&self, approved: ApprovedPlanningSnapshot)
        -> Result<PromptPackCandidate, DomainError>;
}
```

`PlanningInput` 必须包含已批准或显式候选标识的 CharacterSpec、StyleSpec、WeaponSpec、RuntimeSpec 和十个 MotionSpec。规则引擎决定 bone/mesh/attachment/sequence/new-view/vfx 能力类别；LLM 可负责把人类动作文本转换为候选字段和可读提示词。

PromptComposer 只产生提示词包，不提交 HTTP 请求、不保存第三方 API key、不把生图成功作为工作流前置。外部返回的图片只能经过用户重新导入并绑定 PromptItem/AssetSpec。

### 11.1 Candidate provenance 硬门

所有 `Register*Candidate` 命令先按来源执行不可绕过的 ProvenanceSpecification：

- `internal_local` 与 `internal_private_remote` 必须具有 Provider/Worker ID、协议版本、模型/算法 ID 与精确版本、包/manifest hash、参数、seed（适用时）、执行设备/位置、JobId、输入/遮罩/Spec revision 与 hash、输出 hash、开始/结束时间、重试/取消父链和所有质量降级。任一字段缺失时 Job 输出只能隔离，不能注册 candidate。
- 产品外部生成/编辑的图片若无法提供模型、版本、seed 等信息，必须存 `sourceKind=unknown_external`、文件 hash、导入时间和非空用户来源声明；不得伪造内部 provenance 或宣称可复现。
- 手工派生 candidate 保留父产物和编辑命令链；来源标签不可因后续手工修改而被擦除。

`Register*Candidate` 的成功事件必须携带经过 schema 验证的 provenance hash。GateEvaluator 持续向人工审查和导出报告暴露 `unknown_external` 警告。

## 12. 工作流编排

### 12.1 用例主干

```text
Create/Open Project
  → Import Source Image
  → Approve Master
  → Edit/Approve Specs and 10 MotionSpecs
  → Draft/Approve AssetPlan
  → Compose/Export PromptPack (optional external image round-trip)
  → Build/Repair/Approve LayerSet
  → Build/Repair/Approve RigProject
  → Draft/Edit/Approve 10 AnimationClips
  → Approve GameplayMarkers
  → Run QA and resolve/accept warnings
  → Preflight/Export
  → Optional Spine 4.2.43 validation
```

主干是默认导航，不是禁止回退的向导。用户可在任意时候修改上游，但在提交前必须看到 InvalidationPlan。

### 12.2 后台 Job 与领域迁移

```text
Domain command: RequestLayerSuggestion
  → create persisted Job(requested)
  → scheduler runs provider
  → provider returns artifacts
  → verify schema/hash/generation provenance
  → persist immutable JobOutputArtifact in CAS
  → persist Job(succeeded)
  → separate domain command: RegisterLayerCandidate(expected project revision, output hash)
  → human reviews
  → domain command: ApproveLayerSet or RejectLayerSet
```

Worker 不能直接触发产物状态迁移。Job `succeeded` 与 `Register*Candidate` 是两个独立、可审计的事务：

- `succeeded` 只表示输出通过 schema/hash/安全/provenance 校验，已经作为不可变 `JobOutputArtifact` 持久到 CAS；它没有绑定 AssetSpec/LayerSet，也没有创建 project revision candidate。
- `Register*Candidate` 必须携带 output hash、target ID、`expected_revision` 和 idempotency key；成功后才创建 candidate revision。
- 注册遇到 revision 冲突时 Job 仍保持 `succeeded`，输出处于 `unbound`，用户可刷新后重新注册或显式丢弃；不得重跑昂贵推理来掩盖事务冲突。
- 注册前崩溃可从 `JobExecutionRecord` 和 CAS 重建；保留期到期前不能把未绑定输出 GC。
- 如输出校验失败，Job 进入 `failed` 或 `interrupted`，领域中不会出现 candidate。

### 12.3 补素材闭环

```text
Temporary Rig/Animation Preview
  → DiagnosticRecord(transparent hole/stretch/silhouette/contact/...)
  → Create SupplementAssetRequest
  → merge/deduplicate into AssetSpec candidate
  → user approves
  → PromptItem or local repair task
  → import/register new candidate
  → map into LayerSet/Rig
  → rerun affected diagnostics only
```

补素材不默认重做整套图；其 AssetSpec 必须引用证据诊断、所需动作和表现策略。

### 12.4 持久化 Job 状态与终态仲裁

Job 只允许以下持久状态：

```text
requested
queued
running
cancel_requested
cancelled
failed
succeeded
interrupted
```

| 当前状态 | 合法后继 | 说明 |
| --- | --- | --- |
| `requested` | `queued`, `cancel_requested`, `failed` | 请求和资源/协议校验已持久；校验失败直接 failed |
| `queued` | `running`, `cancel_requested`, `failed`, `interrupted` | 调度器尚未将任务交给 Provider |
| `running` | `cancel_requested`, `failed`, `succeeded`, `interrupted` | 进度/阶段是投影，不是额外持久状态 |
| `cancel_requested` | `cancelled`, `failed`, `interrupted` | 不允许直接返回 `running`；后到 success 由下述仲裁 |
| `cancelled` | 无 | 终态 |
| `failed` | 无 | 终态；重试创建新 JobId |
| `succeeded` | 无 | 终态；只表示结果已验证并作为未绑定 `JobOutputArtifact` 持久到 CAS，不表示 candidate 或 approved |
| `interrupted` | 无 | 本次执行终态；恢复/重试使用新 JobId 或明确 resume operation |

`cancel_requested` 和 success 竞态由 Rust 持久化调度器按顺序号仲裁：

1. 如已通过 schema/hash/安全/provenance 验证且 CAS 持久完成的 success 终态先于 cancel command 持久，`succeeded` 胜出，后到取消返回“已终态”。
2. 如 `cancel_requested` 先持久，后到 success 不能转 `succeeded`；输出进 quarantine，Provider 确认停止后转 `cancelled`。
3. 如断连/崩溃使持久顺序无法证明，终态必须是 `interrupted`，不猜测 cancelled 或 succeeded。
4. Worker/Remote 只提供终态候选；它们无权直接写 Job 持久状态。

### 12.5 Job 权威存储

Job 状态的唯一可恢复事实使用 `F2S-DOC-STORE-001` 的 `F2S-SCH-009 JobExecutionRecord` 与按序不可变 transition 记录；`redb` 只保存队列/查询投影。记录至少绑定 JobId、sequence、previous record hash、请求与幂等键、base project revision、Provider/model/protocol provenance、重试/恢复父链、状态和 JobOutputArtifact hash。删除/损坏 `redb` 后必须从这些权威记录重建 queued/running/interrupted/terminal 投影，不能根据 Worker 进程或 sandbox 文件猜测状态。

## 13. 命令、事件和幂等

### 13.1 必要命令族

- Project：`CreateProject`、`OpenProject`、`SaveAs`、`RecoverProject`；
- Import/Master：`ImportSourceImage`、`SetGroundAndScale`、`ApproveMaster`、`RejectMaster`；
- Specs：`UpdateCharacterSpec`、`UpdateStyleSpec`、`UpdateWeaponSpec`、`UpdateRuntimeSpec`、`UpdateMotionSpec`；
- Planning：`DraftAssetPlan`、`OverrideRepresentationStrategy`、`ApproveAssetPlan`、`ComposePromptPack`；
- Layers：`RegisterLayerCandidate`、`ApplyMaskStroke`、`RelinkExternalLayer`、`ApproveLayerSet`；
- Rig：`EditBoneTree`、`EditSlot`、`EditAttachment`、`EditMesh`、`ApplyWeightStroke`、`ApproveRig`；
- Animation：`EditKeyframes`、`EditCurve`、`SetGameplayMarkerCandidate`、`ApproveClip`、`ApproveGameplayMarkers`；
- QA/Export：`RunDiagnostics`、`AcceptWarningRisk`、`RequestExport`、`RegisterSpineVerification`。

每个命令包含 command ID、project ID、expected revision、idempotency key、actor 和语义 payload。重复 idempotency key 必须返回首次结果，不再执行一次。

### 13.2 领域事件

必须有稳定 event type 的事件至少包含：

- `MasterApproved`、`MasterSuperseded`；
- `SpecChanged`、`AssetPlanApproved`、`PromptPackComposed`；
- `LayerCandidateRegistered`、`LayerSetApproved`；
- `RigChanged`、`RigApproved`；
- `ClipChanged`、`ClipApproved`、`GameplayMarkersApproved`；
- `ApprovalBecameStale`、`WarningRiskAccepted`；
- `ExportRequested`、`ExportCompleted`、`SpineVerificationRecorded`。

事件用于审计、索引更新和 UI 刷新，不承诺首版使用完全事件溯源重建 Project。项目快照仍是快速打开的权威形式，事件日志是不可变审计和恢复辅助。

## 14. 领域不变量

1. 每个 Project 必须且只有一个当前母版指针，指向不可变 MasterRevision。
2. 十个基线 action ID 必须存在，可标记尚未制作，不可被删除。
3. 只有 Lifecycle=`approved`、Freshness=`current`、Blockage=`ready` 的产物可进 release export。
4. 每个approved产物都必须有匹配内容哈希/revision/GateReport且未被撤回的active `approve` ApprovalRecord。
5. 内部 AI/Worker 输出只有通过完整 GenerationProvenance 硬门才能注册 candidate；外部缺失信息必须是 `unknown_external` 加用户声明；手工派生版本引用父产物且不可擦除来源。
6. 母版可见像素的 AI 改写必须为显式候选并有差异证据。
7. GameplayMarker 不能在修改 clip 时长或相关关键帧后保持 approved。
8. AssetSpec 的 pivot、父骨建议、重叠和 requiredBy 不可全部缺失后通过 Asset Plan Gate。
9. `primary_weapon` 未具体化时，不允许将武器接触/命中数据标为 release approved。
10. 官方 Spine 验证状态只能由实际 4.2.43 CLI 运行证据产生。

## 15. 失败语义

| 失败类型 | 是否可重试 | 领域结果 |
| --- | --- | --- |
| 输入验证失败 | 修改输入后 | 命令无事件、revision 不变 |
| 版本冲突 | 刷新后人工重放 | 不覆盖、不自动合并 |
| GateReport 过期 | 重新评估 | 审批不产生 |
| Worker 失败/取消 | 视技术错误 | 不注册 candidate，不修改已批准产物 |
| Worker 结果 schema/hash 错误 | 不盲目自动重试 | 结果隔离，记录安全/完整性错误 |
| 导出 preflight 错误 | 修复项目后 | 禁止 release export，可保留不对外的调试 staging |
| Spine CLI 缺失/版本错误 | 配置合法 4.2.43 后 | 可有 `exported_unverified`，不得有 `verified` |
| 用户撤回审批 | 不适用 | 追加引用active approval的revoke；当前对象Lifecycle回到`candidate`（superseded除外），精确下游Freshness=`stale`，历史不删除 |

## 17. 完成定义

- 上述 15 类领域对象均有 Rust 类型、JSON Schema、不变量和迁移策略。
- 每个人工门禁都有 GateReport、证据和 ApprovalRecord，不存在 UI-only 批准。
- 失效引擎可以精确解释每个 Freshness=`stale` 对象的原因，而不改写生命周期。
- Worker、本地规则、手工编辑都通过同一 candidate 注册和审批路径。
- 十个基线动作在无 AI/GPU/网络下也能被建立、编辑、审核和导出。
- 导出门禁能区分 release-ready、exported-unverified 和 incompatible，不伪造 Spine 官方验证。
