# 产品章程、范围与边界

## 1. 章程摘要

Flash to Spine Production Assist 是一款面向 Windows 的闭源商业桌面软件。它帮助技术美术和 2D 动画师把用户合法持有的二次元类人角色图片、动作关键帧图片和动作描述，转化为一套可审阅、可修正、可追踪并能导入 Spine Editor 4.2.43 的生产资料。

产品不承诺“任意单图一键生成商业级完整动画”。它提供的是受约束的人机协作工作流：先确认角色母版和风格，再由规则与 AI 辅助反推动作所需素材，随后完成分层、补全、骨骼、网格、关键姿势、事件和命中帧编辑，最后输出 Rig IR、PNG/PSD、Spine JSON、atlas packing/input manifest 与 AI 动作提示词包。实际 `.atlas/.spine/.skel` 只能由用户本地合法 Spine Professional CLI 4.2.43 生成。

首版产品的内容基线固定为：

- 二次元类人角色；
- 横版侧视；
- 单一语义武器 `primary_weapon`；
- `idle`、`run`、`jump`、`fall`、`dash`、`attack_01`、`attack_02`、`attack_03`、`hit`、`death`；
- 本地单用户；
- Windows 11 x64 为 P0 一级支持平台，Windows 10 22H2 x64 为 P1 尽力兼容平台；
- PNG、JPEG、WebP 图片仅由用户导入；
- 禁止第三方公有云处理；
- 可选连接用户自托管的私有远程 GPU；
- V1 首个外部集成目标选定为仅 Spine Editor 4.2.43；Unity、Godot、Cocos和自研 Runtime 适配均为 V1 非目标。

### 1.1 范围记录 — `F2S-SCOPE-001`

该范围记录包含上述内容基线，以及项目管理、母版审批、素材规划、提示词包、分层与隐藏区修正、Rig/mesh/weight、十动作时间轴、关键姿势/命中帧审核、内部 Rig IR 预览、Rig IR/透明PNG/最小 PSD/Spine JSON/atlas-input manifest 导出和可选本地 Professional CLI 验证/转换。它明确排除内置生图、公有云、视频/3D 动作捕捉、任意视角/多武器、引擎运行时集成、自动判定玩法真值、捆绑 Spine Editor/Runtime，以及由内置writer生成`.atlas/.spine/.skel`。范围变更必须修改本记录、02需求、20路线图、21追踪和24 ADR；页面或实现单方面增加能力不构成范围批准。

## 2. 要解决的问题

### 2.1 生产问题

从一张合成角色图制作 Spine 角色时，生产人员必须反复完成部位识别、隐藏区域补全、图层命名、关节重叠、骨骼布局、网格权重、动作可行性验证和导出整理。重度动作还会暴露单一母版无法覆盖的透视、轮廓、遮挡与拓扑变化，导致“先拆完图再做动画”的线性流程频繁返工。

### 2.2 信息问题

动作描述、关键帧参考、实际素材、Rig 结构、时间轴事件和导出文件通常散落在不同软件与目录中，缺少统一 ID 和来源追踪。生产人员难以回答：

- 某张手型或躯干修正图服务于哪些动作；
- 某个动作为什么需要附件替换或短序列；
- 当前导出的命中帧是否经过策划确认；
- AI 补全的隐藏像素来自哪个模型、参数和遮罩；
- Spine 兼容失败发生在内部数据、导出器还是本地 CLI。

### 2.3 产品机会

产品以 AssetSpec 为中枢，将 `StyleSpec + CharacterSpec + WeaponSpec + MotionSpec + RuntimeSpec` 转化为最小必要素材集合，并通过临时 Rig 试跑形成“发现缺口—补素材—再验证”的闭环。AI 用于建议、补全和检查，人类负责身份、风格、结构和游戏性决策。

## 3. 产品愿景与设计原则

### 3.1 产品愿景

在不把用户角色图片发送给第三方云、不替代专业动画判断的前提下，使一个熟悉 2D 动画基础的用户能够在同一桌面工作区内完成从图片导入到 Spine Editor 4.2.43 可验证交付物的全过程。

### 3.2 产品目标

| 目标 ID | 目标 | 可观察结果 |
| --- | --- | --- |
| `F2S-GOAL-001` | 建立本地优先且数据可控的制作环境 | 未启用私有 GPU 时，全部核心编辑功能离线可用；持久项目真值留在用户目录，受控临时副本仅进入授权本机沙箱 |
| `F2S-GOAL-002` | 以动作需求反推素材，而非盲目生成所有变体 | 每个 AssetSpec 项都能追踪到至少一个 MotionSpec，未使用素材可识别 |
| `F2S-GOAL-003` | 建立可人工干预的 Production Assist 闭环 | 母版、分层、Rig、关键姿势、命中帧和导出均有显式审批状态，任何 AI 结果均可撤销或替换 |
| `F2S-GOAL-004` | 交付可在固定 Spine 版本验证的开放产物 | Rig IR、PNG/PSD、Spine 4.2.43 JSON、atlas packing/input manifest和提示词包可复现；`.atlas/.spine/.skel`仅由本地4.2.43 Professional CLI生成并带真实证据 |

### 3.3 设计原则

1. **母版是真值**：经批准母版的可见像素优先保留；AI 主要补全隐藏区域和生成明确批准的变体。
2. **动作先于素材**：只有 MotionSpec 或修复诊断提出需求时，才创建新增素材任务。
3. **人类拥有最终决定权**：身份、分层语义、骨架、关键姿势、攻击接触、命中帧和导出均不得静默批准。
4. **内部表示优先**：所有编辑写入版本化 Rig IR；Spine JSON 是派生产物，不作为唯一事实来源。
5. **隐私默认拒绝**：网络能力默认关闭；私有远程 GPU 必须显式配置、测试与信任。
6. **失败可恢复**：耗时任务、外部进程和保存失败不得破坏最后一个已确认版本。
7. **能力不夸大**：视觉可接受不等于 Rig 可生产，JSON 可解析不等于已通过 Spine Editor 4.2.43。

## 4. 用户与角色

产品按职责建模；普通制作、检查与人工门禁在首版可以由同一个本地用户承担，不实现应用账号、在线身份或通用权限系统。

| 人物 ID | 角色 | 核心任务 | 关键痛点 |
| --- | --- | --- | --- |
| `F2S-PER-001` | 2D 动画师/主操作者 | 导入、修层、建 Rig、制作动作、审批视觉结果 | 工具切换多、隐藏区域缺失、权重和动作返工 |
| `F2S-PER-002` | 技术美术 | 定义规范、检查拓扑、诊断渲染、验证 Spine 输出 | 数据不可追踪、格式兼容和性能风险出现太晚 |
| `F2S-PER-003` | 动作/游戏设计审核者 | 提供动作描述和关键帧图，审核节奏、接触和命中帧 | AI 容易生成看似流畅但玩法时序错误的动作 |

`F2S-REL-WAIVER-001` 的 governed P2 waiver 是独立、可选的本地多签能力，不是普通审批，也不允许同一用户点击三次代替职责分离。通用商业 build 的签名 application manifest 只钉住发布方的 enrollment 验证公钥，不预信任任意客户根；`OrganizationTrustEnrollmentV1` 必须由对应 publisher enrollment key 签名，并按 organizationId、productId/edition、组织离线根 keyId/SPKI/算法和有效期逐项绑定。该 capability pack 是 per-organization、per-edition 的，不能用裸公钥、通用 trust bundle 或另一 edition 的 enrollment 代替。

新项目的 `ActorRegistry` 只能从唯一 sequence=0 empty genesis 开始：enrollment、trust/revocation bundle 和 change-authorization 引用全为 null，bundle sequence=0 且 actors 为空。首次启用必须由单一 sequence=1 `enrollOrganization` transition 原子固定 enrollment 双 hash、两条 sequence=1 bundle roots、bootstrap authorization 与 initial actors；任何一步失败都保持 empty genesis。零配置、无 enrollment、签名无效、已过期、product/edition 或组织根不匹配时 governed waiver 必须禁用，用户只能修复 finding；普通制作、普通人工审批及无 binding 艺术 Warning 的 `userAcknowledgement` 不受影响。

同一 `ProjectId` 的 enrollment 一经 sequence=1 固定便不可替换或续写成另一组织、edition 或根。到期、组织迁移或 edition 变化时，唯一允许路径是克隆内容到新 `ProjectId`，重新 enrollment、登记 credential、grant 和审批；旧项目与历史证据只读保留，旧 grant、binding 和 approval 不迁移。能力就绪后仍要求 product、quality、release 三种角色分别由三个不同且当前有效的签名 `credentialId` 对同一 proposal 完成 attestation。产品只验证发布方签名、证书链、凭据唯一性、角色和签名，不声称离线软件可以证明三个 `credentialId` 必然属于三个自然人；人员分离由组织的凭据签发和流程控制负责。

V1 的离线 anti-rollback 只在当前项目根、manifest/registry 链、clock sidecar、应用状态与已导入 bundle 未被特权攻击者整体替换的前提下，检测项目内分叉、删撤销、旧 head 替换和局部复制。它不能发现组织已签发但操作者从未导入的更高 sequence bundle，也不能对抗本机管理员或磁盘快照把项目、sidecar、应用状态和已导入 bundle 全套回滚。若 Stable 发布的任一 required gate 使用 governed waiver，签名前必须强制执行“刷新组织 bundle”，并由当前 release 角色 credential 签署 `OrganizationBundleFreshnessDeclarationV1`，逐项绑定本 release candidate、enrollment、ActorRegistry、trust/revocation bundle 的 sequence/fingerprint/issuedAt 与当前 clock evidence。该声明只是“已按组织流程刷新并看到这些 head”的流程声明，不是 global latest 证明；无 waiver 的 Stable 发布不要求该步骤，manifest 中对应声明双 hash 必须为 null。在线 transparency、组织 HSM 单调计数器、TPM 或远端锚属于未来 V2，不得暗中宣传为 V1 能力。

产品不为纯终端玩家、公共图片生成用户或云端协作团队设计。

### 5.1 首版必须具备

| 项目 | 新建、打开、另存、最近项目、自动保存、恢复、只读兼容提示、项目打包与诊断包 |
| 输入 | P0 支持 PNG、JPEG、WebP 角色图和关键姿势图片、透明度检查、色彩与尺寸规范化；V1 不把 PSD 作为输入格式；不提供生图调用 |
| 内容定义 | CharacterSpec、StyleSpec、WeaponSpec、RuntimeSpec、十个 MotionSpec 和 AssetSpec |
| 提示词 | 依据结构化规格生成、版本化、复制和导出 AI 动作提示词包；用户在外部生成后重新导入图片 |
| 分层 | 自动建议分区、蒙版编辑、层级与命名编辑、隐藏区域补全、透明边缘修复、重组对比 |
| 素材规划 | 识别 bone/mesh/attachment/sequence/new-view/vfx 的表现方式，汇总最小素材清单和缺口 |

| 游戏性审核 | 关键姿势、攻击接触、命中帧、恢复阶段和事件标记的人工审批；不自动决定玩法数值 |
| 预览与 QA | 基于 Rig IR 的内部渲染、背景切换、轮廓/网格/权重/过度拉伸/透明洞/脚滑等诊断 |
| 导出 | 内置生成Rig IR、透明PNG、最小可重建分层PSD、Spine 4.2.43 JSON、`atlas-input-manifest.json`、提示词包、清单和验证报告；高级PSD特性可以在报告中明确降级；内置writer不得生成`.atlas/.spine/.skel` |
| Spine 联调 | 检测并调用用户本地合法安装的 Spine Professional CLI 4.2.43 进行可选验证/转换；仅该CLI可生成`.atlas/.spine/.skel`，未安装时清晰降级且只保留内置开放产物 |
| 计算 | 本地 CPU/GPU 配置和可选私有远程 GPU Provider；任务可取消、可恢复、有资源预算 |

### 5.2 首版明确不做

- 不调用 OpenAI、Adobe、Replicate、Hugging Face Inference Endpoint 或其他第三方公有云处理素材；
- 不集成文本生图或图片生图 API，只产出供外部工具使用的提示词；
- 不承诺从一张图自动恢复真实背面、被遮挡物体或艺术家的原始分层意图；
- 不自动生成未经人工审批的攻击命中帧、取消窗口、伤害、位移或状态机逻辑；
- 不制作 Unity、Godot、Cocos 或自研引擎插件；
- 不把 PSD 作为 V1 输入格式，也不承诺保留高级 PSD adjustment、智能对象、图层样式或未栅格化效果；
- 不以内置writer生成或改写`.atlas/.spine/.skel`；这些文件只能来自用户本地合法Spine Professional CLI 4.2.43，并必须携带本轮CLI证据；
- 不捆绑 Spine Runtime、Spine Editor、第三方模型权重或许可不明的素材；
- 不实现账号、云同步、多人实时协作、素材市场、在线遥测和远程技术支持上传；可选 governed waiver 只导入 publisher-signed `OrganizationTrustEnrollmentV1`、组织签名 bundle/authorization 和本地签名凭据描述，不构成账号或协作系统，也不允许为既有 `ProjectId` 替换 enrollment；
- 不承诺多方向、换装组合、多武器、非类人、写实厚涂或全帧手绘的自动生产能力；
- 不用简单形变替代必须新增视角或极端关键姿势的素材。

### 5.3 后续扩展边界

架构可以为下列能力保留接口，但首版不得以未实现接口宣传能力：

- 多方向、多 Rig 和皮肤组合；
- 多武器 WeaponSpec；
- 引擎适配器；
- 受许可约束的本地模型插件；
- 动作视频或 3D 代理输入；
- 团队审阅和受控的私有部署。

## 6. 内容与技术边界

### 6.1 支持内容

首版适合轮廓清晰、肢体相对可分、固定横版侧视、光照稳定的二次元类人角色。长发、裙摆、披风和大面积透明材质允许导入，但会触发更高风险提示和更严格的人工检查。

### 6.2 表现方式边界

系统必须使用策略选择，而不是把所有动作强制变形成同一套基础图：

- 小幅平面内运动优先骨骼和网格；
- 表情、手型、武器状态优先附件替换；
- 关节暴露和中等轮廓变化使用修正附件；
- 强透视、转面、轮廓或拓扑突变使用关键姿势附件或短序列；
- VFX 与基础角色结构分离。

如果所需表现超出首版侧视资产能力，系统应生成明确的“需要新增素材”诊断，而不是产生静默低质量结果。

### 6.3 Spine 边界

目标版本固定为 Spine Editor 4.2.43。内置Spine边界只产生JSON、透明attachment PNG和atlas packing/input manifest；它们必须以该版本实测。`.atlas/.spine/.skel`及实际atlas packing结果只允许由用户本地合法Professional CLI生成并标明provenance。内部预览不依赖 Spine Runtime；外部 CLI 仅调用用户本地合法安装，不复制、不破解、不替用户管理授权。

## 7. 数据、隐私和许可承诺

### 7.1 数据所有权

用户导入的图片、项目文件、补全素材、动作描述和每次导出的canonical snapshot持久真值全部保存在用户选择的项目目录。用户另选项目外目录时只执行可重建的publish copy；外部副本失败、丢失或被修改不改变项目内真值。后台 Job 可把完成本次任务所需的最小 D2 输入材料化到 `%LOCALAPPDATA%` 下由 `F2S-DOC-STORE-001` 定义的专用沙箱；该副本必须使用专用 ACL、绑定 Job/源哈希、受保留期和崩溃对账控制，并在提升或终止后安全清理/隔离。应用数据库、沙箱与外部publish目录都不得成为源资产或导出snapshot的唯一副本，也不得自动上传。

### 7.2 网络边界

- 默认安装完成后不要求登录、不发送遥测、不访问公有云；
- 私有远程 GPU 默认关闭；
- 启用时必须显示端点、协议、证书状态、将发送的数据种类和用户批准记录；
- 仅允许用户控制的局域网、VPN 或企业私有端点；
- 远程失败必须回退为本地待处理或允许取消，不得转发至公共 Provider。

### 7.3 许可证边界

发布依赖只接受`F2S-LIC-POLICY-001`中完成审计的allowlist；精确SPDX列表、外部系统分类和例外流程只由14号文档维护。代码许可证、模型代码、模型权重、训练数据、字体、图标和样例素材分别审计。开发期实验依赖不得自动进入发布包。

## 9. 产品状态模型

项目总体状态由审批门组合推导，不允许用户或实现直接写入虚假的“已完成”：

工作流阶段由 `F2S-DOC-UX-001` 统一定义：

`draft → master_approved → content_approved → layers_approved → rig_approved → motion_review → gameplay_approved → export_ready → exported`

规则如下：

- 任一上游真值变化会使相关下游新鲜度变为 `stale`，但不会伪造或回退工作流阶段，旧版本继续保留供对比；
- `master_approved`、`layers_approved`、`rig_approved`、关键姿势和命中帧批准必须由显式命令产生；
- `blocked` 属于阻塞维度而不是工作流阶段；它说明当前阶段不能前进，但不覆盖对象的生命周期和新鲜度；
- Spine 未安装或 CLI 未运行时，导出验证维度只能是 `exported_unverified`；`spine_verified` 不是工作流阶段；
- 状态迁移、撤销和失效原因必须进入审计日志。

状态细节由 `F2S-DOC-UX-001` 定义，数据实现由后续存储与逻辑设计负责。
