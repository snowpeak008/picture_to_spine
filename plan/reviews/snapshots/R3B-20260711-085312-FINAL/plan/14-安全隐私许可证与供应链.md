# 14. 安全、隐私、许可证与软件供应链设计

## 14.1 目标与强制原则

本项目是 Windows 闭源商业软件，处理可能尚未公开的角色原画、动作关键帧、提示词和游戏设计数据。安全模型采用 local-first、最小权限、默认拒绝和可审计供应链。

以下要求属于发布硬门禁：

1. 用户素材默认且正常路径下不离开本机。
2. 不接入供应商云端 AI，不上传第三方云存储，不包含隐式遥测。
3. “远程 GPU”仅指用户显式配置、用户自行控制的 HTTPS 推理节点；不是厂商 SaaS。
4. Rust 核心是项目写入的唯一授权者；React、PixiJS 和 Python Worker 不直接修改 `.f2sproj`。
5. 发布依赖和模型权重必须有可审计、允许闭源商业使用的许可证。
6. 官方 Spine Runtime 是自定义许可证，不得进入发布包。
7. Spine Editor/Professional CLI 只能作为用户自行安装的外部工具使用。
8. 所有发行物必须有 SBOM、许可证清单、哈希和签名验证记录。

## 14.2 信任边界

```text
不可信输入
  ├── 用户图片/PSD/PNG
  ├── 动作描述与提示词
  ├── 动作关键帧图片
  ├── 导入的项目/模型包
  ├── Python Worker NDJSON
  └── 自托管远程 GPU 响应
           │
           ▼
Rust Validation Boundary
  ├── 文件类型与尺寸校验
  ├── Schema/范围/引用校验
  ├── 路径与权限校验
  ├── 内容哈希和许可证校验
  └── 工作流状态校验
           │
           ▼
可信项目写入器 → .f2sproj/CAS/redb 可重建索引
```

所有跨边界数据均按不可信数据处理。即使来自本机 Worker，也不能绕过 Schema、尺寸、引用和状态校验。

## 14.3 数据分类与处理

### F2S-SEC-DATA-001：数据等级

| 等级 | 示例 | 默认处理 |
|---|---|---|
| D0 公开 | 应用版本、公开许可证文本 | 可进入日志和 SBOM |
| D1 内部 | 非敏感设置、性能指标 | 仅本机；日志限量 |
| D2 机密 | 原图、分层图、动作关键帧、Rig、提示词 | 本地项目；日志禁止正文 |
| D3 秘密 | 远程节点令牌、代码/组织/actor签名私钥 | Windows 凭据库/HSM/组织自有证书设施；禁止项目存储、导出或日志 |

文件名、绝对路径、用户名和项目名可能泄露业务信息，默认按 D2 处理。诊断包必须先预览和脱敏，再由用户主动导出。

### F2S-SEC-DATA-002：保留与删除

- 原始导入文件进入内容寻址存储后保持不可变。
- 临时文件按 operation ID 隔离，成功后删除；崩溃遗留由 `F2S-DOC-RECOVERY-001` 安全回收。
- Worker 临时输入使用受限目录和最短保留期。
- 自托管远程节点必须声明不持久化请求；客户端不相信声明，而是在 UI 中持续显示离机风险。
- 卸载应用不删除 `.f2sproj`、用户素材或导出物。
- “删除项目”必须由用户在文件系统或专门 UI 二次确认，不提供静默清空。

### F2S-SEC-DATA-003：静态数据与物理介质残余风险

应用层不承诺“物理不可恢复删除”。SSD 的 wear leveling、TRIM、控制器保留区、卷影副本、备份和同步软件都可能留下应用无法可靠擦除的副本。删除功能只承诺撤销逻辑引用并按恢复/垃圾回收策略移除可见文件。

- 对 D2/D3 项目建议使用已启用 BitLocker 的 NTFS 卷，并由组织管理恢复密钥。
- 临时目录、Worker 沙箱、日志和导出目录应位于同一受保护卷；不能只加密项目文件夹而把明文 staging 放在未加密盘。
- SSD 退役或转移所有权时采用组织级介质销毁/厂商 Secure Erase 流程，不能把应用“清理缓存”当作物理擦除。
- BitLocker 降低设备遗失风险，但不防御已登录会话、恶意管理员、内存抓取或已导出的明文副本。
- UI 和用户文档必须明确上述剩余风险，不使用“安全擦除”“永久不可恢复”等无法证明的表述。

## 14.4 进程与权限隔离

### F2S-SEC-PROC-001：Rust 核心

Rust/Tauri 核心独占以下权限：

- 项目 manifest 和 revision 写入；
- CAS artifact 提升和引用更新；
- redb 可重建索引事务；
- 导出 staging 与最终提交；
- 外部 CLI 启动；
- 本机设置和凭据句柄读取。

所有 Tauri command 使用显式 DTO，不暴露任意文件读写、任意进程执行或任意 URL 请求命令。前端只能获得业务所需的最小 capability。

### F2S-SEC-PROC-002：Python Worker

Python 3.12 Worker 是可选、独立发布单元。`F2S-NFR-SEC-005` 将商业 D2 Worker 的 OS 隔离定义为 P0 需求；其唯一实现配置是 `F2S-SEC-SANDBOX-001/windows-appcontainer-v1`。其他组合只能作为开发 Spike，不得被标记为等价发布配置。

`windows-appcontainer-v1` 冻结如下：

| 控制面 | 唯一发布要求 |
|---|---|
| 身份 | 专用 AppContainer profile SID；由受限 primary token 创建 lowbox/AppContainer token，移除管理员 SID 和不必要 privilege |
| 启动 | Worker 先以 suspended 状态创建，配置 `PROC_THREAD_ATTRIBUTE_SECURITY_CAPABILITIES`、进程缓解策略和句柄 allowlist，加入 Job Object 后才能 resume |
| 文件 | `%LOCALAPPDATA%/<Publisher>/FlashToSpine/job-sandboxes/<job-id>` 为唯一工作根；`in/`只读、`out/tmp`最小写权限；签名Runtime Pack manifest中确切worker/runtime/DLL只授予profile SID最小RX且不可写，内置模型只读；用户模型验证后只读材料化到本job，绝不开放原目录；项目根、邻接未列文件、用户文档、凭据库和其他job明确拒绝 |
| 进程 | Job Object 启用 `KILL_ON_JOB_CLOSE`、active-process/memory/CPU/time 限额、UI restriction 和 no-breakaway；所有子进程必须留在同一 Job |
| 网络 | AppContainer token不授予 `internetClient`、`privateNetworkClientServer`等任何网络 capability；同时按 profile SID/Worker identity 配置 OS outbound deny，禁止DNS、IPv4/IPv6、TCP/UDP、loopback、代理和继承句柄绕行 |
| 代码身份 | Runtime、Worker、模型 manifest 与原生 DLL 均验证签名/哈希；PATH、PYTHONPATH、site/user-site、启动目录和可加载 DLL 路径由 Core 固定 |
| 证据 | `F2S-TST-RGPU-005` 先证明allowlisted签名runtime/模型可正常加载，再证明邻接文件、原模型目录、文件逃逸、网络、代理、子进程、Job breakaway和句柄继承全部被OS拒绝；证据绑定Windows build、ACL dump、策略hash、二进制hash和普通用户身份 |

商业发布不接受“restricted token 单独使用”“ACL + Job Object 但无 AppContainer”“仅防火墙规则”“环境变量声明”作为替代 SandboxProfile。若平台 Spike 证明本配置无法在 P0 Windows 11 上可靠运行，必须通过新的安全 ADR 和威胁复审修改本 canonical profile；在此之前只能关闭 LocalPythonProvider。

该 Worker 还必须满足：

- stdin/stdout 使用逐行 NDJSON；stderr 仅作受限日志；
- 每条消息包含 `protocolVersion`、`requestId`、`method` 和尺寸受限 payload；
- Worker 只读取由 Rust 放入沙箱输入目录的只读副本；
- Worker 无 `.f2sproj` 写权限、无 redb 访问权；
- 结果先进入 Rust 校验器，不能直接成为已批准资产；
- 超时、输出上限、内存/显存预算和取消必须可配置；
- Worker 及其子进程不得访问项目根、凭据库、用户文档目录或任意网络；
- 任一 SandboxProfile 字段未落实、策略哈希不符或恶意探针成功，都阻断 Worker Pack 和商业发布，不能用文档承诺、依赖审计或 `policy_only` 替代 OS 级阻断；
- `policy_only` 仅允许处理 D0 合成夹具，不能接收 D2 输入，也不能在商业 UI 中显示 LocalPythonProvider 可用。

NDJSON 协议遇到未知字段可按版本策略忽略，遇到未知方法、超限消息、重复 requestId 或 Schema 失败必须终止对应请求。

### F2S-SEC-PROC-003：Spine CLI

仅允许用户显式选择本机 `Spine.com`。调用规则遵循 `F2S-DOC-EXPORT-001`：

- 参数数组，不经 `cmd.exe`/PowerShell/shell；
- probe 使用纯 `--version`；PinnedValidation 进程按13号要求显式带固定 `--update 4.2.43` 但没有网络 grant，以同时消除版本TOCTOU和隐式外连；只有用户明确许可准备/获取该patch时，独立 preparation操作才可联网调用 `--update 4.2.43 --version`；
- 只允许审查过的参数集合；
- 不读取、复制、上传或记录激活码；
- 不捆绑、不自动下载 Spine；
- 不把项目文本或 AI 输出拼入命令行。

## 14.5 文件与项目安全

### F2S-SEC-FS-001：路径规则

- 使用 Rust Path API；不通过字符串拼接路径。
- 所有写入目标先 canonicalize 父目录，再验证位于授权根目录。
- 拒绝逃逸的 `..`、NTFS ADS、设备路径、危险 UNC、junction/symlink 重定向。
- 导入压缩包时防止 Zip Slip、符号链接逃逸、压缩炸弹和超大文件数。
- 图片解码前限制文件大小、像素总数、帧数、层数和 ICC profile 大小。
- 不信任扩展名，使用 magic bytes 与严格解码器确认格式。
- 最终文件通过同卷临时文件、flush、校验和原子 rename 提交。

### F2S-SEC-FS-002：内容寻址存储

Artifact key 使用完整 SHA-256，不用文件名或短哈希作为唯一身份。写入流程：

```text
stream → size limit → decode/validate → SHA-256 → temp file
       → hash recheck → CAS path → manifest transaction
```

CAS 对象不可原地修改。编辑产生新对象和新引用。垃圾回收必须先计算所有项目 revision、快照和导出引用，默认采用保守保留。

## 14.6 网络与远程 GPU

### F2S-SEC-NET-001：默认无外连

应用首次运行和普通本地工作流不得外连。无遥测、无崩溃自动上传、无远程字体、无 CDN 资源、无在线模型下载。更新检查默认由用户触发；其安全规则见 `F2S-DOC-RELEASE-001`。

### F2S-SEC-NET-002：自托管远程节点

远程 GPU 是显式可选 Provider，仅允许用户控制的 HTTPS endpoint：

- 每个项目单独启用，首次发送前展示将发送的数据摘要；
- 默认域名/IP allowlist；禁止自动跟随跨域重定向；
- TLS 1.2+，生产推荐 mTLS；证书错误不得忽略；
- 令牌存入 Windows Credential Manager，不写项目、日志或命令行；
- 请求带 idempotency key、大小上限、超时和取消；
- 响应做 MIME、尺寸、Schema 和哈希校验；
- endpoint 配置区分本机设置与可分享项目，分享项目不得携带凭据；
- 禁止把用户素材转发给节点声明之外的第三方；
- 用户选择“不允许离机”时，Provider 层从能力上禁用远程任务，不仅隐藏按钮。

若“不得上传”被解释为任何网络节点均不可接收素材，则远程 GPU 功能必须整体关闭；不得通过术语把第三方云包装成“远程 GPU”。

## 14.7 许可证政策

### F2S-LIC-POLICY-001：默认许可证白名单

本节是项目许可证政策的 canonical 冻结点。其他计划和任务只能引用 `F2S-LIC-POLICY-001`，不得维护第二份不同 allowlist。发布依赖的代码、二进制、模型、字体和随附资产分别审计；默认可审计宽松 allowlist 为：

```text
MIT
Apache-2.0
BSD-2-Clause
BSD-3-Clause
ISC
Zlib
0BSD
PSF-2.0
CC0-1.0
LicenseRef-Public-Domain
```

`LicenseRef-Public-Domain` 只有在存在可归档的权利人放弃/公共领域证明、来源和哈希时才允许；“网上说免费”不构成证明。多许可证依赖必须明确选择 allowlist 中的许可分支并保留证据。

系统运行时和用户商业工具单列为外部环境，不视为随包依赖：

- Windows、系统 Evergreen WebView2、GPU 驱动等由操作系统/用户环境提供，记录在 Environment BOM 和兼容矩阵；
- 用户自行安装并许可的 Spine Professional/Enterprise 属于外部商业工具；
- 外部工具不得被复制到安装包、更新包、缓存或 Worker Pack；
- “系统已有”不能掩盖应用实际随包复制的 DLL/二进制，实际分发即进入 SBOM 和许可门禁。

任何不在上述 allowlist、也不属于已登记外部环境的组件，必须先建立 ADR、取得法务书面批准、明确分发方式和 notices，之后才能更新本 canonical 政策；单个任务或 release waiver 不能临时放行。

默认禁止：

- GPL、AGPL、SSPL、LGPL、MPL、EPL、CDDL 等强或弱 copyleft；
- CC-BY-NC、Research Only、Non-Commercial；
- 来源不明、无 LICENSE、仅仓库描述许可；
- 禁止再分发、禁止商用或训练数据权属不明的模型权重；
- 官方 Spine Runtime 的自定义 `LicenseRef-LICENSE`。

### F2S-LIC-MODEL-001：模型与数据集

模型代码许可、模型权重许可、训练数据声明和输出使用条款是四个独立审计项。代码为 Apache-2.0 不代表权重或输出可商用。

每个 ModelPack 必须携带：

```json
{
  "modelId": "...",
  "version": "...",
  "sha256": "...",
  "codeLicense": "Apache-2.0",
  "weightLicense": "...",
  "commercialUse": true,
  "redistribution": true,
  "sourceUrl": "...",
  "reviewId": "F2S-LIC-REVIEW-..."
}
```

任一字段未知则不得进入商业发行包。用户自行导入的模型包必须显示“未审计”，与官方支持包隔离，且不能上传到远程节点或进入自动发布产物。

### F2S-SEC-MODEL-001：非代码权重与安全加载

“ModelPack 不包含脚本”还不足以阻止反序列化执行。商业支持的模型只能采用经 M00 审定的非代码格式：

- `safetensors`：严格校验 header、dtype、shape、offset、文件大小和 SHA-256，不允许重叠/越界 tensor；
- `ONNX`：只允许锁定 opset 和审核过的内置 operator；external data 必须在包内、内容寻址且路径不逃逸；禁止自定义 operator DLL、Python op、远程 URI和运行时动态库搜索；
- manifest 必须声明 `format`、opset/dtype、所有分片哈希、预期内存上限和加载器版本。

正式路径禁止 `.pt`、`.pth`、`.ckpt`、Pickle、joblib、任意 Python module、宏及会调用 `torch.load`/等价通用对象反序列化的格式。改扩展名不能绕过 magic/schema 检查。用户自带模型即使为允许格式也保持 `UNREVIEWED`，只能在用户确认后由 `windows-appcontainer-v1` 处理，不能成为随包默认模型或远程上传源。发现需要新格式时先做 parser fuzz、许可和威胁 ADR，不能由模型 manifest 自行扩展 allowlist。

### F2S-LIC-SPINE-001：Spine 边界

Spine Editor 是用户自行许可和安装的外部商业软件，不属于发行依赖。Production Assist 需要 Mesh、Weights、IK 等能力，用户应持有适用的 Spine Professional 或 Enterprise 许可；Essential 无法保存或导出含 Professional 特性的项目。

官方 Runtime 使用专有条款，不在 `F2S-LIC-POLICY-001` allowlist，也不是单纯系统运行时，因此：

- 不依赖、复制、修改或捆绑 `spine-ts`、`spine-c`、`spine-unity` 等 Runtime；
- 内部预览不得通过间接依赖重新引入 Runtime；
- `THIRD_PARTY_NOTICES` 中不应出现 Runtime，若出现则阻断构建；
- 产品只生成 Rig IR、PSD、PNG、兼容 JSON，并可选择调用用户本地 CLI；
- 商业发布前把完整产品用例发给 `contact@esotericsoftware.com` 获取书面许可边界确认并归档。

官方依据：

- [Spine Editor License Agreement](https://esotericsoftware.com/spine-editor-license)
- [Spine Runtimes License Agreement](https://esotericsoftware.com/spine-runtimes-license)
- [Spine Runtimes 4.2 Repository](https://github.com/EsotericSoftware/spine-runtimes/tree/4.2)
- [Spine Purchase and Feature Matrix](https://us.esotericsoftware.com/spine-purchase)

### F2S-LIC-CLOSED-001：闭源边界

许可证扫描必须覆盖：Rust crates、npm packages、Python wheels、原生 DLL、字体、图标、示例素材、模型代码、权重、安装器脚本和更新器。开发依赖与测试依赖也进入内部 SBOM；实际分发清单另行标记。

任何 copyleft、NC、Research-only 或未知许可证依赖均禁止进入发布物，也不得通过“只在构建机使用”或 release waiver 自动豁免。若确有仅构建期工具需求，仍须通过 ADR 与法务确认生成物和分发边界；批准前构建门禁保持失败。

## 14.8 SBOM、来源与构建供应链

### F2S-LIC-SBOM-001：双格式清单

每个正式版本至少生成：

- CycloneDX JSON SBOM；
- SPDX 2.3 JSON 或 tag-value 清单；
- `THIRD_PARTY_NOTICES.txt`；
- `MODEL_BOM.json`；
- `checksums.sha256`；
- 构建来源与工具链版本清单。

每个组件记录名称、精确版本、下载源、SHA-256、许可证、是否分发、依赖关系和审计状态。

### F2S-LIC-SUPPLY-001：依赖固定

- Rust 使用提交入库的 `Cargo.lock` 和 `--locked`。
- npm 使用 lockfile 与 frozen install；禁止宽松浮动版本进入 release。
- Python 使用带 hash 的完全锁定 requirements/lock 文件。
- Git 依赖固定 commit SHA，不固定分支或 tag 别名。
- 构建过程中禁止从未登记 URL 下载二进制或模型。
- 依赖升级由单独 PR 完成，附许可证差异、漏洞差异和回归结果。

### F2S-LIC-PROVENANCE-001：来源证明

构建流水线保存：

- Git commit 和 dirty-state；
- Rust、Node、Python、Tauri、NSIS 版本；
- Windows SDK 与构建镜像标识；
- lockfile 哈希；
- SBOM 哈希；
- 代码签名证书标识和时间戳结果；
- 测试报告与批准人。

发行物签名私钥不得存放于仓库、开发者磁盘明文或 CI 普通变量。优先使用 HSM/受保护证书存储，签名作业与普通构建作业隔离。

## 14.9 应用安全控制

### F2S-SEC-APP-001：前端与 IPC

- Tauri CSP 默认拒绝，仅开放打包资源；禁止远程脚本。
- 不启用任意 shell、任意文件系统或任意 HTTP scope。
- 所有 IPC command 校验窗口身份、project ID、revision 和参数长度。
- 富文本、文件名、提示词按纯文本显示；禁止未经清洗的 HTML。
- 图片元数据不进入 DOM HTML，不执行 SVG 脚本。
- WebView 导航限制在应用 origin，外部链接交由系统浏览器并二次确认。

### F2S-SEC-APP-002：日志

结构化日志只记录 operation ID、错误码、耗时、组件版本和尺寸等必要数据。禁止记录：

- 原图或缩略图 base64；
- 完整提示词和动作文本；
- 远程令牌、Authorization header；
- Spine 激活信息；
- 默认完整绝对路径；
- D2/D3 数据正文。

日志轮转、总量和保留天数可配置，默认保守。诊断导出需用户预览。

### F2S-SEC-APP-003：更新和插件

MVP 不支持第三方插件执行代码。ModelPack 是声明式资产包，不可包含 `.exe`、`.dll`、脚本、宏或安装钩子。更新包必须签名并由 `F2S-DOC-RELEASE-001` 的更新器验证，签名失败不可绕过。

### F2S-SEC-ID-001：离线组织多签身份

governed P2 waiver是可选能力，不是账号系统，也不能靠本地创建三个UUID启用。通用build只在签名application manifest中钉住发布方organization-enrollment验证key；客户组织root必须由发布方签名的`OrganizationTrustEnrollmentV1` capability pack按organizationId/productId/edition/expiry绑定。项目只接受该root签名的ActorTrust/Revocation bundles、registry change authorization和actor certificate chain；不允许用户配置任意裸root。无/过期/错edition enrollment时核心编辑可用，但waiver和签名发布fail closed。

Actor私钥始终留在组织证书设施/Windows受保护key provider中，应用CAS只保存DER证书链、公钥/SPKI hash、签名与双hash引用。三个product/quality/release attestation必须由不同actorId与不同credentialId签同一exact waiverId/proposalId/proposalHash；应用不声称仅凭离线密码学证明三个自然人，职责分离由组织签发流程承担。ActorAttestationPolicy固定bytes/hash必须在启动、proposal、grant、binding、active replay和signer各阶段自校验。

### F2S-SEC-TRUST-001：撤销与反回滚

ActorRegistry唯一未配置genesis为sequence=0、enrollment/bundle/authorization roots全null；首次`enrollOrganization`以一个sequence=1原子transition写入enrollment、两bundle、bootstrap authorization与initial actors，部分成功不得提交。同一ProjectId的enrollment不可替换；需要换root时克隆内容到新ProjectId并重新enroll/reapprove，旧grant不得迁移。

sequence>0时ActorRegistry以artifact/domain双hash锚定current enrollment及trust/revocation bundle heads，再由WaiverRegistry/RevisionManifest间接扎根。revocation subjects只能累加；trust anchor只能active→disabled/revoked，不能删除、换key或复活；actor/credential/role也不可复活。bundle推进必须重验并产生新evidence，只有相关credential/SPKI/serial/anchor失效才使binding inactive，无关entry推进不得误失效。任何domain当CAS地址、合法双hash但删撤销、旧bundle替换current或历史validation复用都按攻击处理。

签名实现必须同时自校验18号固定`SignatureAlgorithmRegistryV1`双hash：ECDSA仅P-256/SHA-256、64-byte IEEE-P1363、low-S；RSA-PSS仅SHA-256/MGF1-SHA256/saltLen=32/trailer=1且modulus≥2048；base64url无padding并严格解码。DER/high-S/错长度、宽松base64或provider默认PSS参数都拒绝，不能让Windows/Rust provider各自猜测编码。

本地双链的反回滚保证有明确边界：它能发现当前项目/进程内分叉、删撤销和旧head替换，但纯离线V1无法知道组织已签发却从未导入的更高sequence bundle，也不对抗本机管理员或磁盘快照把项目、sidecar、应用状态和bundle整体回滚。有governed waiver的Stable签名前必须执行“刷新组织bundle”，由current release credential签`OrganizationBundleFreshnessDeclarationV1`并记录sequence/fingerprint/issuedAt、ClockEvidence及限制确认；声明须与最终manifest同signingAttemptId，且最终签名前年龄不超过900秒，作业重启/换attempt/超时重签。该声明是流程证据，不是全局latest证明。强全局freshness需V2在线transparency、HSM单调计数器或TPM/远端锚。

## 14.12 事件响应

发现漏洞或供应链事件时：

1. 记录稳定事件 ID、影响版本和证据哈希。
2. 停止受影响版本发布与更新分发。
3. 隔离构建凭据，必要时吊销签名或远程节点令牌。
4. 复现并确认是否涉及 D2/D3 数据外泄。
5. 发布修复版本和用户可执行的缓解措施。
6. 更新 SBOM、风险登记、回归测试和事后分析。
7. 不通过自动遥测收集用户证据；由用户主动提供脱敏诊断包。

商业发布前必须指定 `SecurityOwner`、`ReleaseOwner`、Legal联络和用户可见安全报告邮箱；未配置时安全响应能力保持 `UNVERIFIED` 并阻断 stable。事件统一保存：

### F2S-SEC-SIGN-001：签名密钥和更新根失陷

代码签名证书、内容签名 key 或更新 feed 管理权疑似失陷时：

1. 立即停止构建、签名和 stable/preview feed，冻结最后可信 manifest hash。
2. 在 `KeyIncidentRecord` 中记录 `keyId/certThumbprint`、用途、疑似失陷时间、`lastTrustedBuild`、受影响版本、吊销请求、替代 key 和批准人。
3. 向 CA/时间戳/托管签名服务执行适用的证书吊销；内容签名 key 加入客户端和服务端 denylist。不能把“有旧时间戳”自动当成未受影响。
4. 由与日常发布 key 分离的离线 recovery root 签署 key-transition/revocation manifest；客户端只接受单调递增 `securityEpoch`，拒绝被撤销 key 和低于 `minimumSafeVersion` 的更新。
5. 重新构建、签名、验证最终渠道产物并发布用户公告；无法安全建立新信任链时停止自动更新，改走官网人工核验流程。

`KeyIncidentRecord`、revocation manifest、CA回执和最终下载验证都进入不可变发布证据；普通管理员不能删除历史失陷记录。

publisher organization-enrollment key疑似失陷时同样冻结capability pack签发，把受影响enrollmentId/root加入发布方签名撤销清单并使相关项目waiver验证inactive；该事件属于update-root硬门，不能用release waiver处理。客户organization offline root失陷时，旧ProjectId不允许热替换enrollment；组织取得新的发布方签名pack后克隆到新ProjectId、重新建立ActorRegistry并重做全部人工审批。艺术内容可以迁移，旧grant/attestation/active approval状态不得迁移。
