# 测试、质量、性能与可观测性

## 1. 目的与质量声明

本文件定义 FlashToSpine Production Assist 的验证策略和证据格式。测试的目标不是证明“AI 总是正确”，而是保证：确定性软件行为可重复、AI 候选结果可追溯、人工审批不会被绕过、项目资产不会因故障丢失、Spine 4.2.43 输出能够被明确标注为“已验证”或“未验证”。

质量结论分为三类：

- `VERIFIED`：具备自动化或人工证据，且证据与实现版本、fixture 哈希绑定。
- `PARTIAL`：仅部分环境、样例或路径通过，范围必须写明。
- `UNVERIFIED`：尚未获得执行证据；文档完整不能将其提升为 VERIFIED。

## 4. 合法测试素材策略

用户只能提供动作关键帧图片和动作描述，不能提供完整基准集，因此工程必须自带可合法再分发的合成 fixture：

1. 用项目自有矢量形状生成无人物版权风险的基础类人角色；
2. 生成对称/不对称、遮挡、长发、披风、透明边缘和极端关节测试图；
3. 为每张图保存生成脚本、参数、许可声明和 SHA-256；
4. 用户提供的图片只进入本地验收目录，不提交仓库、不写诊断包、不用于训练；
5. 开源样例只有在图像本身许可与代码许可都被核实后才能进入 fixture。

测试集合分为：`happy`、`hard`、`out-of-scope`、`corrupt`、`security`。越界输入正确拒绝同样属于通过。

## 5. 视觉质量检查

### 5.1 自动指标

- setup pose 重新合成的意外透明差异面积，目标不超过角色面积的 0.5%；
- 在目标预览尺寸下，主轮廓不允许出现大于 `2×2 px` 的意外透明洞；
- 网格不得存在 NaN、Infinity、退化三角形或非法索引；
- 95% 可见网格区域局部缩放预警区间暂定 `0.6～1.7`，各向异性拉伸预警暂定 `<2.0`；
- 角色比例锚点偏移预警暂定小于角色高度的 3%；
- 动画循环首尾的接触点、root 和可见附件必须满足 clip-specific tolerance。

`F2S-QA-PROFILE-001` 是上述视觉阈值的唯一 canonical profile，并补充冻结以下默认投影：

- 关键姿势相对批准母版的 bbox 高度偏差 `≤3%`，地面接触高度偏差 `≤1 px`；
- 非特效区域的中位 `DeltaE2000 ≤5`、中位线宽相对差 `≤15%` 为 warning 阈值，不自动判定内容不合格；
- `idle/run` 默认循环首尾 root 位移 `≤0.5 px`、接触脚位移 `≤1 px`、相关骨旋转差 `≤0.5°`；具体 clip profile 只能收紧，放宽必须新增有证据的 profile revision 和人工批准。

其他文档只能引用该 profile，不得复制后独立修改数值。阈值必须用人工批准样例校准；自动指标只负责报警和排序，不能替代动画师对力度、轮廓和人体结构的判断。

### 6.1 Benchmark protocol

性能结论必须由 `benchmark-profile.json` 固定，至少包含：Windows build/补丁、机器型号、CPU、物理/可用内存、GPU/VRAM、驱动、WebView2、DPI/分辨率、供电模式、温控状态、应用/fixture/profile hash、后台进程策略和计时器来源。不同硬件档、驱动或冷/热缓存数据不得混算一个 P95。

统一执行规则：

1. 从签名或可追溯 release build 启动；性能 fixture 使用 `F2S-FIX-PROJ-001` 的冻结 hash，失败样本不得删除。
2. 冷启动至少30次，每次重新创建约定的冷缓存条件；热启动和交互延迟至少100次。先做5次不计分 warm-up，并单独保存原始样本。
3. P95 使用 nearest-rank；同时输出 median、P95、max、样本数和 bootstrap 95%置信区间。不得以“异常值”为由删样；环境干扰只能使整轮作废并保留作废原因。
4. FPS 在10秒warm-up后连续采样至少120秒，分别报告平均、1% low、低于预算的帧比例；context loss/窗口遮挡不混入正常结果。
5. 每项预算分别在最低P0硬件档和参考档执行。只有最低档达到需求才可作为P0 PASS；更高档结果不能替代。
6. CI重试只用于基础设施故障；同一build第一次阈值失败不能靠重跑取最好结果，必须保留失败并建立finding。

基准证据保存原始样本、统计脚本版本/hash、profile、退出码和图表数据；仪表板只能读取证据，不能手工改写结论。
