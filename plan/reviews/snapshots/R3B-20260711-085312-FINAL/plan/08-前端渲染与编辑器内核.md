# 前端渲染与编辑器内核

## 1. 文档目的

本文档定义 React 应用壳、PixiJS 编辑器内核、Rig IR 预览、蒙版/骨骼/网格/时间轴工具和前端性能边界。页面结构和视觉规范以 `F2S-DOC-UX-001` 和 `F2S-DOC-UI-001` 为准；本文档只规定它们如何被可靠、高性能地实现。

核心原则是：

> React 管理应用壳和可访问交互，PixiJS 管理高频画布，Rust 管理项目真值，Rig IR 是预览语义。

ADR owner按 `F2S-ADR-REGISTRY-001` 逐项解析：本文只读引用跨领域ADR，并唯一拥有frontmatter枚举的三个RENDER implementation-local ADR canonical body。

## 2. 必须覆盖的需求

本设计必须完整覆盖：

- `F2S-FR-LAYER-002`、`F2S-FR-LAYER-003`、`F2S-FR-LAYER-004`、`F2S-FR-LAYER-005`、`F2S-FR-LAYER-006`、`F2S-FR-LAYER-007`：蒙版编辑、图层管理、重组差异和外部重载；

## 3. 渲染架构决策

以下三项是渲染子系统的 implementation-local ADR；canonical body 与语义由本文维护。`F2S-DOC-ADR-001` 只在全局 ADR owner registry 登记归属并提供跨文档索引，不复制或改写这些决定。

| 决策 ID | 决策 | 结果 |
| --- | --- | --- |
| `F2S-ADR-RENDER-001` | React 组件不持有 Pixi DisplayObject，两者通过稳定 Editor Kernel 接口交互 | React render 不会重建画布场景树 |
| `F2S-ADR-RENDER-002` | 画布消费不可变 `RenderSnapshot` 和增量 `RenderPatch` | 渲染器不能偷偷修改项目，也无需每帧发送全项目 |
| `F2S-ADR-RENDER-003` | 高频拖拽在前端临时预览，指针释放后只提交一条领域 Command | 达到低延迟且保持可撤销命令边界 |

不使用官方 Spine Runtime 或其派生代码进行预览。PixiJS 只实现已在 Rig IR 支持矩阵中声明的骨骼、slot、attachment、mesh、deform、constraint、event 和 sequence 子集。对 Spine 4.2.43 的最终一致性依靠导出后官方 Editor/CLI 验证，不由内部预览器自证。

## 5. 前端状态分层

不得把所有状态放入一个全局 Store。建议划分为：

| 状态类型 | 示例 | 持久化 | 权威方 |
| --- | --- | --- | --- |
| Server/Project State | AssetSpec、LayerSet、Rig IR、Clip、Approval | Rust 项目库 | Rust |
| Query Cache | 当前项目快照、Job 列表、诊断摘要 | 不独立持久 | Rust 快照的前端投影 |
| Workspace State | 选择对象、展开面板、当前动作、相机 | 可选本地布局设置 | React/Editor Kernel |
| Ephemeral Gesture | hover、拖拽预览、蒙版画笔采样 | 不持久 | Editor Kernel |
| Draft Form | 未提交的 StyleSpec 字段、对话框输入 | 只在显式草稿需求下持久 | React |

当 Rust 拒绝命令或返回 revision 冲突时，前端临时预览必须回滚到最新权威快照，并告知用户未保存操作。

## 6. 渲染协议

### 6.1 `RenderSnapshot` 接口 `F2S-IFC-101`

```ts
interface RenderSnapshot {
  schemaVersion: string;
  projectId: string;
  snapshotId: string;
  streamEpoch: string;
  projectRevision: number;
  rigRevision: number;
  clipRevision?: number;
  coordinateSystem: "x-right-y-up-ccw-degrees";
  viewportHints: ViewportHints;
  textures: TextureDescriptor[];
  drawables: DrawableNode[];
  skeleton: RenderSkeleton;
  animation?: EvaluatedPose;
  references: ReferenceFrame[];
  diagnostics: DiagnosticMarker[];
}
```

`RenderSnapshot` 是给渲染器的精简投影，不包含完整审计日志、PromptPack 或 Worker 详情。渲染器不得将其反序列化为项目。

### 6.2 `RenderPatch` 接口 `F2S-IFC-102`

```ts
interface RenderPatch {
  projectId: string;
  snapshotId: string;
  streamEpoch: string;
  baseRevision: number;
  patchRevision: number;
  operations: RenderOperation[];
}

type RenderOperation =
  | { kind: "replaceTexture"; texture: TextureDescriptor }
  | { kind: "updateTransform"; id: string; world: Matrix2D }
  | { kind: "updateMesh"; id: string; mesh: RenderMesh }
  | { kind: "updateVisibility"; id: string; visible: boolean }
  | { kind: "replaceDiagnostics"; items: DiagnosticMarker[] }
  | { kind: "invalidate"; reason: string };
```

每次全量订阅生成新的不可复用 `snapshotId` 和 `streamEpoch`；项目切换、重新打开、schema 重协商、全量刷新或恢复后都必须换 epoch。Patch 只有同时满足 `projectId`、`snapshotId`、`streamEpoch` 与当前订阅完全相等、`baseRevision` 等于当前 Snapshot project revision、且 `patchRevision` 紧接前一 patch 时才可应用。错项目、旧 epoch、重复、错过或乱序 patch 必须硬拒绝并请求新 Snapshot，不得因 revision 数字相同而跨项目/跨订阅套用。

### 6.3 编辑意图 `F2S-IFC-103`

```ts
type EditorIntent =
  | { kind: "commitBoneTransform"; boneId: string; before: Transform; after: Transform }
  | { kind: "commitPivot"; attachmentId: string; after: Vec2 }
  | { kind: "commitMeshVertices"; meshId: string; changes: VertexDelta[] }
  | { kind: "commitWeightStroke"; meshId: string; stroke: WeightStroke }
  | { kind: "commitMaskStroke"; layerId: string; stroke: MaskStroke }
  | { kind: "commitKeyframes"; clipId: string; edit: KeyframeEdit }
  | { kind: "setGameplayMarker"; clipId: string; marker: GameplayMarkerCandidate };
```

Intent 只表达用户想做什么，不声称已成功。UI 只有在 Rust 返回新 revision 后才将操作加入已提交历史。

### 6.4 诊断证据 `F2S-IFC-104`

```ts
interface DiagnosticMarker {
  diagnosticId: string;
  ruleId: string;
  severity: "info" | "warning" | "error";
  objectIds: string[];
  clipId?: string;
  time?: TimelinePosition;
  bounds?: Rect;
  messageKey: string;
  suggestedAction: SuggestedAction;
  acceptedRisk?: AcceptedRiskSummary;
}
```

状态不能只依赖颜色。每个 marker 都有图标、严重度文字、对象链接和规则 ID。

## 7. 坐标、单位与高 DPI

| 空间 | 原点/方向 | 单位 | 转换位置 |
| --- | --- | --- | --- |
| Source Pixel | 左上原点，X右/Y下 | 图像像素 | Image Adapter |
| Rig World | 用户批准 origin/地面，X右/Y上 | 像素浮点 | Rig IR |
| Bone Local | 父骨局部，逆时针为正 | 像素/度 | Rig IR |
| Canvas Logical | CSS 左上，X右/Y下 | CSS px | Editor Camera |
| Device Pixel | 左上，X右/Y下 | physical px | Pixi Renderer |

坐标转换必须集中在 `CoordinateMapper`，工具不得各自翻转 Y 轴。Canvas 以 CSS 尺寸交互，Renderer 使用 `devicePixelRatio` 渲染；命中半径以 CSS px 定义，所以 100%–200% Windows 缩放下有一致的操作感受。

高 DPI 变化、窗口跨显示器和窗口缩放时，必须重建 render target 和重算 Camera，不得改变 Rig World 坐标或用整数舍入污染 pivot。

## 9. 图层与蒙版编辑

### 9.1 工具集

- 画笔和橡皮：尺寸、硬度、不透明度、压感映射可配；
- 套索：多边形闭合、加/减/交选区；
- 填充：基于当前 mask 而不是母版颜色误差的不可控扩散；
- 收缩/扩张：以像素为单位，预览和提交参数一致；
- 羽化：上限受控，防止将硬边二次元线稿自动糊化；
- 独显/隐藏/锁定/分组/排序；
- 母版重组、差异热图、alpha 洞和边缘颜色污染迭加。

### 9.2 笔画提交

指针移动事件由 Editor Kernel 合并为带压力和时间的 stroke，本地即时更新预览纹理。指针释放后，只提交一个 `ApplyMaskStroke` Command。Rust 使用固定算法重放 stroke，生成新的不可变 mask Artifact。

如 Rust 重放结果与前端预览哈希不符，必须显示权威结果并记录差异，不能以 WebGL 纹理直接写项目。

### 9.3 AI 补全像素识别

新生成/补全像素在画布上使用可切换的纹理网格、轮廓线和“AI 派生”文本标记，不只用单一颜色。修改母版可见像素必须发出错误级诊断，除非用户进入明确的手工重绘模式。

## 10. Rig 编辑工具

### 10.1 骨骼与槽位

- 骨骼树结构在 DOM 面板中编辑，画布提供位置、旋转、缩放和骨长 gizmo；
- reparent 在提交前本地预检测成环，Rust 再做权威验证；
- slot/draw order 在资源树中调整，画布实时预览；
- attachment 替换不隐式改 pivot；尺寸变化需显示映射对话框；
- `primary_weapon` 的握持点、武器接触点、脚底和地面点使用不同形状与文本。

### 10.2 Mesh 与 Weight

- 顶点、边和三角形分模式编辑；选择优先级明确，避免骨骼/mesh 命中冲突；
- 移动顶点时实时标记反转、零面积、交叉边候选，Rust 负责最终不变量；
- 权重画笔显示当前骨影响热图，状态同时有数值；
- 权重提交后由 Rust 归一化并返回修正报告；
- 镜像必须由用户选择对称轴、骨骼映射和覆盖策略，不根据名称静默推测。

### 10.3 约束

UI 只展示 Rig IR 支持矩阵中已实现的 IK、transform 和 path 属性。不支持的 Spine 4.2 属性不使用“高级”空壳控件伪装；它们在能力矩阵中显示 `unsupported`，并在导出 preflight 中保持一致。

## 11. 动画时间轴

### 11.1 时间模型

- Rig IR 持久化时间使用“有符号整数 tick + 有理数 time base”，不存 `f64` 秒。协议中 i64 tick 以十进制字符串表达，避免 JavaScript 安全整数丢失；
- 每个关键帧拥有与时间位置无关的稳定 `keyframeId`。移动、缩放或改变显示帧率不改变该 ID；复制关键帧创建新 ID；
- Clip 的 time base 和用于 UI 显示/吸附的 frame rate 都使用分子/分母；改变显示 frame rate 不自动移动关键帧；
- 只有 Spine 4.2.43 导出 Adapter 可在最后边界依照 `F2S-ADR-TIME-001`，用精确有理数与 checked `i128` 将 tick 转换为最终规范化秒数字面量，并执行文本值与 IEEE-754 binary32 的有限性、顺序及碰撞门禁；`f64` 只可作为不持久化的 UI 显示/瞬时计算值，既不是导出事实源，也不得参与规范序列化或回写 Rig IR；
- 时间轴缩放、滚动和播放头是本地状态；关键帧和游戏 marker 是领域状态；
- 拖拽关键帧时保留原始时间和相对间距，提交时作为一组命令。

```ts
interface TimelinePosition {
  tick: string; // signed i64 decimal
  rateNumerator: number;
  rateDenominator: number;
}

interface KeyframeIdentity {
  keyframeId: string;
  at: TimelinePosition;
}
```

### 11.2 轨道和曲线

必须支持骨骼变换、slot颜色/可见、attachment、draw order、deform、constraint、event 和 sequence 所需轨道。每个轨道行只渲染可见时间窗口，使用虚拟列表，不将所有关键帧各自创建 React DOM 节点。

曲线编辑支持 stepped、linear 和 cubic Bezier。对不合法 control point 做显式约束和错误提示，不在导出阶段静默夹取。

### 11.3 播放与对比

- 播放时钟基于 monotonic clock，渲染掉帧不改变动画时间；
- 逐帧模式按显示 frame rate 精确前后移动；
- 洋葱皮最多默认前后各 2 帧，超限由性能预算提示；
- 工作版与批准版可同时播放，时间映射差异必须显示；
- 差异热图使用独立 render target，不写入项目资产。

## 12. 关键姿势、命中帧和接触点

`attack_01`、`attack_02`、`attack_03` 必须可视化 anticipation、active/hit、recovery、weapon_contact 和用户定义 event。`hit` 和 `death` 必须显示对应关键阶段。

- AI/模板 marker 使用“候选”图形；
- 用户批准后使用“已批准”文本和锁定图标；
- 关键帧、时长、frame rate 或关联骨骼变更使审批生命周期历史保持 `approved`，其新鲜度维度变为 `stale`，不直接删除旧证据；
- 时间轴提交前显示将失效的下游审批数量。

## 13. 诊断迭加

| 诊断 | 画布证据 | 导航行为 |
| --- | --- | --- |
| 透明洞 | 棋盘空洞轮廓+文本标号 | 选中关联 Layer/AssetSpec |
| Mesh 过度拉伸 | 三角形应变热图+峰值 | 跳到对应时间和顶点 |
| 轮廓跳变 | 前后帧轮廓双线 | 对比前后关键帧 |
| 脚滑 | 接触区间轨迹+移动距离 | 选中脚接触 marker/root 轨道 |
| 武器脱手 | grip/socket 连线+距离 | 选中 `primary_weapon` attachment 与手骨 |
| 地面穿透 | 地面线+穿透区域 | 选中 root/脚骨 |
| draw order 冲突 | 冲突 attachment 轮廓和 slot 序 | 打开 draw-order 轨道 |
| 循环接缝 | 首尾帧差异热图 | 同时显示首/尾关键帧 |

诊断迭加可隐藏以达到 60 FPS，但隐藏不改变诊断本身状态。导出前仍按领域报告阻止错误。

## 14. 纹理、内存与性能

### 14.1 资源策略

- 原始高分辨率图像由 Rust 产生代理金字塔；画布按缩放级别选择合适代理；
- 画布不同时常驻全部动作和所有参考图纹理；
- 纹理缓存使用 LRU 和显式 pin，当前可见资产不得被回收；
- 用户可看到代理分辨率、纹理内存估算和当前降级状态；
- 参考图、洋葱皮和差异 render target 应独立统计。

### 14.2 预算

| 指标 | 目标 | 测量方式 |
| --- | --- | --- |
| 画布交互反馈 | P95 不高于 100ms | 输入事件时间至首个可见帧 |
| 普通预览 | 1920×1080、无诊断迭加持续 60 FPS | 目标夹具 60 秒帧时间 |
| 诊断预览 | 不低于 30 FPS | 全迭加夹具 |
| UI 长任务阻塞 | 单次不高于 200ms | Long Task 记录和事件循环 heartbeat |
| 轨道虚拟化 | 10,000 关键帧时 DOM 节点数不随总数线性增长 | 开发计数器 |
| 8GB AI 任务预检 | 任务开始前显示 VRAM/RAM/临时磁盘、模型和质量档；不足时阻止或等待用户选择 | `F2S-NFR-PERF-006` 资源预算事件与 UI 截图 |
| manifest 提交 | 标准 fixture P95 不高于 500ms；大 Artifact 异步且不阻塞画布 | `F2S-NFR-PERF-007` 提交 trace 与事件循环 heartbeat |

不通过降低导出图片分辨率来满足预览性能。代理和源产物必须有不同 Artifact 类型和明确标识。

## 15. WebGL/WebGPU 与上下文丢失

首版优先使用 PixiJS 在 WebView2 上的已验证 WebGL 路径，WebGPU 仅在完整回归后可作受控实验功能，不作为首版必须前置。

Context lost 时：

1. 暂停播放和画布命令提交；
2. 显示不遮挡数据保存的恢复提示；
3. 从 `RenderSnapshot` 和受管纹理描述符重建场景；
4. 丢弃未提交 GPU 预览纹理，但不丢失已提交的 mask stroke/Command；
5. 连续恢复失败时进入只读表单模式，允许保存和导出诊断。

## 16. 键盘、焦点与可访问性

- 所有非画布主流程可用键盘完成；焦点顺序遵循信息架构。
- 画布中的选择对象必须在 DOM 属性面板中有对应可聚焦表示，不将画布作为唯一操作入口。
- 快捷键必须在菜单/命令面板显示，并处理中文输入法 composing 状态，避免误触发。
- 色盲模式不只更换颜色，还为诊断、AI 像素、审批和轨道类型增加形状/纹理。
- 缩放文本、125%–200% DPI 和高对比主题下必须可达所有确认/退回按钮。

## 17. 失败与冲突处理

| 失败 | 前端行为 | 禁止行为 |
| --- | --- | --- |
| Snapshot schema 不兼容 | 停止画布并提供更新/只读诊断 | 忽略未知字段后继续保存 |
| 纹理解码失败 | 占位图+错误 ID+重试/重新链接 | 用空透明图伪装成功 |
| Command revision 冲突 | 回滚临时预览，刷新并显示影响 | 前端自行合并 Rig 结构 |
| 蒙版提交失败 | 保留未提交 stroke 为可重试草稿 | 将 GPU 预览当已保存 |
| 时间轴数据失效 | 暂停播放，导航到冲突轨道 | 继续展示过期 pose |
| 性能超过软限 | 显示代理/关闭迭加/释放非当前纹理选项 | 静默修改源图或删除关键帧 |
| WebGL 上下文丢失 | 按第15节重建 | 让应用整体退出且不保存 |

## 19. 许可与供应链边界

- React、PixiJS、Vite、Zustand/同类候选只能在锁版后按实际传递依赖通过宽松许可门禁。
- PSD 输出 Adapter 的执行与文件写入归属 Rust `F2S-CMP-008 Export Gateway`；React/Pixi 只编辑导出配置、显示 preflight/进度和读取结果清单，不生成 PSD bytes、不把大 PSD 经 WebView IPC 传输、也不写导出目录。具体 Rust 依赖只能在 `F2S-DOC-EXPORT-001` 的 PSD Spike、内存峰值和 Photoshop/Spine UI 夹具通过后冻结；本文不指定第二套前端 PSD 库。
- 不使用 Pixi 生态中的任何 Spine Runtime 包，即使包管理页面显示 MIT，也必须追溯源代码和官方专有 Runtime 许可。
- 字体、图标、光标和演示图像必须有独立许可 manifest；系统字体可作为 OS 前置，不从 Windows 复制到安装包。

## 20. 完成定义

前端渲染与编辑器内核只在以下条件全部成立时完成：

1. 十个动作的 Rig IR 夹具可预览和编辑；
2. 蒙版、骨骼、mesh、weight、时间轴和 marker 均能撤销/重做；
3. 审批版与工作版可比较，诊断可导航到证据；
4. 目标 Windows/DPI/GPU 矩阵达到性能和可访问性验收；
5. 无模型、无网络、无 Spine Runtime 时手工编辑和预览可用；
6. 任何画布失败都不会破坏最后一个已提交项目版本。
