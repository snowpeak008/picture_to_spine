# 本地图像处理与素材规划管线

## 1. 目标和范围

本文档定义从“用户导入一张母版候选图和动作文字/关键姿势图”到“可批准 LayerSet、Rig 建议和补素材请求”的处理管线。管线既要允许 AI 提高效率，也要在没有 AI Runtime Pack、没有 GPU、没有网络时提供完整手工路径。

上游生图不在本产品内。产品可以生成、导出、复制和重新关联提示词包，但不得向公有云或第三方生图 API 发送任何图片、规格、提示词或项目数据。

依 `F2S-ADR-REGISTRY-001`，`F2S-ADR-PIPE-001`至`003`的implementation-local canonical body由本文件拥有；24号只登记其exact owner和状态。

## 2. 已冻结的管线决策

| 决策 ID | 决策 | 实现影响 |
| --- | --- | --- |
| `F2S-ADR-PIPE-001` | 管线采用“动作/需求先行 + 母版真值 + 按需补素材”，不先生成所有可能图片 | AssetSpec 覆盖和试跑诊断驱动素材数量 |
| `F2S-ADR-PIPE-002` | 母版中已批准的可见像素是视觉真值；分层和隐藏补全不得静默改写 | 每个派生图像必须有像素 provenance 和重组差异报告 |
| `F2S-ADR-PIPE-003` | 手工、本地轻量、本地 Python 和私有远程是同一 Inference/Planning Port 的策略，不是不同产品流程 | 所有自动结果均进 candidate→人工审核路径 |

## 3. 管线总览

```text
P00 Intake & Security Validation
  → P10 Decode & Normalize Candidate
  → P20 Master Review & Approval
  → P30 Spec/Motion Parsing & Representation Planning
  → P40 AssetSpec Aggregation & PromptPack
  → P50 Segmentation / Semantic Layer Assignment
  → P60 Hidden-overlap Completion / Corrective Assets
  → P70 Recomposition QA & Layer Approval
  → P80 Rig Hints / Placeholder Rig
  → P90 Motion Trial / Diagnostic Feedback
  → Supplement AssetSpec → P40/P50/P60 ...
```

每个阶段都实现为可重入、可取消、输入哈希明确的 Stage。阶段输出是不可变 Artifact 或结构化 Candidate，不直接改写上游。

## 4. 管线组件

| 组件 ID | 名称 | 输入 | 输出 |
| --- | --- | --- | --- |
| `F2S-CMP-201` | Image Intake | 用户文件 | 受验证 SourceArtifact 或拒绝报告 |
| `F2S-CMP-202` | Master Normalizer | SourceArtifact、用户方向/地面/尺度 | NormalizedMasterCandidate + 差异报告 |
| `F2S-CMP-203` | Representation Planner | 已批准 Specs + 动作文字/关键姿势图 | RepresentationPlanCandidate + 不确定项 |
| `F2S-CMP-204` | Asset Aggregator | RepresentationPlan + 基础分层规则 | AssetPlanCandidate + CoverageReport + PromptPackCandidate |
| `F2S-CMP-205` | Layer Processor | 母版、AssetSpec、用户蒙版/推理建议 | LayerSetCandidate + PixelProvenance + GenerationProvenance + RecompositionReport |
| `F2S-CMP-206` | Trial Feedback Analyzer | LayerSet/Rig/Clip 试跑帧 | DiagnosticRecord + SupplementAssetRequestCandidate |

## 5. P00：导入与安全验证

### 5.1 支持输入

- P0：PNG、JPEG、WebP 静态图片；PNG/WebP 可含 alpha，JPEG 无 alpha 时按显式不透明输入处理；
- 关键姿势参考：同样使用受验证的静态图片，只作为 MotionSpec/PromptItem 参考；
- V1 明确拒绝：PSD/PSB；它们只可作为导出格式，不能进入母版或图层回导；
- 非首版：视频、BVH、3D 模型、分层 Live2D 工程。

导入不信任扩展名，必须验证 magic bytes、声明/实际尺寸、像素总量、通道、bit depth、ICC profile、alpha、解码错误、压缩比和文件哈希。设置可配置硬限，防止解压炸弹和超大纹理导致桌面进程崩溃。

检测到 PSD/PSB magic bytes 时，无论扩展名为何都必须无副作用拒绝，说明“V1 不支持 PSD 输入”，并建议用户从外部编辑器导出 PNG/JPEG/WebP。不得栅格化合成后静默导入，也不得留下半个 SourceArtifact。

## 6. P10：解码与规范化

### 6.1 不可变原始件

SourceArtifact 按 SHA-256 保存原始字节，不进行原位旋转、裁切、去背景、改色或 alpha 修复。解码后的规范化候选是新 Artifact，并记录：

- decoder 名称/版本；
- 原始 media type、尺寸、bit depth、ICC/色空间；
- 转换到工作 sRGB/RGBA 的参数；
- EXIF/方向处理；
- 画布扩展/裁切、透明边界和地面 origin；
- 源哈希和规范化产物哈希。

### 6.2 Alpha 和颜色规则

- 持久化 PNG 使用 straight alpha；Pixi/WebGL 预乘只在渲染 Adapter 内发生；
- RGB 颜色管理转换只做一次，每个 Artifact 记录工作色空间；
- 全透明像素的 RGB 不作视觉真值，但边缘扩色/防溢色必须作为独立派生步骤记录；
- 不在导入时自动“优化”二次元线稿的抗锯齿或饱和度。

## 7. P20：母版候选和人工批准

母版门禁必须允许用户：

1. 在 1:1 像素级别查看原始/规范化对比；
2. 选择横版侧视朝向、根节点原点、地面线和像素尺度；
3. 声明母版是否已包含 `primary_weapon`；
4. 查看 alpha、色彩、裁切、解码和重复图片诊断；
5. 批准或退回并填写必要原因。

只有 approved MasterRevision 可成为正式分层和 Rig 的上游。用户之后可替换母版，但必须经过 InvalidationPlan，旧分层、Rig 和动画保留为 stale 版本。

## 8. P30：规格分析与表现策略

### 8.1 输入分析

规则引擎先读取 CharacterSpec、StyleSpec、WeaponSpec、RuntimeSpec、十个 MotionSpec、动作描述和关键姿势图的用户标注。可选本地语言/VLM 能力只能提出：

- 动作阶段候选；
- 身体区域、武器、长发/衣摆等可能受影响部位；
- 视角变化、轮廓变化、拓扑变化、强透视和特效的不确定性；
- 需要人工回答的结构化问题。

AI 不能从关键姿势图自动声称精确 pivot、骨长、命中帧、root motion 或 weapon_contact 已知。

### 8.2 确定性表现规则

| 需求特征 | 默认策略 | 素材行为 |
| --- | --- | --- |
| 小幅平面内变换，轮廓稳定 | bone/mesh | 优先复用基础层，不新增图 |
| 关节屈曲会露出遮挡区 | mesh + hidden overlap/corrective | 增加隐藏延展或动作修正件 |
| 眼/嘴/手型/握持状态改变 | attachment swap | 建立稳定 slot 变体 |
| 出平面转向/明显新方向 | new-view/new-rig | 生成新视角 AssetSpec，不强拉基础 mesh |
| 极端透视、轮廓/拓扑突变、smear | key-pose attachment/sequence | 生成动作专用姿势或短序列 |
| 刀光、火焰、尘土、残影 | vfx/sequence | 与角色本体分离的 AssetSpec |
| 武器/服装变体 | skin/attachment | 稳定 slot、手型和武器套件 |

用户可覆盖规则结果，但必须选择/填写原因。如用户将极端出平面姿势强制改为 bone-only，系统保留高风险诊断，不把覆盖当成证明质量可达。

## 9. P40：AssetSpec 聚合与提示词包

### 9.1 聚合规则

管线以 `semantic + view + variant + representation + styleRevision` 作为候选合并键，再计算可否共享。以下条件任一不同时不能静默合并：

- pivot 或父骨语义；
- 前后层级/遮挡；
- 所需隐藏重叠范围；
- 强透视或轮廓要求；
- 颜色/光照规则；
- 动作专用与通用素材属性。

CoverageReport 必须同时列出：每个动作需要什么、哪些 AssetSpec 覆盖它、哪些需求仍未覆盖、哪些素材完全未使用。

### 9.2 基础分层

使用 `F2S-DOC-CONTENT-001` 定义的 15 项身体候选最小集：`F2S-AST-BODY-001`、`F2S-AST-BODY-002`、`F2S-AST-BODY-003`、`F2S-AST-BODY-004`、`F2S-AST-BODY-005`、`F2S-AST-BODY-006`、`F2S-AST-BODY-007`、`F2S-AST-BODY-008`、`F2S-AST-BODY-009`、`F2S-AST-BODY-010`、`F2S-AST-BODY-011`、`F2S-AST-BODY-012`、`F2S-AST-BODY-013`、`F2S-AST-BODY-014`、`F2S-AST-BODY-015`，以及 `F2S-AST-WEAPON-001`。长发、裙摆、披风、饰物使用根据母版建立且具有精确 `F2S-AST-SECONDARY-<ULID>` 实例 ID 的 secondary AssetSpec，不强制固定层数。

基础集是规划模板而非分割模型必须一次成功的输出。两个区域可以暂时使用一张占位附件试跑，但 Layer Gate 必须指出未拆分风险。

### 9.3 提示词包

PromptPack 使用确定性模板作为无 AI 的基线，可选本地语言模型只能提供改写候选。每个 PromptItem 包含：

- 全局身份/风格约束；
- 单动作阶段和构图；
- 透明背景、视角、光照和分辨率；
- 需要补全的隐藏区或修正附件；
- 正向、负向、不可改写要素；
- 引用的 StyleSpec/MotionSpec/AssetSpec revision 和母版哈希；
- 未决占位符、templateVersion、resolvedTextHash。

WeaponSpec 未具体化时，攻击 PromptItem 必须带明显 `UNRESOLVED_PRIMARY_WEAPON`，不可作为正式生产包。

## 10. P50：分割和语义分层

### 10.1 策略级别

| 策略 | 能力 | 定位 |
| --- | --- | --- |
| Manual | 画笔、套索、填充、收缩/扩张、语义指定 | 始终可用的权威退路 |
| Local lightweight | SAM 2.1 tiny/small 类点/框/蒙版交互分割、边缘工具 | 8GB GPU 主档，结果仍需命名/修正 |
| Local Python advanced | 经审计的多层分解/补全 | 可选 Runtime Pack，依能力 manifest |
| Private remote | Qwen-Image-Layered 类高成本候选、组织自有模型 | 用户自托管，每 Job 发送前批准 |

SAM 2 代码和 checkpoint 官方声明为 Apache-2.0，Qwen-Image-Layered 官方声明为 Apache-2.0。这些只构成候选的许可起点，实际 Runtime Pack 仍必须审计所有传递依赖、权重文件、训练数据声明和分发约束。

### 10.2 分割不等于动画分层

模型输出的语义区域必须再经 AssetSpec 映射。“手臂”的单个 mask 可能需要分为 upper/lower/hand；一个裙摆区域可能需要多个可动附件；头发可能需要根据 draw order 分前/后。

所以自动分层产出 `SegmentationProposal`，而不是 approved LayerSet。用户必须为每个区域确认语义、父骨建议、前后顺序、是否需隐藏延展。

### 10.3 边缘和重叠

- mask 边界尽量在将来附件遮挡内，不一律沿当前可见线切割；
- 肩、肘、腕、胯、膝、踝使用 AssetSpec 的重叠要求验证；
- 近侧/远侧命名优先于 left/right，避免侧视镜像混乱；
- 边缘颜色污染、白边/黑边、alpha 台阶和锐利线稿羽化是独立诊断。

## 11. P60：隐藏补全和修正附件

### 11.1 像素来源

每个 LayerArtifact 必须伴随同尺寸 `PixelProvenanceMap`，至少区分：

```text
source_visible       # 直接来自批准母版的可见像素
generated_hidden     # 模型在原遮挡区生成
manual_hidden        # 用户手工补画的原遮挡区
manual_visible_edit  # 用户显式重绘可见区
empty
```

不存在“AI 输出所以来源已知”的默认。Worker 返回图像后，Rust 使用请求 mask 和批准母版自己计算 provenance，并验证可见保护区。

### 11.2 可见像素保护

对每个由母版指定的 authoritative visible pixel：

- 默认必须从母版直接复制，不采信模型重绘值；
- 分层重组中由一个 authoritative owner layer 提供，避免多层重叠改色；
- 如用户显式进入重绘可见区模式，产出 `manual_visible_edit` 新母版候选，不在原母版下静默认可；
- 在直接拷贝路径上，重组后可见 RGBA 应字节级相同；发生色空间/混合必要容差时，容差必须在版本化质量配置中说明。

### 11.3 生成式补全门禁

生成式隐藏补全执行前显示输入层、保护 mask、可生成 mask、模型/版本、分辨率和资源估算。输出后必须检查：

- 可见保护区是否被改写；
- 补全内容是否溢出允许 mask；
- 线宽、色板、光向和纹理密度是否违反 StyleSpec；
- 隐藏部分在关节试跑时是否真正覆盖需要范围；
- 是否生成不合理肢体/服饰拓扑。

只有 Layer Gate 能批准补全结果。模型置信度不能替代人工审核。

### 11.4 GenerationProvenance 硬门

`PixelProvenanceMap` 回答每个像素来自哪里，`GenerationProvenance` 回答一次生成/分析任务由什么能力、输入和参数产生；两者都必须存在且不可互相替代。

内部 `internal_local` 或 `internal_private_remote` 任务必须记录：

- Provider/Worker ID、协议版本、模型/算法 ID 与精确版本；
- Runtime/Model manifest hash、参数、seed（适用时）、设备与执行位置；
- JobId、idempotency key、输入资产/遮罩/StyleSpec/MotionSpec/AssetSpec revision 与 hash；
- 输出 hash、开始/结束时间、取消/重试/恢复父链；
- tile/offload/CPU/小模型等全部质量降级及用户确认。

任一字段缺失时输出只能进 quarantine，不能成为 `JobOutputArtifact` 或 candidate。产品外部回导若无法提供工具、模型、版本或 seed，必须存 `sourceKind=unknown_external`、文件 hash、导入时间和非空用户来源声明；不得伪造内部字段或可复现性。P70 Layer Gate 与导出报告必须持续展示该来源警告。

## 12. P70：重组质检

### 12.1 RecompositionReport

| 检查 | 证据 | 错误边界 |
| --- | --- | --- |
| 可见像素差异 | RGBA 差异图、峰值、像素数 | 非显式可见重绘导致任何超容差差异为 error |
| 透明洞 | 母版非透明但重组 alpha 为空的区域 | error |
| 未归属像素 | 母版可见但没有 authoritative owner | error |
| 重复可见 owner | 多层同时改变母版可见结果 | error |
| 边缘色污染 | alpha 边界的黑/白/背景溢色 | warning/error 由阈值 |
| 关节重叠不足 | AssetSpec 要求区与 layer coverage | error 或建立 corrective AssetSpec |
| 层级冲突 | 动作中需要无法表达的 draw-order 循环 | error，要求拆层/替件 |
| AI 像素占比 | generated_hidden 统计和可视化 | info/warning，不允许隐藏 |

报告与 MasterRevision、LayerSetCandidate、qualityProfileVersion 绑定。修改任何 layer/mask 使报告 stale，只重跑受影响的确定性检查。

### 12.2 Layer Gate

用户可以逐层 accept/reject/replace，但整体 LayerSet 只在所有 required AssetSpec 有映射、错误清零、warning 已处理且像素 provenance 完整时可批准。单层 accept 不等于整体 Layer Gate 通过。

## 13. P80/P90：Rig 建议与反馈闭环

### 13.1 Rig Hint

管线可从人工关节点、轻量姿态估计、AssetSpec 父骨/pivot 建议和语义模板生成：

- SkeletonTemplate 映射候选；
- bone position/length 候选及置信度；
- slot/draw-order 候选；
- pivot、socket、地面/脚接触候选；
- mesh 边界和权重初始值候选。

这些全部是 editable candidate，不是从图片已“识别出真实骨骼”。Rig Gate 仍由用户审核。

### 13.2 临时 Rig

当素材未完整时，使用透明彩色占位 attachment 和当前层建立临时 Rig。占位必须在画布和报告中明显，不进入导出。十个 MotionSpec 都应有最小试跑或姿态扫描，用于发现：

- 透明洞/隐藏像素不足；
- 关节轮廓断裂或拉伸；
- 长发/衣摆需要独立层；
- draw order 需求无法表达；
- 强透视/转向需要新视角或序列；
- 武器握持、脚底和地面接触错误。

### 13.3 诊断到修复类型

| 诊断 | 默认修复 | 是否创建新 AssetSpec |
| --- | --- | --- |
| 小幅权重/网格问题 | 调 mesh/weight | 否 |
| 隐藏区不足 | 扩展 hidden overlap | 只有需新像素时 |
| 特定关节轮廓失真 | corrective attachment | 是 |
| 手型/表情/武器状态 | attachment variant | 是 |
| 出平面转向 | new-view/new-rig | 是 |
| 极端透视/smear | key-pose/sequence | 是 |
| 脚滑/时序/接触 marker 错误 | 修动画/人工 marker | 否 |
| 纯 VFX | 独立 vfx/sequence | 是，不合并角色本体 |

SupplementAssetRequest 只在问题需要新图像时创建。诊断是动画、Rig 或 marker 问题时，不通过增加图片转移问题。

## 14. 管线接口

### 14.1 Stage 接口 `F2S-IFC-301`

```rust
struct StageContext {
    job_id: JobId,
    idempotency_key: IdempotencyKey,
    normalized_input_hash: ContentHash,
    cancellation: CancellationToken,
    progress: ProgressSink,
    checkpoints: CheckpointStore,
}

trait PipelineStage {
    fn descriptor(&self) -> StageDescriptor;
    fn estimate(&self, input: &StageInput) -> Result<ResourceEstimate, StageError>;
    fn run(&self, input: StageInput, ctx: StageContext)
        -> Result<StageRunResult, StageError>;
    fn resume(&self, checkpoint: VerifiedCheckpoint, ctx: StageContext)
        -> Result<StageRunResult, StageError>;
    fn verify(&self, output: &StageOutput) -> ValidationReport;
}
```

`StageDescriptor` 包含版本、确定性、所需能力、许可 manifest 引用、输入/输出 schema、取消安全点和可否 checkpoint。契约规则：

1. Stage 在声明的安全点轮询 `CancellationToken`；取消产生 `Cancelled` 或版本化 checkpoint，不伪装成 error/success。
2. `ProgressSink` 只接收带 stage/step/completed/total/unit/sequence 的有界事件，不参与持久状态仲裁。
3. checkpoint 必须绑定 stage ID/version、输入 hash、规范化参数、Provider/model/manifest hash、随机状态、已完成分片和 checksum；任一不匹配时拒绝 resume。
4. `idempotency_key` 由 stage/version、输入 hash、规范化参数和能力版本计算。重复 run 只能复用已验证的 `JobOutputArtifact` 或安全继续，不重复提交 candidate。
5. Stage 不能自行修改 Project、Approval 或 Job 终态；JobEngine 持久化运行/取消/中断，`Register*Candidate` 是后续独立事务。
6. 未声明 checkpoint 的 Stage 中断后从新 Job 重跑，不假装任意算子中点可恢复。

## 15. 8GB GPU 降级策略

在 H1 8GB GPU 档执行前，按顺序提供：

1. 使用模型 tiny/small 版本；
2. 使用用户指定的局部 ROI；
3. 使用代理分辨率生成建议，再在原尺寸由确定性 mask 工具修边；
4. 使用 tile 和 CPU offload，显示预估耗时和临时空间；
5. 切换纯手工路径；
6. 如用户已配置自托管私有端点，显示需发送的具体对象并等待批准。

不得在 OOM 后静默减小最终产物、修改 prompt、切换未审计量化权重或自动上传到远程。Qwen-Image-Layered 类高成本模型不在 8GB 基线中承诺可用。

## 17. 失败和恢复

| 失败 | 阶段结果 | 恢复 |
| --- | --- | --- |
| 文件格式/尺寸非法 | P00 拒绝，无 SourceArtifact | 用户更换/转码后重新导入 |
| 解码器崩溃 | 导入 Job 失败，原文件不变 | 隔离进程/安全解码器重试，限次 |
| 母版未批准 | P30 之后只允许草案探索 | 完成 Master Gate |
| 动作描述歧义 | RepresentationPlan 含 unresolved issue | 用户修正 MotionSpec，不由模型猜测 |
| 分割模型 OOM | Job 失败，无候选注册 | 第15节显式降级 |
| 模型改写保护区 | 输出隔离，完整性 error | 使用原像素重建或重跑，不手动忽略 |
| 重组无法匹配母版 | Layer Gate 阻止 | 导航到差异 owner/mask |
| 外部图无 PromptItem/AssetSpec 或来源信息 | 进待归类区，不进入 LayerSet | 用户选择对应项；缺生成信息时登记 `unknown_external` 与来源声明 |
| 试跑反复出现同一问题 | 保留每次证据并升级严重度 | 要求人工决定改 Rig/改动画/新素材，不无限自动重生成 |

## 18. 测试与验收

用户不能提供可自动化验收的真实角色集，因此工程必须创建和审计自制/合成夹具包，对每个夹具记录来源、商用权利、预期层、骨骼、动作风险和 golden 结果。夹具至少覆盖短发/长发、裙摆/披风、近远肢体交叉、透明配饰、含/不含武器、小/大画布、PNG/JPEG/WebP，以及应被无副作用拒绝的 PSD magic bytes。

| 测试 ID | 场景 | 通过条件 |
| --- | --- | --- |
| `F2S-TST-100` | 导入安全语料 | PNG/JPEG/WebP 正负 fixture 按契约处理；伪扩展、解压炸弹、损坏图、PSD/PSB magic bytes 和超限尺寸被确定性、无副作用拒绝 |
| `F2S-TST-101` | 原图与 Stage 幂等 | 所有管线操作后 SourceArtifact 字节哈希不变；同 idempotency key 重跑不重复输出或注册 candidate |
| `F2S-TST-102` | 可见像素保护 | 伪造模型改写保护区时结果隔离；正常重组字节级匹配 |
| `F2S-TST-103` | Pixel/Generation Provenance | 每个非空像素有唯一来源；内部缺任一 generation 字段拒绝，外部缺信息只能 `unknown_external` 加声明 |
| `F2S-TST-104` | 表现策略规则 | 每类动作特征产生预期 bone/mesh/attachment/sequence/new-view/vfx 策略和解释 |
| `F2S-TST-105` | 素材覆盖 | 十个动作无静默缺口，重复/孤儿 AssetSpec 在报告中可见 |
| `F2S-TST-106` | 无AI路径 | 卸载 Runtime Pack、断网和禁用 GPU 后可手工完成 LayerSet candidate/approval |
| `F2S-TST-107` | 8GB GPU 与 Stage 控制 | H1 预检/OOM 不静默降质；cancel、进度序列、合法 checkpoint resume 和不匹配 checkpoint 拒绝均通过 |
| `F2S-TST-108` | 反馈闭环 | 透明洞创建补素材，脚滑只创建动画修复，不混淆问题类型 |
| `F2S-TST-109` | 无公有云 | 在未启用私有端点时，管线全流程网络观测无数据外发 |

AI 质量测试不使用“看起来不错”作为唯一条件。自动不变量（像素保护、覆盖、alpha、边界、哈希、schema）必须精确检查；风格、轮廓和解剖合理性使用双人人工夹具审查或明确的产品责任人签字。

## 19. 许可与隐私边界

- 核心确定性图像处理优先使用 Rust `image`/`imageproc` 类 MIT/Apache/BSD 可审计库，锁版后仍需扫描编解码特性引入的传递库。
- ONNX Runtime 为 MIT 候选，PyTorch 为 BSD-style 候选；GPU wheel 中可能捆绑 NVIDIA 组件，必须在 Runtime Pack 单独审计。
- 第三方仓库页面的“Apache-2.0”不能替代每个权重和数据集的证据。
- 私有远程只能是用户自托管/企业内部/局域网端点；每次任务必须审批发送对象，不存在失败后公有云 fallback。
- 图像、mask、PromptPack 正文和像素 provenance 不进入普通日志或默认诊断包。
