# 18. 交付、发布、安装、更新与双击入口

## 18.1 目标

本文定义闭源商业 Windows 桌面版的构建产物、安装方式、根目录双击入口、签名更新、回退、卸载和发布门禁。交付结果必须让普通用户在不打开终端、不配置开发环境的情况下启动应用；开发工程根目录也必须保留一个稳定、可诊断的双击入口。

技术基线：

- Tauri 2 + React/TypeScript/Vite + PixiJS 8；
- Rust 核心；
- 可选独立 Python 3.12 Worker 包；
- Windows x64；
- NSIS per-user 安装；
- 系统 Evergreen WebView2；
- V1 首个集成目标仅为用户本地 Spine Editor/Professional CLI 4.2.43；
- 不捆绑 Spine Editor 或官方 Spine Runtime。

## 18.2 发布范围

### F2S-REL-PLATFORM-001：支持矩阵

| 环境 | 级别 | 说明 |
|---|---|---|
| Windows 11 x64 最新受支持更新 | P0/一级 | 发布阻断环境 |
| Windows 10 22H2 x64 | P1 | 必须执行 P1 矩阵；若治理批准不随某版本交付，必须撤下兼容声明 |
| NTFS、本地磁盘、普通用户 | P0/一级 | 项目和安装验收基线 |
| 管理员用户 | 兼容 | 应用不依赖管理员权限运行 |
| ARM64、Windows Server、Wine | 不支持 | 未来另立里程碑 |
| 网络盘、同步盘按需文件 | 不支持写入 | 可只读导入副本，不作原地项目承诺 |

GPU 推理是可选能力。没有兼容 GPU 或 Worker 时，项目管理、人工分层/Rig、内部预览和导出仍需启动并可用；UI 显示明确的能力降级。

### F2S-REL-EDITION-001：交付单元

1. **Core Desktop**：必选主程序、UI、Rust 核心、内部 Rig IR 渲染和导出器。
2. **Local AI Worker Pack**：可选、独立版本和独立 SBOM；只有许可证、签名、`windows-appcontainer-v1` 与 `F2S-NFR-SEC-005` 恶意探针门禁全部通过才可分发。
3. **Model Packs**：可选、独立签名、独立 Model BOM；只允许 `F2S-SEC-MODEL-001` 审定的非代码权重，不允许脚本、任意对象反序列化或安装钩子。
4. **Spine Adapter**：Core 内的外部进程 Adapter，不包含 Spine 二进制或 Runtime。
5. **Developer Symbols**：仅内部符号服务器/受控归档，不进入用户包。

Core 必须能在 2～3 均未安装时正常启动和保存项目。

## 18.3 版本与产物命名

### F2S-REL-VERSION-001：版本规则

应用使用 SemVer：

- major：项目 schema 或用户工作流不兼容变化；
- minor：向后兼容功能增加；
- patch：兼容修复。

正式构建同时记录：

```text
productVersion   1.0.0
buildNumber      CI 单调编号
gitCommit        完整 SHA
projectSchema    1.0.0
rigIrSchema      1.0.0
workerProtocol   1.0.0
spineTarget      4.2.43
targetTriple     x86_64-pc-windows-msvc
```

版本信息显示在“关于”、诊断页、安装包属性、日志头、更新 manifest 和 SBOM 中。

### F2S-REL-ARTIFACT-001：正式产物

```text
FlashToSpine_<version>_windows_x64_setup.exe
FlashToSpine_<version>_windows_x64_core.zip      # 经批准的便携/验收包
FlashToSpineLauncher_<version>_windows_x64.exe  # 签名原生入口；工程根副本命名为FlashToSpineLauncher.exe
FlashToSpineWorker_<version>_windows_x64.zip     # 可选，独立门禁
FlashToSpine_<version>_SBOM.cdx.json
FlashToSpine_<version>_SBOM.spdx.json
FlashToSpine_<version>_MODEL_BOM.json
THIRD_PARTY_NOTICES.txt
checksums.sha256
release-manifest.json
release-manifest.sig
release-notes_zh-CN.md
```

是否公开 portable zip 由商业发布策略决定；干净机验收可使用该包，但不能把无签名内部 zip 伪装成正式发行物。

## 18.4 工程根目录双击入口

### F2S-INSTALL-ENTRY-001：固定入口文件

开发完成后，工程主目录必须存在可直接双击的原生入口：

```text
E:\workwork\flash\flash_to_spine\FlashToSpineLauncher.exe
```

它是最小 Rust/Win32 启动器，不是安装器、构建脚本或在线下载器。Stable/商业入口必须有预期发布主体的 Authenticode 签名；内部开发签名必须使用不同产品身份、可见的“Developer Build”文件描述/窗口标记和更新通道，不能冒充商业入口。

启动器通过 `GetModuleFileNameW` 取得自身绝对路径并以 Windows Path API 派生工程根，不经 batch、PowerShell、环境变量展开或 shell 重解析。双击行为：

1. 优先查找自身根下 `dist\FlashToSpine.exe`，canonicalize 后确认仍在 `dist`、不是 reparse 越界，并验证产品 identity；商业模式还必须验证同一发布主体签名。
2. 若开发产物不存在，可读取本产品写入的 HKCU per-user 安装记录；记录包含 product ID、规范化安装根和主 EXE 相对路径，启动前重新 canonicalize 并验证签名/产品/架构。
3. 两者均不存在或验证失败时，使用原生 `TaskDialogIndirect`/等价窗口显示中文错误码、构建说明和日志目录；不依赖控制台保持可见。
4. 不自动执行 `npm install`、`cargo build`、下载依赖或请求管理员权限。
5. 使用参数数组/CreateProcessW，不接收和拼接网络、项目内容或 AI 输出为命令。
6. 启动器自身只读配置、验签、启动和健康握手，不加载项目、模型、WebView 或第三方插件。

项目根不再以 `FlashToSpine.cmd` 作为 P0 入口；可选 shell shim 只能用于开发者个人便利，不进入验收、安装包、快捷方式或 stable 产物，也不得承载诊断逻辑。这样路径中的 `%`、`!`、`^`、`&`、括号和 Unicode 不会经过 `cmd.exe` 二次解析。

启动请求被 Windows 接受不能证明应用健康。入口使用以下原生协议：

```text
Launcher validates target identity
  → creates current-user-only random named pipe + 256-bit nonce
  → CreateProcessW(target, structured args, explicit working directory)
  → waits at most 15 seconds for matching nonce and HealthReport schema
  → returns 0 only for healthy; otherwise terminates/observes child per policy and shows native diagnostic
```

只有主程序完成以下检查并通过 pipe 回传匹配 nonce 后，启动器才返回 0：

1. Rust Core 完成初始化且 panic handler/log sink 可用；
2. Tauri 主窗口和 WebView 完成加载；
3. 前端成功执行一次 Rust IPC `health.ping`；
4. 本机设置目录可读写或明确进入只读诊断模式；
5. 未发生签名、资源完整性或 schema 致命错误。

超时、子进程提前退出、nonce 不匹配、target签名/identity不符或任一检查失败都返回非零稳定错误码（例如 `F2S-LAUNCH-HEALTH-001`）并显示日志位置。健康确认只证明应用壳和核心可工作，不把 GPU、Worker 或 Spine 缺失视为启动失败。

### F2S-INSTALL-ENTRY-002：发布态入口

安装目录根部提供已签名的：

```text
FlashToSpine.exe
```

NSIS 创建开始菜单入口，桌面快捷方式由用户选择。快捷方式直接指向签名主程序，不经 PowerShell、脚本宿主或临时目录。

### F2S-INSTALL-ENTRY-003：入口诊断

双击后若启动失败，用户必须得到下列之一：

- 应用内诊断页；
- 保持可见的入口错误窗口；
- 原生错误对话框，包含稳定错误码和日志目录。

严禁只有“一闪而过”的控制台。入口 smoke test 覆盖中文、空格、括号、`&`、`^`、`!`、`%`、超长路径、非 C 盘、只读工程根和普通用户权限，并验证任何路径都没有 shell 命令重解析。

## 18.5 NSIS 安装设计

### F2S-INSTALL-NSIS-001：per-user 安装

默认目标：

```text
%LOCALAPPDATA%\Programs\FlashToSpine\
```

默认不需要管理员权限，不写 `Program Files`，不安装系统服务、驱动、浏览器扩展或全局 Python。安装内容使用 NSIS 支持的 zlib 压缩配置；NSIS 及其随附组件仍须通过 `F2S-DOC-SEC-001` 的许可证政策和 SBOM 门禁。

安装器必须：

- 显示发布主体、版本、安装路径和许可证；
- 验证系统架构和最低 Windows 版本；
- 检测 Evergreen WebView2，不悄悄下载未知 bootstrapper；
- 允许用户取消且清理未提交安装 staging；
- 安装主程序、资源、notice、卸载器和签名更新配置；
- 不捆绑 Spine、Spine Runtime 或激活信息；
- 不扫描用户全部磁盘查找项目或 Spine；
- 安装结束提供“立即启动”但不自动新建项目。

### F2S-INSTALL-WEBVIEW-001：WebView2

Core 使用系统 Evergreen WebView2。安装器检测缺失时：

- 离线安装包显示明确缺失项和官方获取方式；
- 企业部署可预装 Microsoft 官方离线 Runtime；
- 本产品包默认不内嵌未经许可证和哈希审计的 bootstrapper；
- 启动器再次检测并给出稳定错误码，不能崩溃或白屏。

### F2S-INSTALL-WORKER-001：可选 Worker

Python Worker 与 Core 解耦，安装状态以签名 capability manifest 决定。商业 D2 处理只允许项目发布方构建的自包含 Runtime Pack：精确 Python patch、wheel/model/native DLL hash、`license-policy.snapshot`、独立 SBOM/Model BOM、产品签名和协议版本全部固定。

用户自备 Python、系统 PATH、venv/conda、user-site、`PYTHONPATH` 或未签名解释器不得进入商业 Provider 路径，也不能满足“完整功能”。它们最多用于显式 developer/`policy_only` 模式处理 D0 合成夹具，UI 持续显示 `UNVERIFIED`，不能接收 D2。

Core 只以 `F2S-SEC-SANDBOX-001/windows-appcontainer-v1` 启动受支持 Worker Pack；启动前验证 pack签名/hash/manifest，固定 DLL 和模块搜索路径并禁用 site/user-site。`F2S-TST-RGPU-005` / `F2S-TST-E2E-SEC-005` 任一文件、网络、代理、子进程或 breakaway 探针成功时，发布流水线必须完全移除 Worker Pack，Core capability 报告为不可用；不得仅隐藏按钮或使用 waiver。

## 18.6 首次启动

### F2S-INSTALL-FIRSTSTART-001：启动检查

首次启动按可取消步骤执行：

1. 校验应用资源和可选 Worker manifest。
2. 检查可写的本机配置、日志和缓存目录。
3. 显示隐私基线：默认无网络、无遥测、无公共云。
4. 检测本地 GPU，仅形成能力报告，不上传。
5. 允许用户稍后选择 Spine `Spine.com`；不自动搜集激活信息。
6. 若用户配置 Spine，验证精确版本 4.2.43。
7. 远程 GPU 保持关闭，只有用户主动添加自托管 HTTPS endpoint 才启用。

任何可选检查失败都不能阻止基础项目打开，除非项目本身损坏或安全验证失败。

## 18.7 Spine 本地集成安装边界

### F2S-REL-SPINE-001：只发现、不分发

- 设置页由用户浏览选择 `Spine.com`。
- 应用保存本机路径和能力探测结果，不把绝对路径写入可分享项目。
- 只接受精确 4.2.43；不使用 `latest`、`4.2.xx` 或 beta。
- 用户负责 Professional/Enterprise 授权；应用不代为激活。
- 卸载本产品不触碰 Spine 安装和项目。
- 发布包和更新差分都扫描 Spine Runtime、Editor 和激活信息。

## 18.8 文件关联与项目入口

### F2S-INSTALL-ASSOC-001：项目目录与轻量入口

权威项目是后缀为 `.f2sproj` 的目录，而不是单一容器文件。Windows 对目录文件关联支持有限，因此 V1 采用以下组合：

- 应用内“打开项目目录”为 P0；
- 最近项目列表保存本机路径，不嵌入项目数据；
- 可选 `.f2slink` 小型 JSON 入口文件为 P1，只记录 schema、project ID 和相对/用户确认路径，不包含任何素材；
- `.f2slink` 关联到 `FlashToSpine.exe --open-link <path>`；
- 打开 link 时重新校验路径、项目 ID 和权限，禁止 URL、UNC 自动跳转和路径逃逸。

项目移动后 link 失效时提示重新定位，不进行全盘扫描。

## 18.9 更新设计

### F2S-REL-UPDATE-001：签名更新

更新是用户可控能力：

- 默认不上传任何数据；更新请求只包含产品版本、平台和发布通道，不含项目标识、机器名或素材。
- 首次启用在线检查前明确告知访问的官方更新域名。
- 更新 manifest 与 payload 均验证独立发布签名和 SHA-256；HTTPS 不能替代签名。
- 签名公钥编译进 Core，密钥轮换采用双签名过渡并留 ADR。
- 签名、版本、架构、大小或哈希不符立即阻断，无“仍然安装”按钮。
- 下载进入受限 staging，验证成功后交给安装器；不从临时目录直接运行未知文件。
- 客户端持久化最近接受的 `securityEpoch/sequence/version/manifestHash`；自动/离线更新均拒绝更低epoch、重复sequence不同hash、低于`minimumSafeVersion`或revoked key/artifact。
- key失陷、撤销和恢复根流程只引用 `F2S-SEC-SIGN-001`；普通更新签名key不能自行解除自己的撤销。

完全离线用户可手工导入已签名更新包，走同一验证路径。

### F2S-REL-UPDATE-SEC-001：反回滚与撤销

内容信任状态是单调的：`securityEpoch` 只可由离线 recovery root 授权提升；同一 epoch 内 `sequence` 严格增加。客户端验签顺序固定为 recovery/key-transition → release manifest → artifact hash/size → Authenticode/产品 identity →安装条件。任何一步 `NOT_RUN` 都不能继续安装。

最近可信状态以 DPAPI CurrentUser 保护并在 HKCU 与应用数据目录保存互相校验的 hash；任一副本缺失、回退或不一致时进入 trust recovery，而不是把状态重置为0。该控制保护本产品更新/离线导入路径，不宣称能阻止拥有本机控制权的用户绕过应用、手工运行历史安装器；已知不安全旧包仍通过撤销清单、`minimumSafeVersion`、发行说明和支持政策阻断。

- 功能性回退仅允许到签名、未撤销且 `version >= minimumSafeVersion` 的包，并要求项目使用兼容迁移快照；
- 已知漏洞版本、撤销 key/artifact、低 security epoch 不得因“用户确认”重新启用；
- 若本机信任状态损坏，进入只读恢复并要求导入由 recovery root 签名的完整trust bundle，不信任网络返回的裸公钥；
- manifest 过期时不得静默继续；离线用户可导入带有效期/撤销快照的签名bundle；
- key compromise 时冻结feed、吊销、minimum-safe-version和用户公告遵循14号PSIRT SLA。

测试必须覆盖旧manifest重放、同sequence换payload、降级包、撤销key、撤销artifact、过期manifest、错误channel/arch、合法双签轮换和recovery-root切epoch。

### F2S-REL-CHANNEL-001：发布通道

- `stable`：唯一面向商业用户的默认通道。
- `preview`：内部/明确加入测试者，使用不同 feed 和醒目标识。
- 开发构建不能读取 stable 签名密钥。
- 不允许从 stable 自动降级到 preview，也不允许跨架构更新。

### F2S-REL-UPDATE-002：更新事务

```text
check manifest
  → verify recovery/key-transition, manifest signature, epoch/sequence, expiry and revocations
  → user confirms version/notes/size
  → download/import to staging
  → verify payload signature + hash
  → ensure app/project jobs are stopped
  → installer creates rollback registration
  → install new version
  → first-launch smoke + config migration
  → mark update healthy
```

应用升级不批量迁移项目；项目迁移遵循 `F2S-DOC-RECOVERY-001`，在用户打开具体项目时发生。

### F2S-REL-ROLLBACK-001：回退

- 安装器保留上一已签名版本的回退信息或允许重新安装已签名旧包。
- 上一版本只有在未被撤销且不低于当前签名manifest的 `minimumSafeVersion` 时才可作为回退目标；否则显示安全阻断和替代修复版本。
- 新版启动失败时显示恢复入口，不自动循环重装。
- 二进制回退不等于项目 schema 降级；已经迁移的项目必须从迁移前快照打开。
- 更新健康标记只有在主窗口可用、项目只读 smoke 和核心资源校验通过后写入。

## 18.10 卸载与重装

### F2S-INSTALL-UNINSTALL-001：用户数据保护

卸载器默认只删除：

- 安装目录内本产品二进制和资源；
- 本产品开始菜单/桌面快捷方式；
- 本产品文件关联和更新注册；
- 明确属于本版本的临时更新文件。

默认保留：

- 所有 `.f2sproj` 项目；
- 用户导出 PSD/PNG/JSON；
- 用户自行安装的 Spine 和 Python；
- 可恢复设置/日志，除非用户单独勾选清理。

“删除设置与缓存”不得包含用户项目。卸载前后输出动作摘要；失败时保留日志和重试建议。

## 18.11 签名、哈希与发布来源

### F2S-REL-SIGN-001：签名链

- Installer、主 EXE、关键 DLL/sidecar 和更新 manifest 必须签名。
- Windows 二进制使用 Authenticode 和可信时间戳。
- 更新与发布 manifest 使用与 TLS/Authenticode 独立的 Ed25519 内容签名；算法或key变更须走ADR和双签/恢复根流程。
- 发布私钥保存在 HSM/受保护签名服务，不进入仓库或普通 CI 变量。
- `checksums.sha256` 由签名发布作业生成，但校验和本身不构成真实性；其hash与SBOM/NOTICE/Model BOM/所有zip、installer和Worker Pack一起进入签名 `release-manifest.json`。

`release-manifest.json` 使用固定 canonical JSON 序列化，并至少包含：

```json
{
  "schemaVersion": "1.0.0",
  "productId": "com.publisher.flashtospine",
  "channel": "stable",
  "version": "1.0.0",
  "buildNumber": "100",
  "securityEpoch": "1",
  "sequence": "100",
  "signingAttemptId": "01H00000000000000000000000",
  "minimumSafeVersion": "1.0.0",
  "publishedAt": "...Z",
  "expiresAt": "...Z",
  "keyId": "release-content-2026-a",
  "artifacts": [{
```

#### 不可变 waiver 流

每个 `waiverId=F2S-REL-WVR-<ULID>` 拥有独立 append-only stream。权威事实只有：

- `WaiverGrantRecordV1`：`recordKind=grant`、`sequence="1"`、`prevRecordArtifactHash=null`、`prevRecordHash=null`；
- `WaiverRevokeRecordV1`：`recordKind=revoke`、`sequence="2"`、`prevRecordArtifactHash=grantRecordArtifactHash`、`prevRecordHash=grantRecordHash`，引用精确 active grant；V1 撤销后重新申请必须新建 waiverId；
- `WaiverRegistrySnapshotV1`：固定`schemaVersion="1.0.0"`、`hashAlgorithmId="f2s-waiver-registry-sha256-jcs-v1"`、`registryId="f2s-waiver-registry-v1"`，以及registrySequence、previousRegistryArtifactHash/previousRegistryHash、projectRevision、currentActorRegistrySequence/currentActorRegistryArtifactHash/currentActorRegistryHash、排序唯一`entries[{waiverId,streamId,sequence,headArtifactHash,headRecordHash}]`和registryHash。`registryHash=SHA256("f2s-waiver-registry-v1\0" || JCS(snapshot without registryHash))`；完整JCS bytes普通SHA-256为registry artifact hash。实现必须提交`waiver-registry-v1.schema.json`及Rust/TS golden，domain hash绝不能当CAS地址。

Registry genesis固定sequence=0、previous双null、entries=[]并指actor genesis。后续sequence+1且previous双指current；transition类型只能是：新增grant entry、推进一个waiver entry，或只把actor registry root推进到其合法next snapshot，三者不可混合。禁止删除/倒退/缺hash/改streamId/跳record。RevisionManifest固定WaiverRegistry三元组并间接固定ActorRegistry。grant/revoke/actor update/Approve/ClockRecovery/signer均CAS current waiver三元组；重放先沿waiver snapshot previous双hash，再验证actor链和entry record双链。

共同 envelope 固定字段为 `schemaVersion="1.0.0"`、`hashAlgorithmId="f2s-release-waiver-record-sha256-jcs-v1"`、recordId/waiverId/streamId/recordKind、`commandKind=grantWaiver|revokeWaiver`、actorId/commandId/idempotencyKey/canonicalPayloadHash、occurredAt UTC、committedProjectRevision、sequence/prevRecordArtifactHash/prevRecordHash/recordHash；`recordKind=grant|revoke`必须分别与两个closed-enum commandKind一一匹配，错配、缺tag或未知tag拒绝。两个command payload是closed tagged union，禁止占位字段或额外键：`GrantWaiverCommandPayloadV1={commandKind:"grantWaiver",waiverId,streamId,proposalArtifactHash,proposalHash,attestationRefs,grantedAt,evidenceRefs}`，其中`attestationRefs`恰含按role/actorId排序唯一的product/quality/release三项`{role,actorId,attestationArtifactHash,attestationHash}`，`evidenceRefs`按evidenceId/contentHash排序唯一；`RevokeWaiverCommandPayloadV1={commandKind:"revokeWaiver",waiverId,streamId,grantRecordId,grantRecordArtifactHash,grantRecordHash,reason,evidenceRefs}`，reason为trim后非空UTF-8/NFC，evidence排序规则相同。`canonicalPayloadHash=SHA256("f2s-waiver-command-payload-v1\0" || JCS(exact tagged variant))`；expected WaiverRegistry/stream/clock heads只存在command envelope的并发前置条件中，不进入上述payload，其他语义字段不得排除，commandKind明确属于哈希域。waiver record链只向项目级幂等重建器贡献首次`{actorId,idempotencyKey,commandKind,canonicalPayloadHash,resultKind="waiverRecord",resultRef}`；必须与全部append权威链汇总后查询。只有全项目首次项kind/payload/resultKind都相同才返回首次同类型结果；同key跨链、跨kind或同kind异payload均为different-payload conflict、零事件且不得返回另一结果类型，recordId/commandId全项目唯一。lowerCamelCase、UTF-8/NFC、可空字段显式null、整数为十进制字符串、数组按各自canonical key排序唯一；`recordHash=SHA256("f2s-release-waiver-record-v1\0" || JCS(record without recordHash))`，完整record bytes普通SHA-256作为recordArtifactHash存于registry/后继ref而不写入自身。

#### GovernedWaiverBindingV1 与动态验证

09号`GovernedWaiverBindingV1`是ApprovalRecord持久化的**静态绑定**：grant/head双hash、proposal/attestation refs、policy/RuleCatalog/RuleSet、prioritySliceHash、finding、target、DependencySnapshot、stable inputs/outcome、scope、clockEpoch、grantedAt/expiresAt及批准时观察到的priority/waiver/actor registry roots；它不含checkedAt或动态Clock/Actor evidence。18号verifier每次从current PriorityRegistry逐slice subject匹配，并产生记录current priority/waiver registry、checkpoint和ClockEvidence双hash的`WaiverValidationEvidenceV1`；该evidence还必须双引用同次`ActorRoleValidationEvidenceV1`，后者固定current ActorRegistry及trust/revocation bundle roots。历史evidence不能证明当前active。
