# 项目索引与计划治理

## 1. 文档目的

本文档是 Flash to Spine Production Assist 项目的计划入口，规定计划文件的权威关系、稳定 ID、变更流程、审查门禁和共同术语。它不替代各领域设计，而是确保产品、内容、UI、架构、实现、测试和发布计划始终指向同一组已确认需求。

项目已确认的开发基线为：闭源商业 Windows 桌面软件、本地单用户、仅接收用户提供的图片和动作文字、不调用第三方云端图像服务、允许连接用户自托管的私有远程 GPU，并固定输出 Rig IR、PSD/PNG 和 Spine 4.2.43 JSON。依据用户要求项目方在五项中选定 V1 首个集成目标，项目选择“仅 Spine Editor 4.2.43”；Unity、Godot、Cocos和自研 Runtime 集成均不进入 V1。

### 1.1 治理记录

| --- | --- | --- |

### 5.2 生命周期

1. ID 一经发布不得重新编号、重用或因文件重命名而改变。
2. 被废弃的 ID 保留原记录，标记 `deprecated` 并填写 `superseded_by`。
3. 每个 ID 只有一份 canonical owner 文档；其他文件只能引用。
4. 新需求必须先注册 ID，再进入设计、任务和测试。
5. 追踪链固定为：`GOAL → UC → FR/NFR/CON → ADR/CMP/IFC/ENT/SCR/ACT → AC → TST → DEV → EVD → Release`。
6. `canonical_for`只能列本文件拥有的exact ID或在本节注册的命名空间；不能用裸前缀夺取其他文件owner。

## 6. 文档状态与修订

| 状态 | 含义 | 可否指导实现 |
| --- | --- | --- |
| `draft` | 内容正在编写，可能存在未决事项 | 仅可用于探索和 Spike |
| `reviewed` | 已完成领域审阅，但尚未取得全局 approval overlay | 只可继续计划审阅、评分和证据整理；不得开始产品代码、打包或发布实现 |
| `approved` | 评分、追踪和合规门禁均通过 | 只可进入该计划层明确声明的下一道门；总体计划批准只允许编写/审查原子计划，原子计划也批准后才允许产品代码 |
| `superseded` | 已由新修订替代 | 不得用于新实现 |

每次实质性修改必须递增 `revision`，在整改记录中说明原因、受影响 ID、同步文件和验证证据。仅修正错别字可递增补丁号，不改变需求语义。

R3 两阶段终审采用一个受控例外以避免自引用：R3a 结果写回后，25 份最终候选的 frontmatter 统一为 `status: reviewed`，并指向预分配的 R3b score ID；随后冻结的 R3b 输入不得再改字节。只有 detached R3b 报告同时满足 manifest/audit 哈希匹配、25/25 分数通过、无 P0/P1、`noPostFreezeMutation=true` 和 overall verdict=`PASS` 时，该报告才作为这组精确 SHA-256 的 **approval overlay**，在不改写输入的前提下把它们视为本表 `approved`。该总体计划 overlay 的授权范围仅为创建、评分和冻结原子 devplan，不是产品代码授权；只有原子计划自己的最终 approval overlay 也为 PASS 后才可实施。任一文件、manifest、audit 或报告变化都会使 overlay 立即失效，状态回到 `reviewed/NO_GO`；不得拿 overlay 批准其他修订或越过下一道门。

## 7. 计划审查与统筹流程

1. 完成全部领域计划初稿，不对半成品做虚假高分评价。
2. 检查重复 ID、悬空引用、孤儿需求、术语漂移和依赖环。
3. 由独立审阅者以第三人称逐文件评分，并为扣分提供具体证据。
4. 低于 95 分、关键维度低于该项 80%、存在 P0/P1 或阻塞性 `TBD` 的文档均为失败。
5. 整改时先修改 canonical owner，再同步所有下游文件，禁止仅润色被扣分文件。
6. 所有总计划通过后拆分原子任务，再重复追踪、评分和统筹审查。
7. 技术能力只有在 Spike 或测试证据存在时才能标记 `verified`；文档写得完整不等于实现已经成立。

## 9. 共同术语

| 术语 | 定义 |
| --- | --- |
| Production Assist | AI 和规则辅助的生产工作台，用户始终能检查、修改和批准中间产物 |
| 母版 | 经用户批准、作为角色身份和可见像素真值的标准角色图 |
| StyleSpec | 比例、线条、色彩、光照、视角、分辨率和动画友好约束的结构化风格规范 |
| MotionSpec | 单个动作的阶段、方向、接触、时序、表现方式和约束 |
| AssetSpec | 从动作需求反推的素材清单，包含用途、视角、父骨、pivot、重叠和来源 |
| Rig IR | 与 Spine 解耦的内部骨骼、槽位、附件、网格、权重、动画和事件表示 |
| 私有远程 GPU | 用户控制、明确信任且不由第三方 SaaS 持有数据的计算端点 |
| 提示词包 | 工具根据 StyleSpec、MotionSpec 和 AssetSpec 生成的可复制文本，不包含生图调用 |
| 人工门禁 | 未经用户明确批准，流水线不得自动越过的生产节点 |

## 10. 全局禁止项

- 不得把“一张图”宣传为可无人工介入地生成任意重度动作游戏角色。
- 不得向第三方云发送原图、图层、项目、提示词或诊断数据。
- 不得用 AI 推断结果替代用户对关键姿势、武器接触和命中帧的决定。
- 不得因当前许可证白名单而捆绑 Spine Runtime；预览必须基于内部 Rig IR。
- 不得直接生成或修改 Spine 专有 `.spine` 工程文件。
- 不得把缓存数据库作为源素材的唯一存储，也不得因清缓存而删除项目源数据。
- 不得将模型代码许可证等同于模型权重、训练数据或字体许可证。
- 不得在未以 Spine Editor 4.2.43 实测的情况下宣称该目标版本兼容；不得将 4.2.43 的验证结论外推到其他 patch。
