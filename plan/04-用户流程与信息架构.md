# 用户流程与信息架构

## 1. 体验目标

产品应让用户始终知道五件事：

1. 当前正在编辑哪一个项目、对象和版本；
2. 哪些内容来自用户，哪些由规则或 AI 辅助产生；
3. 当前阶段还缺什么，为什么不能批准或导出；
4. 一次修改会使哪些下游结果失效；
5. 保存、取消、失败或崩溃后能够回到哪里。

完整 Production Assist 不是一次性向导。应用采用“有推荐顺序的非线性工作区”：首次制作由阶段导航引导，熟练用户可跳转到任意可访问页面，但审批依赖和失效状态始终可见。

## 2. 信息架构原则

### 2.1 对象优先

导航和保存围绕稳定对象展开，而不是围绕临时任务：

`Project → Character → Specs → Motions → AssetSpecs → Assets/Layers → Rig → Animations → Reviews → Exports`

任务、日志、诊断和版本都是上述对象的关联记录，不能替代对象本身。

### 2.2 单一真值与派生视图

- 母版和批准规格是角色身份真值；
- AssetSpec 是素材需求真值；
- Rig IR 是结构和动画真值；
- Rig IR、PSD、透明PNG、Spine JSON、atlas packing/input manifest、提示词包和报告是内置导出快照；`.atlas/.spine/.skel`是可选CLI-owned产物，只能由用户本地合法Professional CLI 4.2.43生成，不能与内置输出混作同一writer来源；
- 首页百分比、阶段勾选和“可导出”均由门禁状态推导，不能手工设置。

### 2.3 渐进披露

默认界面显示当前任务、主要诊断和下一步。高级模型参数、权重数值、曲线手柄、Provider 详情和导出兼容日志在需要时展开，但任何阻塞错误不得被折叠或只显示在日志中。

### 2.4 建议与命令分离

AI/规则建议以“候选”呈现；用户的接受、替换、批准是独立命令。建议生成完成不得自动触发批准、覆盖或导出。

## 4. 全局状态语言

### 4.1 工作流阶段 — `F2S-WFS-STAGE-001`

本文件是工作流阶段词汇的 canonical owner。项目阶段严格使用：

`draft → master_approved → content_approved → layers_approved → rig_approved → motion_review → gameplay_approved → export_ready → exported`

| 阶段 | 到达条件 |
| --- | --- |
| `draft` | 项目已创建，但母版尚未批准 |
| `master_approved` | 当前母版与 StyleSpec 身份基线已经显式批准 |
| `content_approved` | Character/Style/Weapon/Runtime、十个 MotionSpec、AssetSpec 与可生产 PromptPack 已通过内容门 |
| `layers_approved` | 当前最小分层、隐藏区域与重组差异已经批准 |
| `rig_approved` | 当前骨架、slot、mesh、weight、constraint 与 socket 已批准 |
| `motion_review` | 十个动作均具备可审核工作版本，正在逐动作审核 |
| `gameplay_approved` | 十个动作关键姿势及三个攻击命中帧/接触均已批准 |
| `export_ready` | 当前版本 preflight 无阻塞错误 |
| `exported` | 当前 revision 的正式导出事务已提交 |

所有“已批准/门通过”条件均要求09号同一target的当前DependencySnapshot、认证PrioritySlice、stable GateOutcome与DecisionPolicy匹配，并存在由ApprovalStreamRegistry目标head重放出的active approve、精确ack及静态GovernedWaiverBinding；full PriorityRegistry root仅是transport observation。每个binding必须按current registry逐slice subject、WaiverRegistry、ActorRegistry中被引用credential的current角色/撤权状态、clockEpoch/checkpoint与effectiveNow重新验证，历史validation evidence不证明当前active。基础GateReport不读取这些决策。批准动作自身、无关project revision或无关registry entry不得令阶段自失效，实际依赖/policy/关联priority/finding、ack撤回、相关credential撤权或对应waiver撤回/到期才触发重算。

工作流阶段表达生产进度，不承载 `stale`、`blocked` 或 Spine 验证结果。是否可以继续必须同时读取下面四个正交维度；连同阶段共五维。

### 4.2 对象生命周期 — `F2S-WFS-LIFECYCLE-001`

| 状态 | 含义 | 可执行动作 |
| --- | --- | --- |
| `missing` | 必填对象不存在 | 创建、导入 |
| `draft` | 正在编辑，未提交审查 | 编辑、生成候选、提交 |
| `candidate` | 自动或外部产生的候选 | 比较、接受、拒绝 |
| `approved` | 当前对象版本经用户批准 | 继续下游、撤销批准 |
| `rejected` | 候选被明确拒绝并保留原因 | 查看历史、创建新候选 |
| `superseded` | 已由后续对象版本取代，只保留历史 | 查看、追溯，不再编辑或审批 |

`needs_review` 不是持久 Lifecycle 值。它是 GateReport 证明 candidate 已具备审查前置条件时生成的 UI `ReviewReadiness` 投影；显示该标签时底层 Lifecycle 仍为 `candidate`。该投影不得写回项目、审批记录或 Job 状态。

`approved`只由09号定义的active `approve`记录投影；退回、warning确认或已被后继`revoke`撤销的记录均不算批准。active判断绑定stable inputs/outcome而非完整transport report或全局revision；撤销当前批准/被引用ack、waiver失效后对象回到`candidate`（已superseded者除外），依赖它的下游新鲜度变为`stale`，相关release gate立即重算。

### 4.3 新鲜度 — `F2S-WFS-FRESHNESS-001`

| 状态 | 含义 |
| --- | --- |
| `current` | 对象完整DependencySnapshot中的上游ref、rules、policy、evidence与stable outcome仍匹配；无关project revision变化不影响 |
| `stale` | 上游变化使现有对象或审批需要重算/重审；不改变其生命周期历史 |

### 4.4 阻塞维度 — `F2S-WFS-BLOCKAGE-001`

| 状态 | 含义 |
| --- | --- |
| `ready` | 当前动作所需前置条件齐全 |
| `blocked` | 有一个或多个阻塞 issue；必须附 issue ID、原因和解除条件 |

Job 的 `requested`、`queued`、`running`、`cancel_requested`、`cancelled`、`failed`、`succeeded`、`interrupted` 是 `F2S-DOC-DOMAIN-001` 拥有的任务执行状态，不属于对象生命周期；任务失败不会把已批准对象改写为 `failed`，本文也不另造 `paused` 状态。`succeeded` 只表示未绑定的 `JobOutputArtifact` 已验证并持久化；UI 必须通过另一个带 expected revision 的“注册为候选”命令，成功后才显示对象生命周期 `candidate`。

### 4.5 导出验证维度 — `F2S-WFS-EXPORT-VERIFY-001`

| 状态 | 含义 |
| --- | --- |
| `not_exported` | 当前 revision 尚无正式导出 |
| `exported_unverified` | 已写出产物，但未在合法 Spine Editor/CLI 4.2.43 完成验证 |
| `verification_running` | 正在对已提交导出快照运行外部验证 |
| `spine_verified` | 4.2.43 实测及语义比较证据均通过 |
| `incompatible` | 版本、导入或语义比较失败；导出文件仍保留，不能显示为已验证 |

导出写入任务本身的失败使用 Job `failed`；它不会产生 `exported` 阶段。`spine_verified` 只属于本维度。

### 4.6 组合显示规则

UI 必须同时显示“阶段 / 生命周期 / 新鲜度 / 阻塞 / 导出验证”，例如：

`exported · approved · stale · blocked · exported_unverified`

项目可执行下一步的必要条件是：相应对象生命周期为 `approved`、新鲜度为 `current`、阻塞为 `ready`。颜色只是辅助；每种状态都有文本、图标和可访问标签。

### 4.7 影响预览

任何会使已批准内容失效的操作，在提交前显示：

- 变化对象和版本；
- 直接影响对象；
- 传递影响的动作、素材、审批和导出；
- 可自动重算与必须人工重做的项目；
- 预计任务与存储成本；
- “创建新分支版本”“应用并使下游失效”“取消”。

删除、替换母版、修改骨架父级、改变 FPS 和替换 WeaponSpec 均必须使用影响预览。

## 5. 端到端主流程

### 5.1 创建项目 — `F2S-UXF-PROJECT-001`

**入口**：首页“新建项目”或主目录双击入口。

1. 用户填写项目名与目录；
2. 系统检查路径、可写、磁盘空间和项目重名；
3. 页面显示固定目标 Spine Editor 4.2.43、本地单用户、默认离线；
4. 用户选择画布初值和本地计算策略；这些值之后可更改但会显示影响；
5. 系统原子创建项目骨架并进入概览；
6. 概览显示下一步“导入角色母版”，其他阶段可浏览但标记依赖。

**失败分支**：路径不可写、磁盘不足、同名非空目录时不得留下伪项目；用户可以更换目录。

### 5.2 导入并批准母版 — `F2S-UXF-MASTER-001`

1. 用户拖入 PNG、JPEG 或 WebP；若选择 PSD，系统无副作用拒绝并提示先转换为受支持图片；
2. 系统验证并显示文件信息、alpha、色彩、画布、裁切与重复提示；
3. 用户设置面向右、地面线、原点、角色包围盒和 `primary_weapon` 是否在图中；
4. 系统保留原图，只在工作副本上规范化；
5. 用户在原始/规范化/差异三视图检查；
6. 用户补充身份锚点或锁定区域；
7. 点击“提交母版审批”，页面列出检查结果；
8. 用户批准后产生 `master_approved`；拒绝则保留原因并返回编辑。

**门禁**：存在裁切、无可解析像素、方向未确认或身份锚点为空时不能批准。

### 5.3 规格、动作与提示词 — `F2S-UXF-CONTENT-001`

1. 系统从母版提出 StyleSpec/CharacterSpec 候选，候选字段标明置信度和来源；
2. 用户逐项修正并批准 StyleSpec；
3. 用户填写 WeaponSpec；具体武器未知时攻击提示词显示阻塞而非猜测；
4. 用户逐动作输入描述并导入零到多张关键姿势图；
5. 系统创建十个 MotionSpec，提示缺少的阶段、接触和参考；
6. 表现策略生成器提出 bone/mesh/attachment/sequence 等建议；
7. 系统汇总 AssetSpec，用户可以覆盖策略并填写原因；
8. 用户生成 PromptPack，预览未解决占位符；
9. PromptItem 可单项复制或整体导出；此步骤不联网；
10. 用户在外部工具生成图片后，回到提示词页或素材计划页回导并关联；无法提供生成信息时选择 `unknown_external` 并填写来源声明；
11. 用户批准内容规格、十个动作、素材计划和可生产 PromptPack，项目进入 `content_approved`，随后进入分层。

**关键体验**：动作、素材与提示词三栏可相互定位。选中 attack 的关键姿势时，右侧显示所需素材和对应 PromptItem；选中素材时，显示覆盖动作。

### 5.4 分层与补全 — `F2S-UXF-LAYER-001`

1. 用户选择自动建议或完全手工；自动不可用不阻断；
2. 左侧显示语义层树，中央显示母版/蒙版/重组，右侧显示图层属性；
3. 用户修蒙版、命名、近远侧、draw-order band、pivot 和重叠；
4. 隐藏区域可以由本地/私有计算生成候选，也可以从外部编辑器导回；
5. AI 新增像素以覆盖色显示，用户逐层接受/替换；
6. 系统实时或按需重组并运行差异诊断；
7. 透明洞、未覆盖可见像素和身份漂移定位到图层；
8. 用户完成全部必需层后提交审批；
9. 批准时保存层快照和差异证据，产生 `layers_approved`。

**中断**：长任务可以最小化到底部任务抽屉，用户继续处理其他不冲突对象；冲突编辑进入排队或要求取消任务。

### 5.5 Rig — `F2S-UXF-RIG-001`

1. 从批准层创建 starter Rig 候选；
2. 用户在骨架、槽位、网格/权重、约束四个模式间切换；
3. 选择画布对象会同步定位树、检查器和诊断；
4. 用户设置地面、pivot、握持和 socket；
5. 临时 Rig 以动作占位姿势试跑；
6. 问题列表按“Rig 修复/动作修复/新增素材”分类；
7. 新增素材问题返回 AssetPlan，生成补图任务；返回后保持当前上下文；
8. 用户关闭阻塞诊断并提交 Rig 审批；
9. 系统显示将解锁的十个动作和仍需补素材项；
10. 批准后产生 `rig_approved`。

### 5.6 动画制作 — `F2S-UXF-ANIM-001`

1. 动作选择器固定显示十个动作及状态；
2. 用户先选择模板/空动作/复制兼容轨道；
3. 关键姿势图片以参考层叠加，不自动绑定骨骼；
4. 用户编辑时间轴、曲线、deform、draw order、附件和事件；
5. 系统按 MotionSpec 显示阶段带和脚/武器接触标记；
6. 播放、逐帧、洋葱皮和差异比较用于检查；
7. 批量诊断可后台运行，问题跳转到证据帧；
8. 需要新增素材时创建 AssetSpec，相关动作的阻塞维度变为 `blocked`，其对象生命周期保持原值，其他动作可继续；
9. 动作完成后提交关键姿势审核。

### 5.7 动作与游戏性审核 — `F2S-UXF-REVIEW-001`

1. 审核页按动作显示关键姿势条、动作描述、参考图和当前渲染；
2. 用户逐个批准关键姿势，或在证据帧上批注并退回；影响预览列出DependencySnapshot的直接/传递边、stable outcome hash及哪些下游会stale；
3. 对 `attack_01`、`attack_02`、`attack_03`，用户设置/确认 anticipation、active/hit、recovery、weapon_contact；
4. 对 jump/hit/death，确认离地、impact/地面接触等内容门定义的关键事件；
5. 系统显示“AI/模板候选”与“用户已批准”差别；
6. 修改动作后，旧审批的生命周期历史保留，新鲜度进入 `stale` 并可并排比较；
7. 十个动作和三个攻击命中门全部通过后，项目进入 `gameplay_approved`。

### 5.8 导出与验证 — `F2S-UXF-EXPORT-001`

1. 导出页先显示 preflight，不直接显示“导出成功”按钮；
2. 错误按对象分组并可跳转修复；Issue 必须先按下述 `waiverPolicy` 判定，只有艺术建议型 `userAcknowledgement` 可逐项确认，逐需求priority均为P2的`governed`只能进入`F2S-REL-WAIVER-001`，`forbidden`永不显示接受动作；
3. 每次Stable签名作业先生成新的ULID `signingAttemptId`，最终release manifest必须写入同一值；作业重启或重试是新attempt。Preflight从current ReleaseGateCoverage枚举是否有任一required gate依赖governed waiver。若没有，跳过组织freshness步骤并把release manifest中的`OrganizationBundleFreshnessDeclarationV1` artifact/domain双hash固定为null；若有，Stable签名按钮保持禁用并强制进入“刷新组织bundle”；
4. “刷新组织bundle”要求用户显式选择并导入组织提供的current trust/revocation bundle文件，逐项显示sequence、fingerprint、issuedAt、签名、previous双head及相对项目current head的结果；仅接受合法next head或与current head字节一致的重新确认，旧序列、分叉、删撤销或复活anchor立即拒绝；
5. 刷新成功后，确认页必须原文提示：“离线V1无法发现组织已签发但未导入的更新，也不能对抗管理员/磁盘快照的全状态回滚；本声明不是global latest证明。”用户显式确认该限制后，只允许current ActorRegistry中的release角色credential签署`OrganizationBundleFreshnessDeclarationV1`。声明必须绑定本签名作业的`signingAttemptId`、ProjectId、release candidate、enrollment、ActorRegistry及trust/revocation bundle的sequence/fingerprint/issuedAt、ClockEvidence和`limitationAcknowledged=true`，且`declaredLatestAt=ClockEvidence.effectiveNow`。签署后显示从15:00开始的倒计时；最终manifest签名前重读current effectiveNow并强制`0ns <= currentEffectiveNow-declaredLatestAt <= 900s`。attempt变化、作业重启、超过900秒、时间差为负、candidate/roots/clock/credential变化都把声明置为过期并强制重新刷新、重新签署；旧attempt声明不得复用；
6. 用户选择Rig IR、透明PNG、最小可重建分层PSD、Spine JSON、`atlas-input-manifest.json`、PromptPack、清单与报告；高级PSD降级项必须进入报告。内置产物区明确标“由本工具生成”，不提供`.atlas/.spine/.skel`勾选项；
7. 项目内canonical export目录固定为版本化`exports/<export-id>`；用户可另选项目外publish root和“不覆盖/新版本目录”策略，但不能改变真值根；
8. 系统先在项目内同卷staging写入、校验并原子提交canonical snapshot；成功后才在外部publish root内创建同卷临时目录、逐文件hash校验并rename为版本目录。外部publish失败不回滚内部提交，完成页分别显示`export committed`与`publish failed/retryable`；
9. 如检测到Spine 4.2.43 Professional CLI，用户可选择运行验证/转换；只有该步骤可在独立`cli-generated`区创建`.atlas/.spine/.skel`，每项显示CLI operation、实际patch和输入/输出hash；
10. CLI缺失、版本不符或失败时不能显示已验证，用户只能保存内置开放文件；不存在fallback writer，也不得沿用旧CLI文件；
11. 完成页显示每个产物路径、hash、状态和“在文件夹中显示”；
12. 正式写入成功后工作流阶段记录 `exported`；导出验证维度独立记录 `exported_unverified`、`spine_verified` 或 `incompatible`。

## 6. 重要替代流程

### 6.1 完全手工模式

任何 AI/模型组件缺失、8GB 显存不足或用户主动禁用时：

- 规格、AssetSpec、提示词、蒙版、外部导入、Rig、动画、审核和导出仍可用；
- 自动建议入口显示“不可用原因”和安装/配置说明，不隐藏手工入口；
- 产品状态不因自动化比例低而降级。

### 6.2 私有远程 GPU — `F2S-UXF-RGPU-001`

1. 用户在设置页主动添加 Provider；
2. 系统显示只允许私有端点的说明；
3. 用户输入 URL、认证和证书信任；秘密写入 OS 安全存储；
4. 连接测试只发送健康检查，不上传项目图；
5. 首次实际任务显示精确对象、大小、处理类型和保留策略；
6. 用户批准后运行，标题栏显示“私有远程”；
7. 返回字节先进入 quarantine；完整schema/hash/安全/provenance校验并提升CAS后，Job才可成为`succeeded`且只得到未绑定`JobOutputArtifact`；用户执行带expected revision的RegisterCandidate成功后，才进入候选比较区；
8. 超时/断线时用户选择恢复、重试或取消；禁止转到公共云。

### 6.3 项目恢复 — `F2S-UXF-RECOVERY-001`

启动检测到异常关闭时，恢复页展示：

- 最后确认保存：时间、revision、完整性；
- 自动保存草稿：差异对象和完整性；
- 未完成任务 checkpoint：类型、阶段、是否可恢复；
- 只读打开、恢复草稿、放弃草稿但保留副本三个明确选项。

系统不得自动覆盖确认保存。损坏版本进入隔离区，用户可生成诊断包。

### 6.4 Spine 不存在或版本不符

- 设置和导出页显示检测结果；
- 产品仍能完成内部预览和文件导出；
- 4.2.43 未验证时导出验证维度保持 `exported_unverified`；
- 内置阶段只显示Spine JSON、透明PNG和atlas-input manifest；`.atlas/.spine/.skel`入口在合法4.2.43 Professional CLI不可用时隐藏或禁用并解释唯一生成者边界；
- 不弹出误导用户购买或自动下载的操作；
- 用户自行安装后可重新检测和验证已有导出快照。

### 6.5 外部编辑器往返

- 用户可导出单层 PNG 或最小可重建 PSD；PSD 在 V1 仅为输出，不能直接回导；
- 产品保存 round-trip manifest；
- 外部修改后只回导受支持的 PNG/WebP 图层图片并关联 round-trip manifest，比较画布、hash、pivot 和 alpha；
- 同尺寸兼容修改进入候选；
- 尺寸/原点变化先显示映射选择；
- 不兼容修改不会覆盖批准层。

### 6.6 组织信任与可选本地多签

普通制作和人工审批不要求账号或组织身份。只有用户希望使用`F2S-REL-WAIVER-001`时，设置页才提供下列显式流程：

1. 新项目先显示`ActorRegistry sequence=0 / 未配置组织能力`。此时唯一合法状态是enrollment、trust/revocation bundle、change-authorization引用全null、bundle sequence=0、actors为空；“申请延期”禁用，但普通制作和普通人工审批继续可用；
2. 用户导入`OrganizationTrustEnrollmentV1` capability pack。应用用签名application manifest钉住的publisher enrollment key验证签名，并逐项预览organizationId、productId/edition、组织离线根keyId/SPKI/算法、有效期和双hash。裸公钥、未知publisher、错产品/edition、过期、尚未生效、签名或根不匹配均不得进入下一步；
3. 用户再导入该enrollment允许的trust/revocation bundle、bootstrap authorization与initial actor/credential材料。提交前确认页把四组输入、目标`ProjectId`和“首次登记后不可替换”并列展示；只有单一sequence=1 `enrollOrganization` transition可原子固定enrollment、两条sequence=1 bundle roots、authorization和initial actors，任一验证/写入/CAS失败都回到原sequence=0 empty genesis，不显示部分成功；
4. 登记成功后，credential列表展示稳定actorId、credentialId、证书链、角色与current状态；私钥留在Windows安全存储、证书提供者或硬件中，不写入项目。能力就绪仍要求product、quality、release三个不同且current-valid的credentialId；未满足时显示缺少的角色/信任原因，用户仍可修复finding；
5. 每个governed finding生成一个只读WaiverProposal，固定finding、target、DependencySnapshot/PrioritySlice、policy、scope、期限和`proposalHash`；三份attestation必须分别签同一个proposalHash，错proposal、跨waiver复用、重复credential或角色不符均拒绝；
6. 撤权是追加ActorRegistry transition。确认页展示将立即失活的bindings/approvals；撤销credential、移除角色或trust失效后旧grant永不复活，重新授权必须使用新credentialId并重新grant/重审；
7. 同一`ProjectId`已有sequence>0 ActorRegistry时，导入任何不同enrollment必须在预检阶段拒绝，不能以“续期”“换组织”或“换edition”覆盖。到期或迁移状态只提供“克隆内容到新项目”向导：生成新`ProjectId`，不复制enrollment、ActorRegistry、grant/binding或approval，随后要求重新enroll和逐项重审；旧项目/证据只读保留。

设置页必须区分并保留以下可审计状态与提示：`未配置`显示“governed waiver不可用，普通制作不受影响”；`验证中`逐项显示publisher签名、product/edition、组织根和有效期；`验证失败`显示稳定原因码及失败字段，不用笼统“导入失败”；`已过期/尚未生效`显示失效时间、受影响grant/approval并禁用签名；`已登记`显示固定ProjectId、enrollment双hash、ActorRegistry sequence和三角色就绪度。失败或取消不能推进registry，也不能留下“已导入但未登记”的活跃能力。

所有bundle sequence/head旁都固定显示“离线项目链校验，不代表组织global latest”。V1只能在本地项目根、registry/manifest链和clock sidecar未被整体替换时识别分叉、删撤销、旧head替换和局部复制；它看不到未导入的组织更新，也不对抗管理员或磁盘快照的项目+sidecar+应用状态+bundle全量回滚。界面不得使用“已是全局最新”“防管理员回滚”或等价文案。

界面只能声称“已验证publisher-signed enrollment及三个不同有效签名credential的组织角色”，不得声称软件已经证明三个自然人。人员分离由组织签发和流程承担；零配置、无/过期/错edition enrollment或单人模式明确显示“governed waiver不可用”。

## 7. 页面间定位与上下文

### 7.1 稳定深链接

内部路由使用对象 ID：

`/project/{projectId}/animation/{motionId}?keyframeId={keyframeId}&tick={i64-decimal}&rateN={numerator}&rateD={denominator}&object={entityId}&issue={diagnosticId}`

路由不得依赖可变显示名，也不得使用 `f64` 秒。`keyframeId` 优先定位；tick/rate 是对象已删除时的确定性回退。诊断、日志、审批和报告的“前往对象”使用相同定位协议。

所有schema声明为`Ulid`的wire字段（包括导入文件、深链参数、proposal/enrollment/signing attempt等）统一使用严格grammar `^[0-7][0-9A-HJKMNP-TV-Z]{25}$`：恰好26位大写Crockford Base32，首字符只能`0–7`，字符集明确排除`I/L/O/U`。界面和Adapter不得trim、转大写、把`O→0`或`I/L→1`、接受小写/连字符/空白后再规范化；解码再编码必须与原wire逐字相等。错误长度、首字符`8–Z`、禁字符、别名或round-trip变化均在命令/导入/签名前fail closed，项目、registry、签名attempt和路由上下文零迁移。错误必须定位到JSON Pointer/表单字段和字符位置，显示期望grammar与具体原因，不能只报“ID无效”。

### 7.2 上下文保留

- 在 AssetPlan、Layer、Rig、Animation 之间跳转时保留当前动作、素材和证据帧；
- 从问题列表跳转后提供“返回问题”；
- 页面重开恢复上次工作布局，但不恢复未提交的危险确认对话；
- 不同项目的布局和最近选择分开保存。

### 7.3 面包屑

复杂工作台显示对象面包屑，不显示阶段路径：

`attack_02 / contact_pose / upper_arm_near_corrective / diagnostic {diagnosticId}`

点击任一级会改变选择而非关闭项目。

## 8. 任务、问题、日志与版本

### 8.1 任务抽屉

每个任务显示：

- 名称、对象、执行位置（本地 CPU/GPU/私有远程）；
- 阶段、真实进度或“不确定进度”；
- 已用/估算时间和资源；
- 可否取消；只有 StageDescriptor 声明兼容 checkpoint 时才显示“从检查点创建新任务”，它不会把当前 Job 变成未定义的 paused 状态；
- 未绑定 `JobOutputArtifact`、注册状态和日志摘要；
- 冲突对象锁；
- 完成后的“注册为候选并审阅”；若项目revision冲突则保留输出、刷新后重试注册，不得自动批准或把Job终态改写为失败。

命令幂等键按全项目`(actorId,idempotencyKey)`查询，不以`commandKind`分命名空间：同kind且同canonical payload只显示并返回首次结果；同key异payload必须显示payload conflict。尤其跨`commandKind`重用同一pair，即使字段碰巧相同也必须按payload conflict fail closed，零record、零registry/stream transition、零project revision迁移。用户可见诊断必须同时展示本次/首次commandKind、actor、idempotency key指纹、两个payload hash和“项目未发生更改”，并定位到触发按钮/任务；不得静默换key重试或把首次命令结果冒充本次操作。用户审阅后如确需执行，只能显式重新发起并生成新idempotencyKey。

### 8.2 问题中心

问题记录的canonical身份字段是 `findingId`，必须同时包含`severity`、`category`、经PriorityRegistry认证的`priorityBindings[{subjectKind,subjectId,priority,sourceArtifactHash}]`（如适用）、`waiverPolicy`、DependencySnapshot/outcome hash、证据对象和解除条件；UI可有本地列表row key，但不得另造`issueId`领域身份。severity 与每条需求/缺陷的 priority 是正交字段，禁止把P0/P1/P2写进severity、由finding自报priority或把多subject压成单一priority。严重度为：

当governedDefect规则首次命中且尚无DefectRecord时，问题中心显示“缺陷尚未登记”，只允许查看FindingDraft证据或执行“登记缺陷”。登记命令不能编辑priority，成功后必须重新评估，得到正式finding后才可进入修复/延期/审批流程；旧draft、登记冲突或失败不得显示waiver/approve按钮。

当时钟pointer、boot identity或epoch验证阻断且存在旧epochwaiver时，问题中心显示“授权时间无法继续证明”，提供只读查看和显式`ClockRecovery`入口。确认页列出原因、manifest/current checkpoint双heads、旧/新epoch、将永久失活的bindings及waiver-dependent approvals，并要求非空reason；取消后保持查看可用而approve/export/sign阻断。任何cross-boot都必须执行该流程后新grant/重审，UI不得提供“沿用旧waiver”开关。

- Error：阻断当前门或导出；
- Warning：需要处理；是否可确认由 `waiverPolicy` 决定，不能仅凭严重度接受；
- Info：建议和优化。

`stale` 不属于严重度；它只来自 `F2S-WFS-FRESHNESS-001`，可与任一 Error/Warning/Info 同时出现。问题筛选必须把 `severity` 与 `freshness` 分成不同字段。

`waiverPolicy` 只有三种：`forbidden`、`userAcknowledgement`、`governed`，由09号DecisionPolicy唯一计算。P0/P1及compatibility、scopeClaim、security、privacy、license、dataIntegrity、recovery、signature、updateRoot、workerIsolation、spineClaim、userData硬类别一律产生Error+`forbidden`；纯艺术Warning且无priority binding时可为`userAcknowledgement`；只有全部认证binding均为P2的非硬类别Warning可为`governed`并进入`F2S-REL-WAIVER-001`。ack/waiver只形成决策覆盖，不把finding显示为“已修复”或改写基础outcome。筛选支持severity、subjectKind与每条priority分别过滤。所有非修复决定都要求备注和审计；只有策略、类别、规则与stable outcome完全相同的`userAcknowledgement`可批量确认。governed waiver始终一finding一proposal、逐项查看与多签，禁止批量申请、默认勾选或一份attestation覆盖多个proposal。

### 8.3 版本历史

版本页以事件流显示保存、审批、失效、恢复、外部导入和导出。用户可以比较任意两个版本，但恢复旧版本默认创建新 revision，不能改写历史。

## 9. 审批交互规范

审批对话框必须包含：

1. 审批对象target revision/hash、当前project revision与stable outcome hash；
2. 输入来源、DependencySnapshot直接/传递边和变化摘要；
3. 自动检查的通过、警告、错误；
4. 人工检查清单；
5. 将解锁的阶段；
6. DecisionPolicy version/hash，以及后续什么依赖/policy/finding变化会使审批失效；
7. 每个ack record与P2 waiver的ID/hash、精确scope、proposalHash、三个actorId/credentialId/角色attestation、current ActorRegistry及ActorRoleValidationEvidence、到期/撤销状态；waiver必须标“延期/未修复”；未配置trust anchor、凭据不足/失效、错proposal或相关credential撤权时显示准确阻断原因；
8. “批准当前版本”“退回并备注”“取消”。

下列审批禁止批量一键跳过：

- 角色母版；
- 整套分层；
- Rig；
- 每个动作的关键姿势；
- 三个攻击的命中帧；
- 首次含警告的 Spine 导出。

系统可以提供“逐个处理后的批量确认摘要”，但不能以默认勾选代替检查。

## 10. 错误与空状态

### 10.1 错误消息结构

`发生了什么 → 哪个对象受到影响 → 数据是否安全 → 用户下一步 → 技术详情入口`

示例：

> 隐藏区域补全因显存预算不足而在写入候选前终止。已批准的 torso 图层未被修改；checkpoint 可恢复。可以调整任务配置后创建新任务，或保留现状进入手工编辑。错误 ID：F2S-WORKER-RESOURCE-001。

### 10.2 空状态

空状态必须解释价值和前置条件：

- 没有关键姿势图：允许仅用文字继续，但提示审核成本更高；
- 没有 GPU：显示完全手工和 CPU 路径；
- 没有 Spine：允许导出，解释验证状态；
- 没有诊断：区分“尚未运行”和“已运行无问题”；
- 没有 PromptItem：说明需要先批准 StyleSpec/MotionSpec，而不是只显示空白。
