# 17. 异常恢复、兼容与项目升级

## 17.1 目标

本文规定应用崩溃、掉电、磁盘异常、Worker/远程 GPU/Spine CLI 失败、项目版本变化和应用升级时的行为。核心目标是：任何故障都不能破坏最后一个已确认 revision，任何自动恢复都不能替用户选择未经批准的 AI 结果。

恢复设计遵守以下不变量：

1. `.f2sproj` 中的版本化 manifest、内容寻址 Artifact、Approval/Audit 事件块和 JobExecutionRecord 是项目事实源。
2. redb 只保存可重建索引、Job 投影和缓存，不保存不可重建的作业事实或源素材。
3. 所有持久化操作有 operation ID、输入 revision 和幂等键。
4. 所有会改变项目内容真相的多文件写入先进入 staging，验证通过后一次提交新的项目 revision；JobExecutionRecord 依11号独立hash链协议追加，不伪装成项目revision。
5. 上一个已确认 revision 永不被就地覆盖。
6. 恢复只恢复数据和候选状态，不自动恢复人工批准。
7. Spine 兼容固定在 Editor 4.2.43，不将其他补丁的成功外推为已兼容。
8. PriorityRegistry内requirements-change/rule-set-activation治理heads、ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1 refs、WaiverRegistry→ActorRegistry→OrganizationTrustEnrollmentV1→trust/revocation bundle间接roots及全项目ClockCheckpoint链都是恢复真值；存在governed-waiver release时，OrganizationBundleFreshnessDeclarationV1及其signingAttempt/ClockEvidence绑定也是发布恢复真值。恢复器不得从当前配置合成policy/algorithm registry proof、freshness declaration、enrollment、缺失head、签名proposal、credential或epoch边界。
9. 所有`Ulid`只接受`^[0-7][0-9A-HJKMNP-TV-Z]{25}$`的26位大写Crockford wire并须逐字roundtrip；恢复器不得把小写、I/L/O/U、alias、错长度或首字符8–Z规范化为“可用ID”。

## 17.2 项目持久化模型

### F2S-REC-PROJECT-001：权威布局引用

项目目录、项目外 Job sandbox、Approval/Audit CAS 事件、JobExecutionRecord 和 redb 投影的唯一布局由 `F2S-DOC-STORE-001` 定义。本文禁止复制目录树或另设真相源；恢复器必须按所打开项目声明的 Storage Schema 选择对应 reader。未知 major 只能只读，不得用本文示例猜测路径。

`project.json`提交后代表current revision；`clock-state.json`可领先但不分叉/落后。恢复从RevisionManifest三个top-level registry三元组及每份proposal/grant/binding固定的ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1 refs开始：PriorityRegistry内继续重放requirements-change/rule-set-activation双heads，WaiverRegistry内沿currentActorRegistry再重放ActorRegistry所锚定的OrganizationTrustEnrollmentV1、ActorRegistryChangeAuthorizationV1、ActorTrustBundle/ActorRevocationBundle双链、credential/certificate、WaiverProposalV1/attestation/ActorRoleValidationEvidence；随后重放全部defect/approval/waiver entry双heads、clock checkpoint/recovery与audit。release evidence另从final manifest双hash取回OrganizationBundleFreshnessDeclarationV1并核对同signingAttemptId、candidate、current roots、ClockEvidence及release credential签名。ActorRegistry sequence=0只接受enrollment/bundle/authorization及commandKind/actorId/commandId/idempotencyKey/canonicalPayloadHash/committedProjectRevision全null、actors为空的唯一empty genesis，并禁用governed waiver；sequence>0要求六个command字段全非null且enrollment双hash从首次enroll起固定。ClockRecoveryRecordV1必须保存`commandKind=clockRecovery`、canonicalPayloadHash与committedProjectRevision；其他append record同样保存其closed-enum tag。artifact寻址、domain认证；非法ULID、policy/algorithm registry正文或双hash错配、宽松签名解码、commandKind/metadata错配、单hash、部分bootstrap、同ProjectId换enrollment、错proposal、同credential伪三人、bundle回滚、旧attempt freshness、JSONL/redb active都不是真值。

### F2S-REC-COMMIT-001：提交协议

提交步骤、flush/rename 顺序、事件冻结点、journal Schema 和 commit point 只引用 `F2S-DOC-STORE-001` §8；本文不得维护第二份算法或状态枚举。恢复实现以 `F2S-TST-111` 绑定的真实 Windows/NTFS 证据解释 durable 边界，在该证据出现前原子能力保持 `UNVERIFIED`。

恢复器只做以下判定，不改变 canonical 顺序：

- current pointer 指向完整 manifest 且所有 CAS/事件引用有效：该 revision 可打开；
- 存在完整但 current 未引用的 revision：列为孤立候选，由用户选择恢复为新 revision，绝不按编号自动采用；
- pointer、manifest、CAS 或 journal 无法共同证明提交完成：保持最后可证明的 current，只读展示候选并隔离不确定输出；
- pointer 已提交但 redb 缺失/损坏：从权威记录重建投影，不能回滚已提交 revision；
- 任何修复写入都必须在用户选择后作为新的 operation 重新走11号提交端口。

### F2S-REC-JOURNAL-001：操作日志

恢复器按11号版本化 OperationJournal 和 JobExecutionRecord Schema 读取 `operationId/jobId`、base/intended revision、幂等键、序列/hash链、脱敏 staging/sandbox reference 和外部步骤证据。本文不增加 `prepared/revision_ready` 等私有枚举。未知 Schema/state、hash链断裂、sequence 跳号或 current pointer 不匹配时 fail closed，进入只读恢复。

Journal/Job 记录不存图片字节、完整提示词、令牌、Spine 激活信息或非必要绝对路径。恢复器不得修改既有不可变 transition；需要标记中断时，由 Rust 在取得写锁并重新验证后追加 canonical `interrupted` transition。

## 17.3 故障分类和错误码

### F2S-REC-ERROR-001：稳定错误分类

所有错误码由 `F2S-ERROR-CODE-REGISTRY-001` 唯一拥有并统一为 `F2S-<SUBSYSTEM>-<CATEGORY>-<NNN>`：三个段均为大写 ASCII，`NNN` 为同一 subsystem/category 下永不复用的三位编号。本文只投影恢复分类和处置，禁止再引入旧式通用错误前缀、自由文本前缀或改变已发布错误码语义。

| Registry exact code（只读投影） | 类别 | 示例 | 默认处置 |
|---|---|---|---|
| `F2S-STORE-IO-001` | 文件/磁盘 | 空间不足、权限、只读 | 停止写入，保留旧 revision |
| `F2S-STORE-DATA-001` | 数据 | Schema、哈希、引用损坏 | 隔离对象，进入恢复 UI |
| `F2S-WORKER-PROCESS-001` | Worker | 崩溃、协议、超时 | 终止请求，可安全重试 |
| `F2S-GPU-RESOURCE-001` | 本地 GPU | OOM、驱动、设备丢失 | 提出调整建议；用户批准后创建新Job |
| `F2S-RGPU-NETWORK-001` | 私有远端 | TLS、断线、响应非法 | 不公共云回退，保留本地状态 |
| `F2S-SPINE-COMPAT-001` | Spine | 缺失、版本不符、CLI 失败 | 保存未验证输出，不伪造通过 |
| `F2S-MIG-SCHEMA-001` | 迁移 | 不支持版本、迁移校验失败 | 只读打开或保持旧版 |
| `F2S-DOMAIN-STATE-001` | 领域状态 | lifecycle、Gate输入或decision前置不满足 | 不追加事件，定位对象/finding后重试 |
| `F2S-DOMAIN-CONFLICT-001` | 并发 | project revision、priority/approval/waiver registry三元组、requirements-change/rule-set-activation治理head、target/waiver stream双head、Actor/bundle root或clock checkpoint/recovery head CAS冲突 | 刷新全部冲突head并由用户安全重放，不覆盖 |
| `F2S-DOMAIN-POLICY-001` | 审批策略 | policy/hash、waiver scope/expiry或受信时钟不合法 | fail closed；重评、修正时钟或新grant，旧记录不改 |
| `F2S-SEC-PATH-001` | 安全 | 路径逃逸、签名失败 | 阻断且记录脱敏证据 |
| `F2S-APP-RUNTIME-001` | 应用 | panic、WebView、IPC | 恢复最后确认 revision |

错误实例包含稳定 code、operation ID、用户可行动建议、是否可重试、日志关联 ID 和安全脱敏的技术原因。不得只显示“未知错误”。

### F2S-REC-RETRY-001：重试规则

仅在操作满足幂等性时自动重试：

- 纯读取/分析：最多有限次数指数退避；
- Artifact 上传到用户自托管端点：要求 idempotency key；
- 本地 Worker 请求：以相同输入哈希重启，不复用半完成输出；
- 项目 revision commit：不自动重放未知状态事务，先执行恢复判定；
- Spine CLI：默认不自动重试，避免重复覆盖；用户确认后使用新 staging 目录重试；
- TLS、许可证、版本不符、Schema 和路径安全错误永不自动重试。

## 17.4 启动恢复流程

### F2S-REC-STARTUP-001：只读扫描

应用打开项目时先执行只读扫描：

1. 解析 `project.json` 并验证 schema/version。
2. 验证 current revision 文件和必要 Artifact 引用。
3. 验证GateReport/RuleCatalog+RuleFamilyMigration+RuleSet/RuleSetActivation/Dependency/Priority observation+slice/RequirementsPriorityBaseline/RequirementsChange/GovernanceApprovalEvidence/Defect/FindingDraft/ResolutionEvidence/DecisionPolicy/ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1、三top-level registries、PriorityRegistry内两个governance heads、Waiver→ActorRegistry→OrganizationTrustEnrollmentV1+ActorRegistryChangeAuthorizationV1+ActorTrustBundle/ActorRevocationBundle及全部defect/target/waiver streams、credential/certificate、WaiverProposalV1/attestation/ActorRole/ClockCheckpoint/Recovery evidence的可达性与双hash/previous/sequence；验证clock-state全epoch ancestry。ActorAttestationPolicy与SignatureAlgorithmRegistryV1 authoritative bytes及artifact/domain双hash必须与签名application manifest固定值一致；签名只接受ECDSA P-256/SHA-256/P1363-64/low-S、RSA-PSS SHA-256/MGF1-SHA256/salt32/trailer1/modulus≥2048及strict no-padding base64url。ActorRegistry seq0/seq>0 command字段nullability、commandId唯一、canonicalPayloadHash与committedProjectRevision必须合法，ClockRecoveryRecord也必须含后两字段。sequence=0 empty genesis缺enrollment是合法禁用态，sequence>0按publisher key、product/edition、organization root、expiry与同ProjectId固定性验证enrollment。按current slice、waiver entry、trust/revocation head、三个不同credential对同一proposalHash的签名、current actor roles、effectiveNow/epoch重验binding并产新evidence；policy/algorithm registry错配、无/过期/错edition enrollment、错proposal、同credential复用、bundle回滚或相关credential/SPKI/serial/anchor撤销/失效都fail closed，无关actor或bundle entry推进不得误失效。若发现使用governed waiver的release evidence，还须验证OrganizationBundleFreshnessDeclarationV1与final manifest同signingAttemptId，且声明产生至最终签名前为`0..900s`；旧attempt、作业重启复用、超15分钟或错candidate/root/clock/signature只使该release candidate不可签名/推广，不伪造成全局latest证明。
   同一步还必须逐字段验证ULID严格wire，以及每个append的commandKind与canonicalPayloadHash：tag必须属于closed enum并进入JCS/hash；同`(actorId,idempotencyKey)`跨kind不得被解释为另一结果类型。
4. 检查未完成 OperationJournal、JobExecutionRecord hash 链、staging 和临时指针。
5. 按 `F2S-DOC-STORE-001` §11.1 对账项目外 Job sandbox；只读验证目录边界、ACL、`sandbox.json`、job ID、输入hash和进程lease。
6. 检查 redb 是否可打开；失败则标记待重建。
7. 检查是否存在 current 未引用但结构完整的孤立 revision。
8. 生成恢复候选、sandbox处置建议和证据，不立即写项目或删除D2文件。

扫描发现异常时先以只读方式展示。用户决定恢复旧确认版本、采用完整候选、导出诊断或打开副本；不得在 UI 出现前自动覆盖。

### F2S-REC-SANDBOX-001：项目外 Job sandbox 对账

恢复器以项目 JobExecutionRecord 为一侧、11号 canonical sandbox root 为另一侧建立一对一清单。它不能根据目录时间或文件名猜测成功：

| 观察 | 恢复判定 |
|---|---|
| 非终态 Job 有合法 sandbox，但没有匹配的存活 Worker lease | 追加动作候选为 `interrupted`，`reasonCode=core_restart_or_worker_lost`；`out/` 先 quarantine，不注册 Candidate |
| 非终态 Job 的 sandbox 缺失/ACL或manifest损坏 | `interrupted` 候选，`reasonCode=sandbox_missing_or_invalid`；不自动重跑 |
| sandbox 存在但项目无对应 Job/hash链 | 视为 orphan；拒绝导入，显示来源/大小/最后活动时间并提供隔离或清理选择 |
| Job 已 `succeeded` 且输出CAS hash已由权威记录证明 | sandbox仅是可清理副本；清理失败不改变Job终态 |
| Job 已 `cancelled/failed/interrupted` | 任何输出都不是正式候选；仅可隔离供诊断或按用户选择清理 |
| sandbox输出完整但终态先后不可证明 | 保持 `interrupted`；重新校验后也只能由新命令注册为新Candidate，不能回写旧Job为succeeded |

只有 Rust Core 在取得项目写锁、重新验证 current/sequence/hash 后才能追加 `interrupted` transition。只读扫描本身不杀未知进程、不改 ACL、不删目录；用户确认清理前显示文件数、D2分类和逻辑删除/SSD残余风险。输入可由CAS重建时优先短保留；orphan 默认24小时后再次提示，最长保留策略不得超过安全配置的7日且不能越过用户pin/取证保留。

### F2S-REC-UI-001：恢复中心

恢复界面至少显示：

- 最后确认 revision、时间、审批状态和哈希；
- 最近 autosave/candidate revision；
- 未完成任务种类和最后阶段；
- 每个项目外 sandbox 的匹配 Job、进程lease、D2大小、对账结论和建议处置；
- 每个`F2S-SCH-010` PublishAttempt的append-only transition head、已登记外部root、临时目录或receipt对账；外部root不可达时追加非终态`recovery_pending`，重新定位后从该状态继续；rename后receipt完整时须重新hash并以expected revision/head追加committed transition，不复活terminal记录；
- 每个候选缺失/损坏项；
- “恢复为新 revision”“只读打开”“复制项目后修复”“放弃候选”选项；
- 操作预计影响的文件数量与不可逆风险；
- 诊断日志路径和脱敏导出按钮。

恢复未经批准的候选时，旧审批按依赖关系变为 `stale`，对象Lifecycle保持`candidate`（或恢复前的canonical值）；UI由ReviewReadiness派生`needs_review`。不得持久化`needs_review`或新增`unapproved`私有状态。

### F2S-REC-INDEX-001：redb 重建

redb 打不开、版本不符或校验失败时：

1. 关闭并保留损坏数据库副本到 quarantine；
2. 从current manifests/CAS、clock全epoch roots、ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1 refs、三top-level registry、PriorityRegistry内requirements-change/rule-set-activation双heads、Waiver→ActorRegistry→OrganizationTrustEnrollmentV1/change authorization/trust+revocation bundle链、credential/certificate、OrganizationBundleFreshnessDeclarationV1、全部entry/recovery/audit heads、journal与Job链重建；同时从**全部append权威链**重建`(actorId,idempotencyKey)→{commandKind,canonicalPayloadHash,resultKind,resultRef}`，仅同kind同payload返回首次同类型结果，同key跨kind或同kind异payload均different-payload且零transition。empty genesis及无waiver release保留各自null roots，重建器不得规范化非法ULID，不得从当前配置补commandKind/metadata、policy/algorithm registry proof、freshness declaration，也不得创建enrollment或bootstrap；
3. 在新数据库通过引用计数和查询 smoke test 后原子替换；
4. 重建失败不影响以只读模式打开权威 manifest。

索引重建不能删除 CAS。垃圾回收是独立、显式且可取消的维护任务。

## 17.5 各阶段恢复策略

### F2S-REC-IMPORT-001：图片与 PSD 导入

- 解码失败：不进入 CAS，保留源文件不动。
- 哈希计算中断：删除 temp，不生成引用。
- 多层导入部分失败：整次导入保持候选状态，用户可以剔除失败层后重新提交。
- 超大/恶意文件：进入安全错误，不自动重试。

### F2S-REC-AI-001：本地 Worker

- Worker 异常退出、协议 EOF 或 Core 无法证明终态：追加 canonical `interrupted`，并写 `reasonCode=worker_exit|protocol_eof|core_restart`；只有明确、完整且可归因的校验/模型错误才进入 `failed`。
- 通过校验的部分候选可保存在 staging，但必须明确标记 incomplete，不能自动提交。
- 重启 Worker 不是旧 Job 状态回跳；用户批准后创建新 Job ID，记录 `retryOf`、相同输入hash和新幂等键，不能依赖进程内状态。
- Worker 明确报告 OOM 时旧 Job 为 `failed(reasonCode=resource_oom)`；若进程直接消失且无法证明原因则为 `interrupted(reasonCode=worker_lost)`。系统只生成新的 JobRequest 建议，列出分辨率、tile、模型/CPU变化、预计时间与质量影响；用户批准后才创建新 Job，禁止原 Job 自动降质重跑。

### F2S-REC-RGPU-001：用户自托管远程 GPU

- 断网时保持本地项目不变，Job 进入 canonical `interrupted`，并在独立字段写 `reasonCode=remote_disconnect`、`providerStatus=unknown|cancelled|completed_unclaimed`；不得创建 `interrupted_remote` 等私有持久状态。
- 不自动改用公共云或未知端点。
- 若端点支持查询 idempotency key，则先查询再重试，避免重复作业。
- 响应下载到 staging，完成哈希/Schema/图片验证后才能进入候选。
- 远端状态永远不能单独证明本地提交成功。

### F2S-REC-EXPORT-001：导出

- PSD/PNG/JSON 全部先写同一 operation staging。
- 任一必要产物失败时，不更新最终 export manifest。
- 成功产物可以供诊断查看，但 UI 不显示为正式交付。
- 上一个成功 export 目录不可覆盖；重复导出使用 content-derived export ID 或新版本目录。

### F2S-REC-SPINE-001：Spine Professional CLI

失败分类：

| 情况 | reason/evidence code（不是Job状态） | 行为 |
|---|---|---|
| 未配置/不存在 | `cli_unavailable` | 保存基础产物，标 `exported_unverified` |
| 不是 4.2.43 | `cli_incompatible` | 禁止调用，提示选择正确版本 |
| 无合法授权/未激活 | `cli_license_required` | 不代为激活，保留未验证输出 |
| 非零退出码 | `cli_failed` | 保存脱敏输出摘要与 staging |
| 超时/取消 | `cli_interrupted` | 终止进程树，隔离临时 `.spine` |
| round-trip 差异 | `semantic_mismatch` | 阻断 verified 状态，生成结构差异报告 |

Spine 验证使用分层证据，不新增与04/09冲突的项目状态：

- `validation.l0/l1`：内部结构与语义；通过后 Candidate 仍是 `exported_unverified`；
- `validation.l2=cli_roundtrip_verified`：精确4.2.43转换后版本证明和语义比较通过，但尚不能单独授予最终状态；
- `validation.l3=editor_review_approved`：人工在4.2.43打开并核对关键动作/命中/视觉，审批绑定export hash；
- 只有 L0–L3 均为pass、全部使用特性为 `VERIFIED` 且审批未 stale，派生状态才是 `spine_verified`。

CLI 失败不会破坏 Rig IR，也不能让应用尝试直接修补 `.spine`；任一层失败或 `NOT_RUN` 时保持 `exported_unverified/incompatible` 并给出精确原因。

## 17.6 项目 schema 升级

### F2S-MIG-UPGRADE-SCHEMA-001：版本规则

每类数据独立版本：

- project manifest；
- Rig IR；
- StyleSpec/MotionSpec/AssetSpec/PromptPack；
- Approval record；
- Worker protocol；
- Export profile/compatibility manifest；
- redb index schema。

Schema 使用 SemVer 解释：

- patch：语义不变的兼容修复；
- minor：旧 reader 可忽略的可选增加，或有明确默认值的迁移；
- major：不兼容变化，必须显式迁移。

应用不得仅比较一个全局版本就假定所有子 schema 兼容。

### F2S-MIG-UPGRADE-PROJECT-001：迁移算法

```text
open source project read-only
  → detect every schema version
  → choose explicit migration path
  → copy current pointer + manifests to migration snapshot
  → migrate into sibling staging directory
  → validate references/domain invariants
  → render/check critical previews
  → compare approval dependency hashes
  → commit new revision with migration report
```

迁移必须是链式的、可重复的，例如 `1.0 → 1.1 → 2.0`，不允许每个旧版本直接散乱跳到最新。每个迁移函数包含 fixture、property test、失败注入和反复执行幂等测试。

涉及人工决策的历史fixture还必须显式迁移RuleCatalog/RuleSetActivation、RequirementsPriorityBaseline/RequirementsChange/GovernanceApprovalEvidence/ResolutionEvidence与PriorityRegistry内两个governance heads，并保留历史最严priority floor。缺任一head、RuleFamilyMigration、ReleaseWaiverPolicy/ActorAttestationPolicy/SignatureAlgorithmRegistryV1正文或双hash、OrganizationTrustEnrollmentV1、ActorTrustBundle/ActorRevocationBundle、credential/WaiverProposalV1/attestation role evidence或Clock跨epoch successor证明时，迁移只能保留历史并将相关binding/approval置inactive；严禁从当前配置补policy/algorithm registry proof或为旧ProjectId合成/替换enrollment、bundle head、proposal、priority floor或active批准。ActorRegistry sequence>0、ClockRecoveryRecord或其他append缺commandKind/canonicalPayloadHash/committedProjectRevision时不得合成首次结果或active transition；非法ULID不得case/alias归一化、补位或重新编码后冒充原ID。无enrollment或enrollment过期的项目保留艺术内容但禁用governed waiver；若要启用，只能克隆到新ProjectId、重新enroll/reapprove，旧grant不迁移。依赖governed waiver但缺OrganizationBundleFreshnessDeclarationV1或signingAttempt/time-window证明的旧release只保留历史且不得promotion/re-sign；迁移器不得合成声明，无waiver release继续保持声明双hash为null。

### F2S-MIG-UPGRADE-BACKUP-001：迁移备份与回退

- 迁移开始前创建只读快照或同盘复制的最低必要 metadata；CAS 不重复复制，但冻结引用集合。
- 新版本成功打开并经过用户确认前，不删除旧 manifests。
- 回退是重新把旧项目副本作为独立项目打开，不在原目录执行破坏性“降级”。
- 新版写入过的项目默认不保证可由旧版打开；UI 必须在首次升级前告知。
- 迁移报告记录源/目标 schema、步骤、警告、旧新哈希和失效审批。

### F2S-MIG-UPGRADE-UNKNOWN-001：未来版本

当项目 schema 主版本高于当前应用：

- 禁止写入；
- 尽可能显示只读元数据，但不得猜测未知字段语义；
- 提示安装兼容新版本；
- 允许复制原文件和导出诊断，不允许“强制打开并保存”。

## 17.7 Spine 版本兼容与升级

### F2S-MIG-UPGRADE-SPINE-001：固定兼容矩阵

V1 唯一正式矩阵：

| 项目 | 固定值 | 漂移处理 |
|---|---|---|
| Spine Editor | 4.2.43 | 其他版本不标 verified |
| Spine data major/minor | 4.2 | 4.1/4.3 均不兼容 |
| JSON `skeleton.spine` | 4.2.43 | Adapter 阻断错误值 |
| CLI 默认版本探测 | `--version`，结果必须为 4.2.43 | 默认离线，不隐式切换/下载 |
| CLI 显式切版本 | 用户单独许可后才可 `--update 4.2.43 --version` | 禁止 latest/4.2.xx/beta；随后重新 probe |
| Runtime | 不捆绑、不验证引擎 | 需新 ADR 才扩展 |

官方说明 Editor/Runtime 的 major/minor 必须匹配；Spine 项目文件可由新版本打开旧版本，但一旦由新版本保存，通常不能由旧版本重新打开。因此所有 CLI 临时项目只在 staging 生成，不能对用户已有 `.spine` 原件就地处理。

官方依据：

- [Spine Versioning](https://esotericsoftware.com/spine-versioning)
- [Spine Changelog](https://esotericsoftware.com/spine-changelog)
- [Spine CLI](https://esotericsoftware.com/spine-command-line-interface)

### F2S-MIG-UPGRADE-SPINE-002：未来 patch/minor 升级

更换目标版本必须：

1. 新建并批准 ADR，不能修改 4.2.43 记录以抹去历史。
2. 固定新的精确 Editor patch 与 Runtime 边界。
3. 新建 Adapter 或显式 Adapter version。
4. 对所有 golden fixtures 全量导出和 round-trip。
5. 对用户项目只新增导出 profile，不自动覆盖旧 profile。
6. 保留 4.2.43 导出器，直到正式迁移政策允许废弃。
7. 更新许可证确认、SBOM、文档、错误文案和原子任务。

升级结果没有实测证据前状态只能是 `experimental`。

## 17.8 应用升级与回退

### F2S-MIG-UPGRADE-APP-001：安装新版本

- 安装器不能在应用运行时覆盖正在使用的二进制。
- 升级先验证发布签名、版本单调性、目标架构和最低系统要求。
- 首次启动先扫描项目版本，不批量后台迁移所有项目。
- 每个项目在用户打开时单独迁移，并允许取消。
- 应用配置迁移与项目迁移分离；配置损坏时恢复安全默认值，不重置项目。

### F2S-MIG-UPGRADE-APP-002：应用回退

二进制回退由安装器保留的上一版本或重新安装已签名旧包完成。项目若已经升到旧应用不认识的 schema，只能使用迁移前快照；不执行未知字段丢弃式降级。

## 17.9 Autosave、撤销与审批失效

### F2S-REC-AUTOSAVE-001：Autosave

- Autosave 保存候选 revision，不替代用户确认保存。
- 以编辑命令边界和节流策略触发，不能在每个指针移动像素时写盘。
- 只写新增 Artifact 和 manifest，不复制整个项目。
- 保留数量和时间窗口可配置；清理前检查审批、恢复和导出引用。
- Autosave 恢复后按09号完整DependencySnapshot与stable inputs/outcome重算：无关project revision不误失效，实际依赖/policy/finding变化或ack/waiver撤回/到期才使相关审批inactive/stale。

### F2S-REC-UNDO-001：撤销重做

撤销/重做是领域 Command，不是 UI 状态快照。连续编辑重放必须得到相同 Rig IR。跨应用重启无需保留无限命令栈，但最后保存 revision 和 autosave 必须可恢复。

修改以下内容时相关审批变为 `stale`：

- 母版或可见像素；
- Attachment pivot、层级或透明边缘；
- Bone、Mesh、Weight、Constraint；
- 动作时间线、FPS、关键姿势；
- 命中帧、接触点、武器挂点；
- 导出 profile 或目标 Spine 版本。

仅修改全局project revision、不在TrackedGateInputResolver精确依赖闭包内的对象或无关priority/waiver/actor/bundle entry不得使审批stale；关联priority从P2收紧为P1时旧binding必须精确失效。SignatureAlgorithmRegistryV1缺失、固定bytes/双hash错配或非规范签名编码必须返回明确“governed waiver disabled”诊断，不得尝试算法猜测/宽松fallback。OrganizationTrustEnrollmentV1在同ProjectId首次固定后不可替换；无/过期/错edition/错签名enrollment必须返回明确“governed waiver disabled / clone-new-project-and-reapprove”恢复诊断，不能退化为普通warning或沿用旧grant。ActorTrustBundle/ActorRevocationBundle推进必须按current bundle重验并产新evidence；只有相关credential/SPKI/serial/anchor被撤销、过期或失效时才将binding置inactive，无关entry推进不得误失效。同boot使用`QueryInterruptTimePrecise`（含sleep）从durable checkpoint推进；pointer落后/分叉、wallUtc<lastSeenUtc、checkpointUtc/lastSeenUtc/counter任一回退均阻断。V1无可信外部时钟，因此**任何boot identity变化/不确定都禁止继续旧epoch waiver**，clean shutdown也不例外；非waiver审批不因此失效。ClockRecovery以expected project/waiver registry/checkpoint/recovery双heads追加record、epoch+1与全项目sequence+1的跨epoch successor checkpoint，该checkpoint必须prev双指旧epoch head并与RecoveryRecord唯一配对；prev-null、跳sequence、伪epoch或时间回退均不可恢复为active。全部旧epoch waiver bindings及依赖它们的approvals永久inactive，旧grant即使重启或再次到达旧时间也不复活。签名作业重启或超过freshness 900秒窗口只废弃本次signingAttempt并要求重新刷新/声明，不改变艺术项目revision，也不能复用旧attempt声明。

## 17.10 磁盘与资源异常

### F2S-REC-RESOURCE-001：容量和配额

- 开始大型任务前估算输入、staging、输出和安全余量。
- 磁盘不足时停止新写入，不通过删除旧 revision 自动腾空间。
- 日志、缩略图和可重建缓存有独立上限。
- CAS 垃圾回收只删除经完整引用扫描证明不可达的对象，并先进入可恢复隔离期。
- 内存/显存 OOM 不应导致 Rust 项目进程退出；Worker 独立进程承担高风险推理。

### F2S-REC-LOCK-001：项目锁

单用户桌面仍需防止双实例写同一项目：

- 打开写模式时获取项目锁并记录 PID、启动 nonce 和时间；
- 锁存在时验证进程是否存活和 nonce 是否匹配；
- 不因 PID 不存在就直接删除锁，先展示恢复选择；
- 第二实例默认只读打开；
- 网络盘/同步盘不作为 V1 受支持写入位置。

## 17.11 兼容性范围

V1 的 P0/一级验收环境为 Windows 11 x64、系统 Evergreen WebView2、本地 NTFS、普通非管理员用户。Windows 10 22H2 x64 为 P1 兼容目标，必须执行明确的 P1 恢复/升级矩阵，但不阻塞 P0 平台以外未承诺环境；网络盘、FAT/exFAT、OneDrive 按需文件和大小写敏感目录不作 V1 写入承诺。

路径测试必须覆盖中文、空格、括号、`&`、超长路径、只读目录和非系统盘。CLI 路径含特殊字符时仍使用参数数组，不能切换到 shell。

## 17.13 稳定验收映射

- `F2S-NFR-REL-001`：任意可撤销编辑不破坏最后确认 revision。
- `F2S-NFR-REL-002`：所有关键提交点故障注入后，只能恢复完整旧版或完整新版。
- `F2S-NFR-REL-003`：连续 500 个领域编辑命令的重放结果一致。
- `F2S-NFR-REL-004`：相同输入、Rig IR 和 profile 的导出语义稳定。
- `F2S-NFR-REL-005`：恢复 UI 同时展示确认保存、Autosave、未完成任务和项目外sandbox对账，由用户选择。
- `F2S-AC-RGPU-001`：私有远程任务断线可解释、可重试或取消，不覆盖批准资产。
- `F2S-AC-EXP-003`：Spine 缺失、版本不符或失败时，输出不显示为 verified。

## 17.14 明确禁止

- 禁止把 redb、缩略图缓存或 Worker 临时文件当作唯一源数据。
- 禁止为恢复方便而就地修改旧 revision 或 CAS 对象。
- 禁止把完整 journal、路径或提示词上传为遥测。
- 禁止在 CLI 失败后直接编辑 `.spine` 文件。
- 禁止通过“强制打开”丢弃未知项目字段。
- 禁止把其他 Spine patch/minor 的成功结果标为 4.2.43 已验证。
- 禁止应用升级时批量静默迁移用户全部项目。
- 禁止自动删除迁移前快照或上一个成功导出。
